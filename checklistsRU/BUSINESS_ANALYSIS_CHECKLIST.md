# Чек‑лист бизнес‑анализа и документирования для ZooLink

*Используйте этот чек‑лист при захвате новых бизнес‑идей, уточнении спецификаций или обновлении документации на этапе анализа/проектирования (без реализации кода).*

---

## ✅ Новая документация для создания

- [ ] **Документ идеи функции** – например, `docs/02-requirements/business-requirements/<новая-функция>-domain.md` или под `docs/02-requirements/` с подходящим именем.
- [ ] **Перевод на русский** – зеркалить вышеуказанное в `docsRU/02-requirements/business-requirements/`.
- [ ] **Записи в глоссарий** – добавить определения для любых новых терминов/понятий в `docs/specs/glossary.md` и `docsRU/specs/glossary.md`.
- [ ] **Обновления матрицы прослеживаемости** – добавить новые строки, связывающие ID бизнес‑требования → спецификация → критерии верификации → ADR → схема БД → API‑контракты (обе языковые версии).
- [ ] **Обновления моделей домена** – если предлагаются новые сущности/атрибуты, обновить концептуальные таблицы моделей в соответствующих файлах спецификаций доменов (как на английском, так и на русском).
- [ ] **Обновления описания ERD** – описать новые таблицы/отношения в `ERD_DESCRIPTION.md` (общий файл для обеих локалей).
- [ ] **Эскиз ERD в Mermaid** – при желании обновить `ZooLink_ERD.mmd` для визуализации новых сущностей (всё ещё только документация).

---

## 📝 Существующая документация для проверки и возможного обновления

### Общее
- [ ] **Файлы README** – убедиться, что верхнеуровневый `README.md`, `docs/README.md` и `docsRU/README.md` упоминают любые новые значимые области функционала.
- [ ] **Карта структуры проекта** – после добавления новой документации обновить `docs/project-structure-map.md` (см. раздел её поддержки).

### По домену (подстроить под функцию)
- [ ] **Домен животных** – рассмотреть, нужны ли новые поля (например, подтипы здоровья, поведенческие черты) или новые правила.
- [ ] **Организация / филиал** – подумать о иерархической организации, отчётности по нескольким филиалам, уровнях верификации.
- [ ] **Маркетплейс домашних животных и скота** – новые типы объявлений, дополнительные поля метаданных, workflow продвижения/комиссии.
- [ ] **Подбор** – новые критерии подбора (например, генетическая совместимость, географическая близость), модели оценки.
- [ ] **Администрация / модерация** – новые действия модерации, автоматизированное обнаружение мошенничества, панели отчётов.
- [ ] **Geo‑поиск** – новые аспекты поиска (например, по оказываемым услугам, по типу объекта).
- [ ] **Фронтенд** – новые концепции UI/страницы (например, профиль организации, панель аналитики) – захватить как эскизы wireframe или описание в `docs/specs/08-frontend-architecture.md`.
- [ ] **Нефункциональные требования** – если идея влияет на производительность, масштабируемость, безопасность или расширяемость, обновить соответствующие файлы в `docs/02-requirements/nfr/` и `docsRU/02-requirements/nfr/`.

### Контракты API (только как документация)
- [ ] Если идея вводит новый эндпоинт или изменяет существующий, обновить соответствующий YAML‑файл OpenAPI в `03-architecture/api-contracts/` **как документацию** (даже если ещё не реализовано). Оставить файл в репозитории, чтобы контракт был видим.
- [ ] Добавить примеры запросов/ответов, описать новые параметры запроса, заголовки, коды ошибок.

### Архитектурные решения (ADR)
- [ ] Если идея приводит к новому значимому выбору дизайна (например, внедрение event‑sourcing для поддомена, внедрение CQRS, выбор конкретного стороннего сервиса), создать новый ADR в `docs/04-decisions/` по существующему шаблону (например, `0002-<краткое-имя>.md`).

## 🔍 Шаги верификации (только документация)

### Согласованность и полнота
- [ ] Все newly созданные файлы должны быть корректным Markdown (без синтаксических ошибок). При наличии запустите `markdownlint` или аналогичный инструмент.
- [ ] Все перекрёстные ссылки (например, `см. animal-domain.md`) должны указывать на существующие файлы; если ссылка на планируемый файл, отметьте её как «(планируется)» и создайте заглушку.
- [ ] Терминология согласована: используйте одни и те же русские и английские термины во всех документах (сверьтесь с глоссарием).
- [ ] Английские и русские версии должны зеркально отражать друг друга там, где ожидается перевод (за исключением Intentional locale‑specific notes).
- [ ] В разных спецификациях не должно появляться дублирующихся или противоречащих друг другу бизнес‑правил.

### Прослеживаемость
- [ ] Для каждой захваченной бизнес‑правила или идеи функции должно быть хотя бы одно соответствие в матрице прослеживаемости (обе языковые версии), связывающее её с хотя бы одной спецификацией, одним критерием верификации, одним ADR (если применимо) и примечаниями о воздействии на схему БД/API.
- [ ] Любая новая сущность/атрибут, упомянутая в спецификации, должна отражаться в концептуальной таблице модели этой спецификации и, при возможности, в описании ERD.

### Гигиена артефактов
- [ ] В корне репозитория не должно оставаться посторонних файлов; всё должно находиться в соответствующих каталогах (`docs/`, `docsRU/`, `03-architecture/`, `checklists/` и т.п.).
- [ ] Имена файлов должны следовать правилу kebab‑case и установленной схеме нумерации (где применимо).
- [ ] Диаграммы (если добавлены) должны быть в формате `.svg` или `.mmd` и быть ссылками из соответствующей спецификации.

### Ревью и подпись
- [ ] **Ревью коллегой**: как минимум один другой участник команды (или системный аналитик) должен проверить новую/обновлённую документацию и оставить комментарии или одобрение.
- [ ] **Подпись владельца документа**: вы (автор) подписываете чек‑лист – добавьте строку с датой завершения внизу.
