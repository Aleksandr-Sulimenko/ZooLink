# ZooLink — Implementation Playbook (док↔код: протокол согласованности)

> **Назначение.** Обязательный рабочий протокол для фазы реализации бэкенда (и далее). Любой агент/разработчик в
> **любой сессии** обязан следовать ему, чтобы код всегда был согласован с документацией, а каждое изменение
> документации сопровождалось объяснением **что / почему / почему так лучше для проекта в целом**.
> Этот файл — «навык» проекта; он версионируется в репозитории и грузится через `CLAUDE.md`.

---

## 0. Золотое правило
**Документация — это контракт. Код реализует контракт, а не наоборот.**
Если при реализации обнаруживается, что контракт неверен/неполон — **сначала меняем документацию (с объяснением),
затем код**, в одном и том же изменении. Никаких «молчаливых» расхождений код↔док.

## 1. Иерархия источников истины (при конфликте — выше побеждает)
1. **ADR** (`docs/04-decisions/00NN-*.md`) — архитектурные решения. Accepted ADR не переписывают: новое решение =
   новый ADR со ссылкой «Amended/Superseded by».
2. **`database_schema.sql`** — единственный источник истины по схеме БД (валидируется на живом PostgreSQL).
   `ZooLink_ERD.mmd` и `docs/03-architecture/data-model.md` — его зеркала.
3. **`docs/03-architecture/api-contracts/API_CONVENTIONS.md`** — нормативные правила для всех OpenAPI-контрактов;
   сами `*.yaml` — производные и обязаны ему соответствовать.
4. **Доменные спеки** (`docs/specs/NN-*.md`, стейт-машины, `event-catalog.md`, `data-governance.md`,
   `market-differences.md`, `rbac-matrix.md`, `17-media-handling.md`) — бизнес-правила и контракты доменов.
5. **Нормативные baseline'ы**: `BACKEND_MVP_BASELINE.md` (что строим в MVP), `BACKEND_TECH_AUDIT.md` (обоснования).

Если код требует отклонения от (1)–(4) — это сигнал изменить документ, а не «обойти» его в коде.

## 2. Протокол изменения документации (объяснять ВСЕГДА)
Каждое изменение/дополнение док оформляется с тремя ответами — в теле ADR/спеки и/или в сообщении коммита:
- **ЧТО** изменилось (конкретно: поле, правило, эндпоинт, инвариант).
- **ПОЧЕМУ** — какая проблема/пробел/противоречие закрывается (ссылка на находку аудита/issue, если есть).
- **ПОЧЕМУ ТАК ЛУЧШЕ в контексте всего проекта** — как решение согласуется с ADR, MVP-границами, RF-контекстом,
  безопасностью/ФЗ-152, производительностью, и не ломает смежные домены. Указать рассмотренные альтернативы, если выбор неочевиден.

Решения уровня архитектуры → **новый ADR** по шаблону `docs/04-decisions/template.md`. Точечные правила → секция
«(round-N, normative)» в соответствующей спеке + та же тройка в коммите.

## 3. Дисциплина «код ↔ контракт» при разработке
- **БД-сущность/правило** → код использует имена/типы/инварианты ровно как в `database_schema.sql`. Любая новая
  колонка/таблица/индекс/триггер: правка `database_schema.sql` + **новая идемпотентная миграция** в `migrations/`
  (`YYYYMMDD_NNNN_*.sql`, гард `IF [NOT] EXISTS`/`DROP ... IF EXISTS`) + обновить `ZooLink_ERD.mmd` и `data-model.md` +
  счётчики таблиц/сущностей в `CLAUDE.md`.
- **API** → эндпоинт соответствует своему `*.yaml`, а `*.yaml` — `API_CONVENTIONS.md` (auth/roles `x-required-roles`,
  RFC7807 `Problem`, `page/limit`+`PageMeta`, `*_minor`/ISO-4217, §10 concurrency `If-Match`, §11 `Idempotency-Key`,
  §12 sort/filter, §13 caching, §14 deprecation).
- **Бизнес-логика/стейт-машины** → переходы/гарды/статусы в коде = значения CHECK в схеме = стейт-машина в спеке.
  Инварианты, которые можно выразить в БД (триггер/CHECK/UNIQUE), **должны** быть в БД, а не только в сервисе.
- **Локализация** → `*_localized` JSONB по контракту; язык/фолбэк по `data-governance.md`/`localization`.
- **События** → продьюсеры/консьюмеры/payload строго по `event-catalog.md` (outbox at-least-once + идемпотентность).

## 4. Дисциплина валидации (не «на глаз»)
- Изменения схемы/миграций: прогнать `database_schema.sql` + **все** миграции на одноразовой БД PostgreSQL,
  убедиться в идемпотентности (двойной прогон), добавить **негативные тесты** на новые инварианты.
- ERD: проверять рендер/парс mermaid после правок.
- EN↔RU: правки док — **синхронно в `docs/` и `docsRU/`** (идентификаторы/числа/структура идентичны, переводится
  только проза). Расхождение EN/RU — это баг.

## 5. Границы фаз — по стоимости изменения, не по ярлыку (forward-compat против ВСЕХ будущих фаз)
Решение «делать сейчас или отложить» принимается по **cost-of-change**, а не по номеру фазы. Граница MVP↔Фаза-2
размыта (MVP = тестовый режим, мало пользователей); Фаза-2-фичи подтягиваются вперёд и доходят до реализации
одновременно с MVP или сразу после. Правило шире Фазы 2 — учитываем **весь известный roadmap**: текущая реализация не
должна упираться в будущие фазы так, что потребуется массовое переписывание.

**Тащим в реализацию сейчас, если выполняется хотя бы одно:** (1) это **бизнес-требование** (вершина истины) — напр.
**ИИ-агентная модерация и админ-роль (ADR-0006)** → design-in с первого дня; (2) отсрочка вызовет **переписывание**
схемы / контракта / actor-principal-модели / authz / любого необратимого артефакта на любой будущей фазе («дешевле
всего менять в разработке») → форму фиксируем сейчас; (3) просто **дёшево** сейчас.

**Откладываем только если** реально дешевле потом **И** нет риска переписывания **И** это не бизнес-требование (напр.
фронтенд-accessibility/WCAG → фронтенд-фаза; чат — ADR-0005, явно out).

**Agent-first по умолчанию (α).** Любой код, фиксирующий актёра (модерация, админ-действия, audit), пишется
**agent-ready с первого коммита**: `actor_id` + `principal_type (HUMAN|AGENT)` + human-override + источник-агностичный
principal. Human-реализация — частный случай, не наоборот.

**Тест на переписывание — обязателен в DoD (β).** Прежде чем отложить что-либо, ответить письменно: «вызовет ли
отсрочка переписывание схемы / контракта / actor / authz на этой или будущей фазе?». Если **да или неясно → форму
делаем сейчас**. Молчаливый defer запрещён — как и молчаливое расхождение код↔док.

**Дисциплина.** Необратимое (форма схемы, actor/principal incl. agent-service-auth, authz-субъекты, форма контракта) —
**сейчас**; *поведение* катим итеративно, но за **настоящими, forward-compatible гейтами**, не плейсхолдерами-под-
переписывание. Payment за `feature_toggles.payments` (off) — образец «форма есть, поведение отложено». Если фича
требует крупной инфраструктуры (микросервисы/K8s/Elasticsearch/PostGIS, ADR-0009) — это сигнал к ADR/обсуждению **формы
сейчас**, а не к скрытой реализации и не к слепой отсрочке.

## 6. Definition of Done для изменения (чек-лист)
- [ ] Код соответствует канону (схема/контракт/спека/ADR); расхождений нет.
- [ ] Если контракт менялся — обновлены документы + тройка ЧТО/ПОЧЕМУ/ПОЧЕМУ-ЛУЧШЕ (ADR или секция спеки + коммит).
- [ ] Схема/миграции прогнаны и идемпотентны на живом PG; есть тесты на инварианты.
- [ ] EN↔RU синхронизированы; ERD/счётчики обновлены.
- [ ] API соответствует `API_CONVENTIONS.md`.
- [ ] Пройден **тест на переписывание** (§5): ничего необратимого не отложено так, что будущая фаза потребует переписывания (форма схемы/контракта/actor/authz — forward-compatible); отложенное — за настоящим гейтом и явно затрекано. Код-фиксатор актёра — agent-ready (`principal_type`).
- [ ] Коммит/пуш — только по явной просьбе пользователя.
