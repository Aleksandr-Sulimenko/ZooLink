# Административный домен: ZooLink



## Цель
Управляет конфигурацией системы, справочными данными, рабочими процессами модерации и ролями пользователей. Этот домен обеспечивает согласованность данных, поддерживает контроль качества контента и предоставляет операционные инструменты для обслуживания платформы.

## Основные концепции
- **Справочные данные**: Стандартизированные списки, используемые во всех доменах (виды, породы, города, черты, сертификаты здоровья и т.д.)
- **Очередь модерации**: Список объявлений, ожидающих проверки, с инструментами для одобрения/отклонения и предоставления обратной связи
- **Роли пользователей и разрешения**: Определяет, что пользователи могут делать в системе (USER, MODERATOR, ADMIN)
- **Системные настройки**: Флаги конфигурации и параметры, влияющие на поведение платформы
- **Журнал аудита**: Записи действий модерации, изменений ролей и системных событий для подотчетности

## Бизнес-правила

### 1. Управление справочными данными
- Справочные данные курируются и поддерживаются администраторами/модераторами (не редактируются пользователями на MVP).
- Общие наборы справочных данных включают:
 - **Виды**: Таксономические классификации (Canis lupus familiaris, Felis catus, Bos taurus и т.д.)
 - **Породы**: Видоварные разновидности (Labrador Retriever, Holstein Friesian, Siamese и т.д.)
 - **Города/Регионы**: Географическая иерархия для geo-поиска (Страна → Регион → Город → Район)
 - **Черты и описания**: Стандартизированные описания темперамента, здоровья и продуктивности
 - **Сертификаты здоровья**: Признанные типы тестов/вакцинаций (TB-free, Brucellosis-negative и т.д.)
 - **Генетические маркеры**: Известные результаты ДНК-тестов (окрас шерсти, polled/horned, устойчивость к болезням)
 - **Типы объявлений**: SALE, MATING, ADOPTION, STUD_SERVICE, LEASING (расширяемый)
 - **Статусы животных**: ACTIVE, ARCHIVED, DECEASED (для будущего)
- Справочные данные могут быть:
 - **Активированы/Деактивированы**: ללא удаления (для целостности исторических данных)
 - **Версионированы**: Изменения отслеживаются через журнал аудита (кто что изменил и когда)
 - **Локализованы**: Поддержка нескольких языков в Фазе 2+ (RU основной на MVP)
- Валидация: Записи справочных данных должны иметь уникальные коды/названия внутри своего набора.

### 2. Рабочий процесс модерации
- **Типы очередей**: Отдельные очереди для PET и LIVESTOCK объявлений (можно объединить в один вид)
- **Порядок очереди**:
 - По умолчанию: FIFO (first in, first out)
 - Параметры приоритета:
 - Сначала новые
 - По видам (для пакетной проверки аналогичных объявлений)
 - По репутации/рейтингу пользователя (будущее)
- **Действия модератора per объявление**:
 - **ОДОБРИТЬ**:
 - Меняет статус объявления на PUBLISHED
 - Не требует дополнительного ввода (хотя комментарий необязателен)
 - Объявление становится сразу видимым в поиске
 - **ОТКЛОНИТЬ**:
 - Возвращает объявление в состояние ДРАФТ
 - **Требует обязательной причины отклонения** (выбор из предопределенного списка + необязательный пользовательский текст)
 - Пользователь уведомляется с причиной и может исправить/повторно отправить
 - **ПОМЕТИТЬ НА ПЕРЕПРОВЕРКУ**:
 - Особое состояние для сложных случаев, требующих ввода старшего модератора
 - Не меняет состояние объявления, но добавляет индикатор
 - **ЗАБАНИТЬ ПОЛЬЗОВАТЕЛЯ** (изビュー модерации):
 - Доступно, если модератор обнаруживает паттерн abuse/spam
 - Требует указания причины и длительности (временной/постоянной)
- **Причины отклонения** (настраиваемые администратором):
 - Spam / контент низкого качества
 - Неподходящие фото (не соответствуют животному, оскорбительные)
 - Вводящая в заблуждение информация (ложные заявления о породе, нереалистичные цены)
 - Нарушение политики (продвижение нелегальной деятельности)
 - Неполная информация (отсутствуют ключевые детали)
 - Дубликат объявления
 - Конcерн по благосостоянию (несовершеннолетние животные, очевидная neglect)
 - Другое (с пояснением)
- **Лимиты времени модерации**:
 - Целевой показатель: <4 часа для pet объявлений, <6 часов для livestock объявлений в рабочее время (9AM-9PM)
 - Эскалация: Если очередь >50 элементов, уведомить старших модераторов
 - Старение: Объявления >24h в очереди выделяются в представлении модератора

### 3. Роли пользователей и разрешения
- **USER** (роль по умолчанию после регистрации):
 - Может создавать/редактировать собственный профиль и животных
 - Может создавать/объявления (попадают в модерацию)
 - Может искать и просматривать публичные объявления
 - Может показывать контакты на ОПУБЛИКОВАННЫХ объявлениях
 - Может деактивировать/повторно активировать собственных животных/объявления
 - Не может модерировать контент или управлять справочными данными
- **MODERATOR**:
 - Все разрешения USER
 - Может модерировать объявления (одобрять/отклонять с обязательными комментариями для отклонения)
 - Может просматривать очередь модерации и аналитику
 - Может управлять справочными данными (активировать/деактивировать, предлагать правки)
 - Может временно банить пользователей (до 30 дней) с причиной
 - Может просматривать základní пользовательскую аналитику (даты регистрации, количество объявлений)
 - Не может менять роли пользователей на/с MODERATOR/ADMIN
 - Не может получать доступ к системным настройкам
- **ADMIN**:
 - Все разрешения MODERATOR
 - Может управлять ролями пользователей (повышать до MODERATOR, понижать с MODERATOR/ADMIN)
 - Может банить пользователей постоянно или на длительные периоды
 - Может управлять системными настройками и флагами функций
 - Может просматривать полную системную аналитику и журналы
 - Может управлять API-ключами и сторонними интеграциями
 - Инициировать запросы на экспорт данных/GDPR
 - Не может изменять основной код платформы (требуется развертывание)
- **SUPER ADMIN** (системная роль, не назначается через интерфейс):
 - Полный доступ к системе (для экстренных случаев/обслуживания)
 - Обычно belongs to владельцы платформы/devops

### 4. Назначение и управление ролями
- Новые пользователи регистрируются как USER по умолчанию.
- Повышение ролей:
 - USER → MODERATOR: Требует одобрения ADMIN + базовое обучение
 - MODERATOR → ADMIN: Требует одобрения ADMIN + оценка доверия
 - Обратное понижение возможно в любое время
- Обучение модераторов включает:
 - Изучение политик и рекомендаций платформы
 - Обучение распознаванию частых нарушений
 - Период теневого наблюдения с опытным модератором
- Администраторы могут видеть статистику модераторов (обзоры/час, точность через уровень апелляций)
- Изменения ролей логируются в журнале аудита с причиной.

### 5. Системные настройки и флаги функций
- Настройки, контролируемые ролью ADMIN:
 - **Флаги функций**: Включение/выключение предстоящей функциональности (чат, видео, форумы)
 - **Лимиты скорости**: Корректировка порогов для действий (создание объявлений, показ контактов и т.д.)
 - **Параметры модерации**: Пороговые значения очереди, времена auto-expiration
 - **Настройки поиска**: Радиус по умолчанию, порядок сортировки, элементы на страницу
 - **Интеграционные настройки**: API-ключи для SMS, провайдеров OAuth, карт, email
 - **Режим обслуживания**: Read-only режим для обновлений/бэкапов
 - **Лимиты и пороги**: Макс. фото на объявление, макс. длина описания и т.д.
- Настройки хранятся в базе данных с журналом изменений.
- Некоторые настройки требуют перезапуска/ повторного развертывания для вступления в силу (документируется).
- **Флаги функций конкретно**:
 - `CHAT_ENABLED`: Управляет реальным временем сообщений (выключено на MVP)
 - `VIDEO_ENABLED`: Управляет загрузкой/воспроизведением видео (выключено на MVP)
 - `FORUM_ENABLED`: Управляет досками обсуждений (выключено на MVP)
 - `CALENDAR_ENABLED`: Управляет репродуктивными календарями (выключено на MVP)
 - `AI_MODERATION`: Управляет предварительной проверкой с помощью ML (выключено на MVP)
 - `NEGOTIABLE_PRICE`: Управляет возможностью указания "необсуждаемой" цены в поле цены (включено на MVP)
 - `CUSTOM_BREED_TEXT`: Управляет возможностью ввода пользовательского текста породы (включено на MVP)

### 6. Журнал аудита и соответствие
- Критические действия логируются для подотчетности и отладки:
 - Действия модерации (кто одобрил/отклонил какое объявление и почему)
 - Изменения справочных данных (кто добавил/удалил/изменил что)
 - Изменения ролей (кто повысил/понизил кого и почему)
 - Баны пользователей (кто забанил кого, причина, длительность)
 - Изменения настроек (что изменилось, от/до, кто)
 - Запросы на экспорт данных/GDPR
- Журналы аудита включают:
 - Метку времени (UTC)
 - ID пользователя-актёра и его роль во время действия
 - Тип действия и целевой ID сущности
 - Значения до/после (для изменений)
 - Причина/комментарий (если применимо)
 - IP-адрес и user-agent (для мониторинга безопасности)
 - Журналы хранятся согласно политике хранения данных (минимум 1 год для модерации/журналов).
 - Доступ к журналам аудита ограничен ролью ADMIN.
 - НЕ логирует конфиденциальные личные данные (пароли, токены, полная PII).

## Нефункциональные требования (специфичные для Admin)
- **Производительность**:
 - Загрузка очереди модерации: <2s для 100 элементов
 - Поиск в справочных данных: <100ms для любого набора
 - Запись в журнал аудита: <50ms накладных расходов на каждое аудитируемое действие
- **Масштабируемость**:
 - Поддержка 100+ активных модераторов
 - Обработка 1000+ действий модерации в день в пиковые периоды
 - Справочные наборы до 10k записей каждый (виды, породы и т.д.)
- **Согласованность**:
 - Изменения справочных данных видны глобально в течение 1s (инвалидация кеша)
 - Действия модерации атомарны (либо полностью применены, либо не применены)
 - Изменения ролей вступают в силу немедленно для новых сессий
- **Расширяемость**:
 - Новые наборы справочных данных могут быть добавлены через интерфейс администратора
 - Новые действия/причины модерации могут быть настроены
 - Новые флаги функций следуют стандартному шаблону
- **Безопасность**:
 - Требования к надежным паролям для ролей MODERATOR/ADMIN (мин. 12 символов)
 - MFA поощряется для роли ADMIN (планируется на Фазу 2+)
 - Таймаут сессии: 15 минут бездействия для чувствительных действий
 - Журнал аудита сам по себе является защитным от вмешательства (добавление-only, подписанные записи)
 - Принцип наименьших привилегий: MODERATOR не может повышать пользователей до ADMIN

## Модель данных (Концептуальная)
### Сущность справочных данных (универсальный паттерн)
| Атрибут | Тип | Обязателен | Описание |
|---------|-----|------------|----------|
| `id` | UUID | Да | Первичный ключ |
| `dataset` | VARCHAR(50) | Да | Название набора справочных данных (виды, породы, города и т.д.) |
| `code` | VARCHAR(50) | Да | Уникальный код внутри набора (например, "LAB", "HOL") |
| `name_localized` | JSONB | Да | Мультиязычные имена (RU основной на MVP: {"ru": "Лабрадор-ретривер"}) |
| `description` | TEXT | Нет | Расширенное описание |
| `is_active` | BOOLEAN | Да | Доступно ли для выбора |
| `sort_order` | INT | Нет | Для пользовательского порядка в списках |
| `metadata` | JSONB | Нет | Атрибуты, специфичные для набора (например, для видов: таксономический класс) |
| `created_at` | TIMESTAMP | Да | |
| `updated_at` | TIMESTAMP | Да | |
| `created_by` | UUID (FK to Users.id) | Да | Кто создал запись |
| `updated_by` | UUID (FK to Users.id) | Да | Кто последним обновил её |

### Сущность журнала модерации
| Атрибут | Тип | Обязателен | Описание |
|---------|-----|------------|----------|
| `id` | UUID | Да | Первичный ключ |
| `listing_id` | UUID (FK to Listings.id) | Да | Объявление, подвергающееся модерированию |
| `moderator_id` | UUID (FK to Users.id) | Да | КтоPerformed действие |
| `action` | ENUM('APPROVE', 'REJECT', 'FLAG') | Да | Что сделано |
| `reason_code` | VARCHAR(50) | Нет | Стандартизированная причина отклонения (если REJECT) |
| `reason_text` | TEXT | Нет | Пользовательское объяснение (обязательно для REJECT) |
| `created_at` | TIMESTAMP | Да | Когда действие произошло |
| `metadata` | JSONB | Нет | Дополнительный контекст (например, IP, user-agent) |

### Назначение ролей пользователей (упрощенное представление)
| Атрибут | Тип | Обязателен | Описание |
|---------|-----|------------|----------|
| `user_id` | UUID (FK to Users.id) | Да | Пользователь |
| `role` | ENUM('USER', 'MODERATOR', 'ADMIN') | Да | Назначенная роль |
| `assigned_at` | TIMESTAMP | Да | Когда роль была назначена |

| `reason` | TEXT | Нет | Почему роль была назначена/изменена |

## Пользовательский путь: Модерация объявления
```mermaid
sequenceDiagram
 participant Moderator
 participant Frontend
 participant Backend (NestJS Admin/Moderation Module)
 participant Database
 participant Listing (в домене Pet/Livestock)

 %% Модератор логинится и просматривает очередь
 Moderator->>Frontend: Открывает страницу модерации
 Frontend->>Backend: GET /moderation/queue?type=pet&limit=20&offset=0
 Backend->>Database: Возвращает ожидающие объявления с основной информацией
 Backend->>Frontend: Возвращает элементы очереди (название, вид, возраст, время ожидания)

 Moderator->>Frontend: Выбирает объявление #3 для проверки
 Frontend->>Backend: GET /moderation/listing/{id} (получает полные детали)
 Backend->>Database: Извлекает объявление + связанное животное + фотографии
 Backend->>Frontend: Возвращает полные данные объявления для проверки

 Moderator->>Frontend: Проверяет объявление, решает одобрить
 Frontend->>Backend: POST /moderation/action {listing_id: X, action: APPROVE}
 Backend->>Database:
 1. Обновляет статус объявления на PUBLISHED
 2. Вставляет запись в журнал модерации (APPROVE, moderator_id, timestamp)
 Backend->>Frontend: Возвращает успех

 %% Альтернатива: Отклонить с причиной
 Moderator->>Frontend: Проверяет объявление, решает отклонить (spam)
 Frontend->>Backend: POST /moderation/action {listing_id: Y, action: REJECT, reason_code: SPAM, reason_text: "Использовано стоковое фото, не соответствующее описанному животному"}
 Backend->>Database:
 1. Обновляет статус объявления на DRAFT
 2. Устанавливает запись в журнале модерации с причиной REJECT
 3. Уведомляет пользователя через email/in-app (отделенный сервис)
 Backend->>Frontend: Возвращает успех

 %% Модератор управляет справочными данными
 Moderator->>Frontend: Переходит к Справочные данные -> Породы -> Добавить новую
 Frontend->>Backend: GET /reference-data/breeds/new (получает список видов для выпадающего списка)
 Backend->>Reference Data Service: Возвращает активные виды
 Backend->>Frontend: Возвращает список видов

 Moderator->>Frontend: Выбирает вид=Dog, вводит код="LABZ", имя="Лабрадор-ретривер (Золотистый)", описание="..."
 Frontend->>Backend: POST /reference-data/breeds {species_id: X, code: "LABZ", name: {...}, description: "..."}
 Backend->>Database:
 1. Проверяет уникальность кода внутри набора пород
 2. Вставляет новую запись справочных данных
 3. Логирует действие в журнале аудита (reference_data_create)
 Backend->>Frontend: Возвращает успех

 %% Модератор банит пользователя за abuse
 Moderator->>Frontend: Просматривает профиль пользователя из объявления, видит паттерн спама
 Frontend->>Backend: POST /moderation/ban-user {user_id: Z, duration_days: 7, reason: "Повторяющийся спам в объявлениях"}
 Backend->>Database:
 1. Устанавливает user.banned_until = now + 7 days
 2. Устанавливает user.ban_reason = "Повторяющийся спам в объявлениях"
 3. Вставляет запись в журнал аудита (user_ban)
 4. По желанию: скрывает текущие объявления пользователя из поиска
 Backend->>Frontend: Возвращает успех
```

## Открытые вопросы и предположения
- **Предположение**: Исходные справочные данные будут загружены из открытых источников (AKC, FCI, FAO списки пород) и уточняться со временем.
- **Предположение**: Нагрузка модератора на MVP будет управляема 1-2 штатными модераторами на неполный рабочий день (<50 объявлений/день всего).
- **Открытый вопрос**: Следует ли внедрить систему репутации/рейтинга пользователей для приоритизации очереди модерации? (Решено: Нет для MVP; полагаемся на хронологическую очередь + ручноеflagging.)
- **Предположение**: Модераторы получат базовое обучение по политикам, но не формальную сертификацию на MVP.
- **Предположение**: Система не будет использовать автоматическую фильтрацию контента (AI/ML) для модерации на MVP, чтобы избежать ложных срабатываний/пропусков.
- **Предположение**: Процесс апелляции для отклоненных объявлений: пользователь может повторно отправить после правок; формальный процесс апелляции зарезервирован для Фазы 2+.
- **Предположение**: Географические справочные данные (города) будут упрощенной иерархией; подробные уличные адреса никогда не будут храниться/отображаться.

## Связанные домены
- **Домен идентификации**: Предоставляет базу пользователей; роли расширяют разрешения пользователей; аутентификация обязательна для доступа к админке.
- **Домен животных**: Управляет видами и породами; Административный домен предоставляет интерфейс курирования.
- **Домен объявлений для животных и скота**: Опирается на Административный домен для справочных данных (виды, породы, города) и рабочего процесса модерации.
- **Домен подбора**: Может использовать справочные данные для целей разведения, библиотек черт или видовоспецифичных факторов совместимости.
- **Будущие домены**:
 - **Соблюдение нормативных требований**: Расширит Административный домен для управления справочными данными по нормамативам (например, сертифицированные лаборатории, одобренные лекарства).
 - **Управление контентом**: Для управления статическими страницами, FAQ и центром помощи.

## Ссылки на контракт API (см. 03-architecture/api-contracts/admin-api.yaml)
- `GET /reference-data/{dataset}` (получить все активные записи в наборе)
- `GET /reference-data/{dataset}/new` (получить форму для создания новой записи)
- `POST /reference-data/{dataset}` (создать новую запись справочных данных)
- `PATCH /reference-data/{dataset}/{id}` (обновить запись справочных данных)
- `PATCH /reference-data/{dataset}/{id}/toggle-active` (активировать/деактивировать запись)
- `GET /reference-data/{dataset}/{id}` (получить конкретную запись)
- `GET /moderation/queue` (получить ожидающие объявления с фильтрами: type, species, limit, offset)
- `GET /moderation/listing/{id}` (получить детали объявления для модерации)
- `POST /moderation/action` (одобрить/отклонить/пометить объявление)
- `GET /moderation/log/{listing_id}` (получить историю модерации для объявления)
- `POST /moderation/ban-user` (забанить пользователя с причиной/длительностью)
- `GET /system/settings` (получить текущие настройки - только для ADMIN)
- `PATCH /system/settings/{key}` (обновить настройку - только для ADMIN)
- `GET /audit/log` (получить записи журнала аудита с фильтрами - только для ADMIN)
- `GET /users/roles` (получить пользователей с их ролями - только для ADMIN)
- `PATCH /users/{id}/role` (изменить роль пользователя - только для ADMIN)
