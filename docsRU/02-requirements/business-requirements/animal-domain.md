# Домен «Животное»: ZooLink

## Цель
Управляет основной сущностью "Животное" как агрегатным корнем. Животное может иметь несколько объявлений (на продажу, для спаривания, на мероприятия) и принадлежит либо пользователю, либо организации. Эта область обеспечивает целостность данных профилей животных, поддерживает поиск по характеристикам и создаёт основу для будущих функций, таких как родословная, медицинские записи и репродуктивные календари.

## Основные понятия
- **Животное**: Живое существо (домашнее животное или скот), зарегистрированное в системе. Имеет внутренние атрибуты (вид, порода, пол, возраст) и опциональные атрибуты (медицинские тесты, прививки, микрочип). Дополнительные атрибуты включают метку времени владения, ссылки на родителей для отслеживания родословной и метку времени деактивации.
- **Владелец**: Пользователь или организация, которая зарегистрировала животное в системе (может отличаться от юридического владельца; система отслеживает опекунство для целей объявлений). Животное может принадлежать как личному пользователю (`owner_id`), так и организации (`organization_id`).
- **Справочник пород/видов**: Справочные данные, управляемые Административной областью (см. admin-domain.md).
- **Статус здоровья**: Структурированные данные о прививках, тестах, лечениях (расширяемые в будущем).
- **Репродуктивные данные**: Для использований, связанных со спариванием (циклы течки, последнее спаривание, предполагаемые даты родов) – актуально только для определённых видов. Включает ссылки на мать и отца для отслеживания родословной.

## Бизнес-правила
### 1. Создание животного
- Только аутентифицированные пользователи могут создавать профиль животного.
- Обязательные поля при создании:
  - Вид (из справочника, например, кошка, собака, крупный рогатый скот, лошадь, овца, коза, свинья, курица, кролик и т.д.)
  - Порода (из справочника для выбранного вида; допускается "Смешанная/Неизвестная" или пользовательский текст, если порода не указана в списке)
  - Пол (Мужской/Женский)
  - Дата рождения или приблизительный возраст (пользователь может ввести годы/месяцы; система хранит как дату или ISO неделю)
  - Прозвище (отображаемое имя, опционально, но рекомендуется)
- При создании должно быть указано либо личное владение (`owner_id`), либо организационное владение (`organization_id`), но не оба одновременно.
- Опциональные поля при создании:
  - Цвет/узор шерсти (свободный текст)
  - ID микрочипа (если предоставлен, должен быть уникален на вид? Не проверяется глобально, но выдаётся предупреждение при дубликате)
  - ID татуировки/клейма (для скота)
  - Начальные записи о здоровье (прививки, дегельминтизация) – могут быть добавлены позже
- При создании животное привязывается либо к создающему пользователю через `owner_id`, либо к организации через `organization_id`.
- Пользователь может создавать несколько профилей животных (без ограничений в MVP).
- Одно и то же животное не может быть зарегистрировано двумя разными пользователями (для предотвращения споров). При подозрении на дубликат (тот же микрочип + схожие атрибуты) модератор должен провести расследование.
- Животное не может принадлежать одновременно и пользователю, и организации (валидация на уровне приложения обеспечивает, что именно одно из `owner_id` или `organization_id` установлено).

### 1.5 Организационная собственность
- Животное может принадлежать либо отдельному пользователю, либо организации, но не обоим одновременно.
- Поле `organization_id` является опциональным и nullable; когда оно установлено, это указывает на то, что животное принадлежит организации, а не физическому лицу.
- Когда `organization_id` установлено, поле `owner_id` должно быть null, и наоборот.
- Правило валидации: ровно одно из полей `owner_id` или `organization_id` должно быть NOT NULL для корректной записи животного.
- Организационная собственность обеспечивает:
  - Возможность для нескольких пользователей внутри организации управлять объявлениями животного (через связи organization-user)
  - Централизованное владение животными для ферм разведения, приютов или бизнеса
  - Четкую атрибуцию животных организационным сущностям вместо индивидуальных пользователей
- Пользователь организации, действующий от имени своей организации, может создавать объявления для животных, принадлежащих этой организации.

### 2. Обновление животного
- Владелец (пользователь или представитель организации) может обновлять большинство полей в любое время:
  - Прозвище
  - Цвет/узор шерсти
  - Микрочип/татуировка (если добавляется или исправляется)
  - Медицинские записи (добавлять новые)
  - Репродуктивные данные (обновлять даты течки, даты спаривания)
- Поля, которые нельзя изменить после создания (для сохранения целостности):
  - Вид
  - Порода (если выбрана из справочника; пользовательский текст можно редактировать)
  - Пол
  - Дата рождения (можно уточнить, но не изменять произвольно; например, обновление с "примерно 2 года" на "2023-05-12" допустимо, если согласовано)
- Изменение владельца **поддерживается** через формальный рабочий процесс передачи владения (НЕ через создание нового профиля). Текущий владелец инициирует передачу получателю (пользователю или организации); при принятии запись животного переатрибутируется, а изменение фиксируется в `ownership_transfers` (плюс `animal_ownership_history`), сохраняя идентичность животного, связи родословной и аудиторский след. Точные состояния и переходы нормативно описаны в `docs/specs/02-animal-domain.md` и `docs/specs/statemachines/ownership_transfer_state_machine.md`. (Ручной сценарий «деактивировать + перерегистрировать» не является механизмом передачи владения — он разрушил бы родословную и историю.)

> **(GAP-TRACE-007, normative) — передача владения входит в scope MVP, а не запрещена.**
> **ЧТО:** Заменено «изменение владельца в MVP не допускается / создать новый профиль» на поддерживаемый рабочий процесс `ownership_transfers` (spec 02 + стейт-машина передачи владения).
> **ПОЧЕМУ:** GAP-TRACE-007 — этот BR прямо запрещал флоу, который смоделирован и в плане: таблица `ownership_transfers` (`database_schema.sql`), её стейт-машина и spec 02 трактуют передачу как входящую в scope. BR устарел (ранняя редакция) и противоречил вышестоящим схеме/спеке.
> **ПОЧЕМУ ТАК ЛУЧШЕ для проекта в целом:** Формальная передача сохраняет историю/родословную и предотвращает дублирующие регистрации, ради чего и вводился старый запрет — она достигает исходной цели (нет споров) без потери данных, как при «создании нового профиля». Один канон истины по линии BR ↔ spec ↔ схема.

### 3. Деактивация/архивирование животного
- Владелец может деактивировать животное (soft delete):
  - Животное исчезает из поиска и списка владельца.
  - Все существующие объявления, ссылающиеся на это животное, остаются видимыми, но помечаются как относящиеся к деактивированному животному (в интерфейсе показывается предупреждение).
  - Новые объявления для деактивированного животного создавать нельзя.
- Деактивация не удаляет данные; они сохраняются по историческим и юридическим причинам.
- Повторная активация возвращает животное в активное состояние и повторно разрешает создание новых объявлений.
- Жёсткое удаление не допускается в MVP (для сохранения аудиторского следа). Может быть добавлено позже через рабочий процесс запроса GDPR.

### 4. Связь с объявлениями
- Животное может быть связано с нулём, одним или множеством объявлений.
- При создании объявления пользователь должен выбрать существующее животное из своих принадлежащих животных (личных) или из животных, принадлежащих их организации (если действует от имени организации).
- Объявление не может быть создано без указания животного (животное обязательно).
- Если животное деактивировано, его существующие объявления остаются активными (чтобы не прерывать текущие сделки), но владелец не может их редактировать (за исключением закрытия/завершения).
- Если объявление удалено/завершено, животное остаётся незатронутым.

### 5. Медицинские и репродуктивные данные (расширяемые)
- Медицинские записи хранятся как список событий с датой, типом и деталями (например, прививка: бешенство, дата: 2024-03-01, ветеринарная клиника: "Zoovet").
- Актуально для спаривания: у самок можно фиксировать даты течки.
- Эта область не проверяет корректность медицинских данных (например, даты прививок) – полагается на честность пользователей и выборочные проверки модераторов.
- Будущие расширения: интеграция с ветеринарными клиниками, цифровые паспорта здоровья.

### 6. Поиск и обнаружение
- Животные доступны для поиска по их атрибутам, когда они связаны с публичным объявлением.
- Поля поиска (через область Объявлений):
  - Вид
  - Порода
  - Пол
  - Возрастной диапазон (вычисляется из даты рождения)
  - Цвет/узор шерсти (поиск по свободному тексту)
  - Наличие микрочипа? (логическое)
  - Флаги здоровья/тестов (например, "привито", "ДНК-тестировано")
- Сам по себе домен Животное не предоставляет публичный API поиска в MVP (чтобы уменьшить поверхность атаки). Данные о животных доступны только через объявления.
- В Фазе 2+ может быть добавлен отдельный "Каталог животных" или "Профили домашних животных" (например, для просмотра животных, которые в настоящий момент не выставлены на продажу).

### 7. Целостность данных и валидация
- Вид и порода должны существовать в справочнике (управляется Администрацией) ИЛИ пользователь может выбрать "Другое" и ввести пользовательский текст (помечен для проверки модератором).
- Дата рождения должна быть в прошлом и не раньше чем 30 лет назад (настраивается в зависимости от вида).
- Поля микрочипа/татуировки, если заполнены, должны соответствовать подсказкам формата (длина, символы), но глобальная проверка уникальности не выполняется.
- Все текстовые поля имеют разумные ограничения (прозвище: 50 символов, цвет/узор: 100 символов, медицинская заметка: 500 символов на запись).
- **Важно**: Должно быть точно одно из полей `owner_id` или `organization_id` установлено (NOT NULL), а другое должно быть NULL.

## Нефункциональные требования (специфичные для области Животное)
- **Производительность**: Получение животного по ID (с основной информацией) должно выполняться за <100 мс.
- **Масштабируемость**: Должно поддерживать до 100 000 профилей животных без ухудшения операций создания/обновления/поиска.
- **Согласованность**: Сильная согласованность в пределах транзакции (например, создание животного и привязка его к пользователю/организации происходит в одной транзакции БД).
- **Расширяемость**: Использование колонок JSONB для медицинских/репродуктивных данных позволяет добавлять новые поля без изменений схемы.
- **Конфиденциальность**:
  - Данные о животном (кроме той части, которая показывается в публичных объявлениях) считаются персональными данными, связанными с владельцем (под ФЗ-152).
  - Раскрывается публично только минимум необходимый для объявлений (вид, порода, пол, возраст, цвет, фотографии, базовые флаги здоровья).
  - Контактная информация владельца не раскрывается через профиль животного; только через функцию раскрытия контактов в объявлении.
- **Резервное копирование/восстановление**: Данные о животных являются частью обычных резервных копий БД; поддерживается восстановление на момент времени.

## Концептуальная модель данных
| Атрибут | Тип | Обязательно | Описание     |
|-------------------|-------------------|-------------|--------------------------------------------------------------------------|
| `id` | UUID | Да | Первичный ключ    |
| `owner_id` | UUID (FK к Users.id) | Нет | Пользователь, создавший профиль (nullable если `organization_id` установлен) |
| `organization_id` | UUID (FK к Organizations.id) | Нет | Организация, владеющая животным (nullable если `owner_id` установлен) |
| `species_id` | INT (FK к справочнику видов) | Да | Из справочных данных    |
| `breed_id` | INT (FK к справочнику пород) | Нет (nullable, если выбрано "Другое/пользовательский") | Из справочных данных; может быть null, если указан `breed_text_localized` |
| `breed_text_localized` | JSONB | Нет | Локализованный пользовательский текст породы, если `breed_id` равен null (для проверки модератором) |
| `sex` | ENUM('Мужской', 'Женский') | Да |     |
| `date_of_birth` | DATE | Да | Оценённая или фактическая дата рождения   |
| `nickname_localized` | JSONB | Нет | Локализованное отображаемое имя    |
| `color_coat` | VARCHAR(100) | Нет | Свободное текстовое описание   |
| `microchip_id` | VARCHAR(50) | Нет | Если предоставлен    |
| `tattoo_brand_id` | VARCHAR(50) | Нет | Для скота    |
| `is_active` | BOOLEAN | Да | True = виден для новых объявлений; False = деактивировано |
| `owned_since` | DATE | Нет | Дата, когда текущий владелец приобрел животное |
| `mother_id` | UUID (FK к Animals.id) | Нет | Ссылка на мать для отслеживания родословной |
| `father_id` | UUID (FK к Animals.id) | Нет | Ссылка на отца для отслеживания родословной |
| `deactivated_at` | TIMESTAMP | Нет | Отметка времени, когда животное было деактивировано (мягкое удаление) |
| `created_at` | TIMESTAMP | Да |     |
| `updated_at` | TIMESTAMP | Да |     |
| `health_records` | JSONB | Нет | Массив объектов: [{type: 'vaccination', detail: 'Rabies', date: '2024-03-01', provider: 'VetClinic'}] |
| `reproductive_data`| JSONB | Нет | Для самок: [{event: 'heat_start', date: '2024-05-10'}, {event: 'mating', date: '2024-05-12', partner_id: null}] |

## Правила валидации (примеры)
- Вид должен существовать в справочнике `species`.
- Если `breed_id` равен null, то `breed_text_localized` должен быть указан (и наоборот, желательно).
- Дата рождения ≤ сегодня и ≥ сегодня - 30 лет (корректируется в будущем в зависимости от вида).
- Прозвище, если указано, не может быть пустой строкой.
- ID микрочипа, если указан, должен содержать минимум 8 символов.
- JSONB поля медицинских записей должны соответствовать схеме (валидация на уровне приложения).
- **Точно одно из полей `owner_id` или `organization_id` должно быть NOT NULL** (валидация на уровне приложения).
- `owned_since`, если указан, должен быть датой в прошлом и не позже чем `date_of_birth` + текущий возраст (проверка здравого смысла).
- `mother_id` и `father_id`, если указаны, должны ссылаться на существующие записи животных соответствующего пола (женский для матери, мужской для отца).
- `deactivated_at`, если установлен, должен быть позже `created_at` и `is_active` должен быть false.

```mermaid

    participant FE as Фронтенд
    participant BE as Бэкенд
    participant DB as База данных
    participant Dir as Справочный сервис

    Note over U,DB: Регистрация нового животного
    U->>FE: Открыть "Добавить животное"
    FE->>BE: GET /animals/new
    BE->>Dir: Получить species/breeds
    Dir-->>BE: Списки
    BE-->>FE: Списки видов/пород
    U->>FE: Заполнить вид, породу или свой текст, ДР, кличку, пол
    FE->>BE: POST /animals {species_id, breed_id|breed_text_localized, sex, date_of_birth, nickname_localized}
    alt Валидно (ровно одно из breed_id / breed_text, обязательные есть)
        BE->>DB: insert animal (owner_id = текущий пользователь)
        DB-->>BE: animal_id
        BE-->>FE: 201 Created (animal_id)
    else Оба или ни одного breed_id и breed_text (chk_animals_breed_dep)
        BE-->>FE: 422 Укажите ровно одно: породу или свой текст
    else Отсутствует обязательное поле
        BE-->>FE: 422 Validation error
    end

    Note over U,DB: Добавить медицинскую запись
    U->>FE: Добавить вакцинацию
    FE->>BE: PATCH /animals/{id} {health_records}
    alt Пользователь владеет животным
        BE->>DB: обновить health_records JSONB
        BE-->>FE: 200 Success
    else Не владелец
        BE-->>FE: 403 Forbidden
    end

    Note over U,DB: Изменение неизменяемого поля (заблокировано в MVP)
    U->>FE: Попытка сменить species/ДР/пол/владельца
    FE->>BE: PATCH /animals/{id} {species_id}
    BE->>DB: UPDATE animals (триггер trg_animals_immutable_and_owner)
    alt Изменено неизменяемое поле
        DB-->>BE: ERROR immutable field
        BE-->>FE: 409 Нельзя менять неизменяемое поле после создания
    else Разрешённое поле
        DB-->>BE: обновлено
        BE-->>FE: 200 Success
    end

    Note over U,DB: Размещение животного на маркетплейсе

        BE->>DB: insert listing (status=DRAFT)
        BE-->>FE: 201 Created (listing_id)
    else Дубликат активного листинга того же типа (uq_active_listing_per_type)
        BE-->>FE: 409 У животного уже есть активный листинг этого типа
    else Животное не в собственности или неактивно
        BE-->>FE: 403 Forbidden
    end
```

## Связанные области
- **Административная область**: Предоставляет справочники видов и пород; управляет модерируемыми записями пользовательского текста пород.
- **Область Объявлений**: Связывается с животными через `animal_id`; отображает краткую информацию о животном в объявлениях.
- **Область Идентификации**: `owner_id` ссылается на пользователя; аутентификация обязательна для создания/модификации животного.
- **Организационная область**: `organization_id` ссылается на организацию; животное может принадлежать организации вместо личного пользователя.
- **Область Сопоставления**: Может использовать атрибуты животных (вид, порода, пол, возраст, флаги здоровья) для предложения партнёров по спариванию.
- **Будущие области**: Паспорт здоровья, репродуктивный календарь, конструктор родословных будут расширять эту область.

## Ссылки на контракт API (см. 03-architecture/api-contracts/animals-api.yaml)
- `GET /animals/new` (получение справочника для создания)
- `POST /animals` (создание животного с owner_id ИЛИ organization_id)
- `GET /animals/{id}` (получение животного по ID – владелец или представитель организации, либо публично, если связано с активным объявлением)
- `PATCH /animals/{id}` (обновление животного)
- `GET /animals` (список животных пользователя/организации с фильтрами: активные/неактивные, вид, порода)
- `PATCH /animals/{id}/deactivate`
- `PATCH /animals/{id}/reactivate`
- Примечание: Публичный поиск по атрибутам животного в одиночку отсутствует в MVP (для предотвращения скрейпинга); данные о животном доступны только через объявления.
