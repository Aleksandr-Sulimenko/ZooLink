# Домен идентификации: ZooLink

## Цель
Управляет аутентификацией, авторизацией и информацией о профиле пользователя. Этот домен является шлюзом в систему и обеспечивает безопасный доступ при минимальном сборе персональных данных в соответствии с ФЗ-152.

## Основные концепции
- **Пользователь**: Человек, зарегистрированный в системе. Может быть частным лицом, заводчиком, фермером, модератором, ветеринаром или грумером.
- **Метод аутентификации**: Как пользователь доказывает свою личность (SMS-код телефона, провайдеры OAuth).
- **Роль пользователя**: Определяет разрешения в системе (обычный пользователь, модератор, администратор, ветеринар, грумер).
- **Профиль**: Публичная и личная информация, связанная с пользователем.

## Бизнес-правила
1. **Аутентификация**
 - Пользователи должны аутентифицироваться одним из разрешенных методов:
 - Подтверждение номера телефона (SMS-код)
 - OAuth 2.0 с Google, Apple, Telegram, VK
 - Подтверждение email является опциональным и не блокирует регистрацию.
 - **Аутентификация конечных пользователей беспарольная** (только phone OTP + OAuth); конечные пользователи никогда не задают и не используют пароль (см. `docs/specs/01-identity-domain.md`).
 - После успешной аутентификации выдаётся access-токен (JWT) со сроком действия **15 минут** и refresh-токен со сроком действия **7 дней**.
 - Refresh-токены хранятся безопасно и ротируются при использовании (family-based ротация).

2. **Регистрация пользователя**
 - Минимальные обязательные поля для регистрации:
 - Номер телефона (уникальный) ИЛИ связанный аккаунт OAuth
 - Полное имя (для отображения, не обязательно юридическое имя)
 - Город (выбран из справочника, используется для гео-поиска)
 - Пароль при регистрации не собирается — аутентификация конечных пользователей беспарольная (phone OTP / OAuth).
 - Необязательные поля при регистрации:
 - Адрес email (для уведомлений и восстановления)
 - Изображение аватара (URL во внешнем хранилище)
 - После регистрации пользователю назначается роль `USER` по умолчанию.
 - Пользователи не могут зарегистрировать более одного аккаунта на один номер телефона (предотвращает спам).

3. **Роли пользователей и разрешения**
 - `USER`: Может создавать/редактировать собственный профиль, создавать/просматривать объявления, искать, просматривать публичные данные, показывать контакты после модерации.
 - `MODERATOR`: Все разрешения USER + может модерировать объявления (одобрять/отклонять), управлять справочными данными (породы, виды), блокировать пользователей.
 - `ADMIN`: Все разрешения MODERATOR + может управлять ролями модератора/администратора, просматривать системную аналитику, изменять глобальные настройки.
 - `VETERINARIAN`: Может просматривать медицинские записи животных, проверять информацию о вакцинации, предоставлять профессиональные консультации по здоровью животных (конкретные разрешения будут определены в будущих фазах).
 - `GROOMER`: Может просматривать информацию, связанную с грумингом животных, предоставлять информацию об услугах груминга (конкретные разрешения будут определены в будущих фазах).
 - Роли аддитивны (ADMIN включает разрешения MODERATOR и USER; VETERINARIAN и GROOMER включают разрешения USER для базовой функциональности).

4. **Управление профилем**
 - Пользователи могут обновлять свой профиль в любое время:
 - Полное имя
 - Город (изменение запускает повторную индексацию для гео-поиска)
 - Аватар
 - Номер телефона (требует повторной верификации через SMS)
 - Связанные аккаунты OAuth (можно добавлять/удалять)
 - Email
 - Пользователи не могут удалить свой аккаунт на MVP (чтобы сохранить целостность данных для объявлений и истории модерации). Вместо этого они могут:
 - Деактивировать (профиль скрыт, объявления удалены из показа, невозможно войти в систему)
 - Позже повторно активировать (восстанавливает профиль и объявления)
 - Деактивация не удаляет персональные данные; они сохраняются по юридическим и операционным причинам (например, разрешение споров). Автоматической очистки по времени нет. ПДн удаляются только при **явном стирании** (`erase_user`, проставление `erased_at`) согласно ФЗ-152, по запросу пользователя или действию администратора (миграция 0015; см. `docs/specs/01-identity-domain.md` и data-governance).

5. **Безопасность и конфиденциальность**
 - На MVP не хранятся данные паспорта, ИНН или другие конфиденциальные идентификаторы.
 - Номера телефонов хранятся как **детерминированный HMAC-SHA256(phone, server_pepper)** в `phone_hash` для уникального поиска (НЕ bcrypt — bcrypt солёный/недетерминированный и не может использоваться для поиска); в интерфейсе показываются только последние 4 цифры для верификации.
 - Токены OAuth хранятся зашифрованными и обновляются через API провайдера.
 - Пароли существуют **только для операторских ролей** (ADMIN/MODERATOR), а не для конечных пользователей; при установке хешируются с bcrypt (фактор стоимости 12) в `password_hash`.
 - Все конечные точки аутентификации ограничены по скорости (макс. 5 попыток за 15 минут на IP).
 - Сессионные токены аннулируются при явном выходе, при смене пароля/роли/статуса (операторские аккаунты) или при обнаружении повторного использования токена.
 - Система логирует события аутентификации (успех/неудача) для аудита, но не логирует пароли или токены.

## Пользовательский путь: Регистрация и вход
```mermaid
sequenceDiagram
 participant User
 participant Frontend
 participant Backend (NestJS Auth Module)
 participant SMS Gateway
 participant OAuth Provider
 participant Database

 User->>Frontend: Открывает страницу регистрации
 Frontend->>Backend: Запрашивает варианты регистрации (phone/OAuth)
 Backend->>Frontend: Возвращает список включенных провайдеров OAuth + опция телефона

 %% Путь регистрации по телефону
 User->>Frontend: Вводит номер телефона, выбирает "Зарегистрироваться через SMS"
 Frontend->>Backend: POST /auth/register/phone {phone}
 Backend->>SMS Gateway: Запрашивает SMS-код
 SMS Gateway->>User: Отправляет SMS с кодом
 User->>Frontend: Вводит полученный код
 Frontend->>Backend: POST /auth/verify-phone {phone, code}
 Backend->>Database: Проверяет, существует ли телефон
 alt Телефон не зарегистрирован
 Backend->>Database: Создает новую запись пользователя (хеш телефона, имя, город)
 Backend->>Frontend: Возвращает JWT + токен обновления
 else Телефон уже зарегистрирован
 Backend->>Frontend: Ошибка 409 (Телефон уже используется)
 end

 %% Путь регистрации через OAuth
 User->>Frontend: Нажимает "Зарегистрироваться через Google"
 Frontend->>OAuth Provider: Перенаправляет на вход в OAuth
 User->>OAuth Provider: Вводит учетные данные, соглашается на передачу профиля
 OAuth Provider->>Frontend: Перенаправляет обратно с кодом авторизации
 Backend->>OAuth Provider: Валидирует код, получает информацию о пользователе
 alt Нет существующего пользователя с этим OAuth ID
 Backend->>Database: Создает нового пользователя (OAuth ID, имя от поставщика, город по умолчанию/null)
 Backend->>Frontend: Возвращает JWT + токен обновления
 else Существует пользователь с этим OAuth ID
 Backend->>Frontend: Возвращает JWT + токен обновления (вход)
 end

 %% Поток токена обновления (общий)
 User->>Frontend: Делает запрос с просроченным JWT
 Frontend->>Backend: POST /auth/refresh {refresh_token}
 Backend->>Database: Валидирует токен обновления, проверяет повторное использование/ротацию
 Backend->>Frontend: Возвращает новый JWT + новый токен обновления
```

## Пользовательские истории

### Аутентификация и регистрация
**UC-ID-01:** Как новый пользователь, я хочу быстро зарегистрироваться, используя свой номер телефона или аккаунт социальной сети, чтобы сразу начать пользоваться платформой без лишних усилий.
- Критерии приёмки:
  - Форма регистрации загружается за <1с
  - Подтверждение телефона завершается в течение 30 секунд
  - Вход через соцсети (Google/Apple/Telegram/VK) требует не более 2 кликов
  - Понятные индикаторы прогресса во время верификации
  - Сообщения об ошибках конкретны и побуждают к действию
  - Подтверждение успеха показывается сразу после верификации

**UC-ID-02:** Как возвращающийся пользователь, я хочу входить без затруднений, чтобы получать доступ к своему аккаунту с минимальными усилиями.
- Критерии приёмки:
  - Опция «запомнить меня» для доверенных устройств
  - Биометрический вход там, где доступен (будущее улучшение)
  - Восстановление доступа через phone OTP / повторную привязку OAuth в течение 2 минут (беспарольно — восстановления пароля для конечных пользователей нет)
  - Сохранение сессии между перезапусками браузера (refresh-токен 7 дней)
  - Понятная индикация статуса аутентификации в интерфейсе

**UC-ID-03:** Как пользователь, заботящийся о конфиденциальности, я хочу контролировать, какая личная информация видна, чтобы комфортно пользоваться платформой, сохраняя приватность.
- Критерии приёмки:
  - В публичном профиле показывается только минимально необходимая информация (имя, город, аватар)
  - Номер телефона маскируется (показываются только последние 4 цифры)
  - Email никогда не отображается публично
  - Простой переключатель показа/скрытия ссылок на соцсети после модерации
  - Понятное объяснение, какие данные хранятся, а какие отображаются

### Управление профилем
**UC-ID-04:** Как активный пользователь, я хочу легко обновлять информацию в своём профиле, чтобы моя информация оставалась актуальной и точной.
- Критерии приёмки:
  - Редактирование профиля доступно из главной навигации
  - Валидация ввода в реальном времени (формат телефона, длина имени и т.д.)
  - Мгновенный предпросмотр изменений аватара
  - Деактивация/повторная активация в один клик
  - Диалоги подтверждения для деструктивных действий
  - Уведомление об успехе после обновления профиля

**UC-ID-05:** Как пользователь, обеспокоенный безопасностью аккаунта, я хочу отслеживать и контролировать доступ к своему аккаунту, чтобы предотвратить несанкционированное использование.
- Критерии приёмки:
  - История входов с указанием местоположения/времени/устройства
  - Управление активными сессиями (завершение конкретных сессий)
  - Повторная аутентификация через phone OTP для чувствительных действий (беспарольно — пароля конечного пользователя для проверки нет)
  - Оповещения безопасности о новых устройствах/попытках входа
  - Простое привязывание/отвязывание провайдеров OAuth

## Концептуальная модель данных
| Атрибут | Тип | Обязателен | Описание |
|---------|-----|------------|----------|
| `id` | UUID | Да | Первичный ключ |
| `phone_hash` | VARCHAR(60) | Нет (если OAuth) | Детерминированный HMAC-SHA256(phone, server_pepper) для уникального поиска (НЕ bcrypt) |
| `oauth_google_id` | VARCHAR(255) | Нет | Уникальный ID от Google |
| `oauth_apple_id` | VARCHAR(255) | Нет | Уникальный ID от Apple |
| `oauth_telegram_id` | VARCHAR(255) | Нет | Уникальный ID от Telegram |
| `oauth_vk_id` | VARCHAR(255) | Нет | Уникальный ID от VK |
| `full_name` | VARCHAR(100) | Да | Отображаемое имя |
| `city_id` | INT (FK to city directory) | Да | Для гео-поиска и локализации |
| `avatar_url` | TEXT | Нет | URL аватара в объектном хранилище |
| `email` | VARCHAR(255) | Нет | Для уведомлений (опционально) |
| `email_verified` | BOOLEAN | Нет | True если email подтвержден через ссылку |
| `password_hash` | VARCHAR(60) | Нет (только оператор) | Хеш пароля bcrypt (cost 12); устанавливается только для операторских ролей (ADMIN/MODERATOR) — конечные пользователи беспарольны |
| `role` | ENUM('USER', 'MODERATOR', 'ADMIN', 'BREEDER', 'FARMER', 'VETERINARIAN', 'GROOMER') | Да | По умолчанию: USER. Канон из 7 ролей согласно `docs/specs/security/rbac-matrix.md` / ADR-0011; роли аддитивны (BREEDER/FARMER/VETERINARIAN/GROOMER = USER + дополнительные возможности). `principal_type` (HUMAN\|AGENT) ортогонален `role` (ADR-0006). |
| `is_active` | BOOLEAN | Да | True = может войти; False = деактивирован |
| `created_at` | TIMESTAMP | Да | Время регистрации |
| `updated_at` | TIMESTAMP | Да | Последнее обновление профиля |
| `last_login_at` | TIMESTAMP | Нет | Для отслеживания активности |
| `deactivated_at` | TIMESTAMP | Нет | Когда пользователь выбрал деактивацию |

> **(role-canon sync, normative) — `users.role` приведён к канону из 7 ролей.**
> **ЧТО:** Добавлены `BREEDER` и `FARMER` в enum `role`, так что identity-BR теперь перечисляет все 7 ролей
> `{USER, MODERATOR, ADMIN, BREEDER, FARMER, VETERINARIAN, GROOMER}`, совпадая с `database_schema.sql` и
> `docs/specs/security/rbac-matrix.md`.
> **ПОЧЕМУ:** GAP-TRACE-004 — набор ролей был рассинхронизирован в трёх доках (в identity-BR отсутствовали BREEDER/FARMER).
> Три источника правды по ролям — прямой риск для RBAC, который и есть предмет Admin-домена.
> **ПОЧЕМУ ТАК ЛУЧШЕ для проекта в целом:** один канон ролей (валидированный на PG) означает, что RBAC-матрица, CASL-abilities,
> гварды и миграции опираются на одно множество; additive-модель и `principal_type ⟂ role` (ADR-0006) сохранены.

> **(GAP-TRACE-009, normative) — аутентификация конечных пользователей беспарольная.**
> **ЧТО:** Убраны «пароль обязателен при аутентификации по телефону» и смена/восстановление пароля для конечных пользователей; уточнено, что `password_hash` существует только для операторских ролей (ADMIN/MODERATOR).
> **ПОЧЕМУ:** GAP-TRACE-009 — BR описывал сознательно отвергнутый, несуществующий флоу (`auth-api.yaml` содержит OTP/OAuth, нет `/auth/login`; `database_schema.sql` помечает `password_hash` как operator-only; spec 01 round-4 беспарольный). Устаревший BR вводит в заблуждение новых разработчиков/агентов.
> **ПОЧЕМУ ТАК ЛУЧШЕ для проекта в целом:** Снижает PII/attack surface (нет паролей конечных пользователей для фишинга, хранения или утечки), согласуется с round-4, реализованным Identity-доменом и минимизацией по ФЗ-152.

> **(GAP-TRACE-010, normative) — срок токенов 15m/7d; `phone_hash` — HMAC, а не bcrypt.**
> **ЧТО:** Срок access JWT — 15 минут, refresh — 7 дней (убрана цифра «24ч»); `phone_hash` исправлен с bcrypt на детерминированный HMAC-SHA256(phone, server_pepper). `password_hash` остаётся bcrypt (operator-only) — корректно.
> **ПОЧЕМУ:** GAP-TRACE-010 — внутри `02-requirements/` были две взаимоисключающие цифры по токенам; bcrypt для хэша телефона технически невозможен как ключ уникального поиска (солёный/недетерминированный), что противоречит `database_schema.sql` и глоссарию `phone_hash (HMAC + pepper)`.
> **ПОЧЕМУ ТАК ЛУЧШЕ для проекта в целом:** Один источник правды по сессиям; корректная крипто-модель (HMAC = детерминированный поиск + защита; bcrypt = только проверка паролей), совпадающая со схемой, валидированной на PG.

> **(auto-purge removal, normative) — авто-очистки по времени нет; стирание явное.**
> **ЧТО:** Убрано «очищаются через 30 дней бездействия»; ПДн удаляются только при явном `erase_user` (`erased_at`) согласно ФЗ-152.
> **ПОЧЕМУ:** BR обещал автоматический retention-job, которого не существует; стирание — это явный, аудируемый флоу `erase_user` (миграция 0015, spec 01, data-governance).
> **ПОЧЕМУ ТАК ЛУЧШЕ для проекта в целом:** Честная карта хранения данных; явное стирание безопаснее и аудируемо, и BR больше не обещает автоудаление, которого нет.

## Нефункциональные требования (специфичные для Identity)
- **Производительность**: Аутентификация (вход/регистрация) должна завершаться в течение 2с при нормальной нагрузке.
- **Масштабируемость**: Должна поддерживать 1000 одновременных запросов аутентификации (сессий) без degradation.
- **Доступность**: Сервис аутентификации должен иметь uptime 99.9% (критический путь для всех остальных функций).
- **Безопасность**: 
 - Соответствие OWASP ASVS Level 2 для аутентификации.
 - Защита от brute force, credential stuffing и захвата сессий.
 - Все пароли и токены передаются только по HTTPS.
- **Конфиденциальность**: 
 - Минимизирует сбор ПДн (только номер телефона/OAuth ID, имя, город).
 - Предоставляет возможность экспорта/удаления персональных данных (GDPR/ФЗ-152) – будет реализовано в Фазе 2+ через рабочий процесс по запросу пользователя.
- **Логирование**: События аутентификации (успех/неудача входа, обновление токена, деактивация) логируются для аудита безопасности, но исключают конфиденциальные данные.

## Связанные домены
- **Домен Администрирования**: Управляет ролями пользователей, привилегиями модерации и справочными данными (города, породы).
- **Домен Животных**: Связан с пользователями через `owner_id` (пользователь, создавший профиль животного).
- **Домен Объявлений**: Связан с пользователями через `creator_id` (пользователь, создавший объявление).
- **Домен Сопоставления**: Может использовать предпочтения пользователей (хранящиеся в профиле) для предложения подборов.
