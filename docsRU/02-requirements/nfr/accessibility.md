# Требования к доступности нефункционального характера: ZooLink

## Цель
Определяет требования к доступности, чтобы обеспечить использование платформы людьми с различными возможностями, включая нарушения зрения, слуха, двигательной активности и когнитивные нарушения. Соответствует стандартам WCAG 2.1 AA и учитывает Федеральный закон РФ № 381-ФЗ «О доступности среды обитания для инвалидов».

## Область применения
Применяется ко всем пользовательским компонентам: веб-приложение (SPA/PWA), документация и рекомендации по пользовательскому контенту.

## Принципы доступности (POUR)
1. **Воспринимаемая**: Информация и компоненты интерфейса должны представляться таким образом, чтобы пользователи могли их воспринимать.
2. **Управляемая**: Компоненты интерфейса и навигация должны быть управляемыми.
3. **Понятная**: Информация и работа интерфейса должны быть понятными.
4. **Надёжная**: Содержание должно быть достаточно надёжным, чтобы его можно было надёжно интерпретировать широким спектром пользовательских агентов.

## Целевой уровень соответствия WCAG 2.1 AA
Платформа нацелена на достижение критериев успеха WCAG 2.1 уровня AA. Ключевые области фокуса:

### 1. Воспринимаемая
- **Текстовые альтернативы**:
 - Все не декоративные изображения имеют значимый альтернативный текст
 - Иконки имеют доступные имена (aria-label или видимый текст)
 - Сложные изображения (графики, диаграммы) имеют более длинные описания при необходимости
 - Аудио контент имеет транскрипты (в будущем: для видео контента)
 - Видео контент имеет subtitles и аудио описания (планируется для Фазы 2+)

- **Времязависимые медиа**:
 - Нет автозапуска аудио/видео дольше 3 секунд без элементов управления
 - Приостанавливаемый, останавливаемый, скрываемый для движущегося/мигающего/прокручивающегося контента
 - Нет контента, который мигает более 3 раз в секунду (или ниже общих flash и red flash порогов)

- **Адаптируемая**:
 - Информация и структура могут быть определены программно (правильный семантический HTML)
 - Логический порядок чтения и навигации
 - Инструкции не зависят исключительно от сенсорных характеристик (например, «нажмите красную кнопку»)
 - Содержание не ограничивает вид/операции одной ориентацией дисплея (поддерживаются как portrait, так и landscape)

- **Отличимая**:
 - Цвет не используется как единственный визуальный способ передачи информации
 - Коэффициент контрастности текста: минимум 4.5:1 для обычного текста, 3:1 для большого текста
 - Большой текст (18pt+ или 14pt bold) соответствует коэффициенту 3:1
 - UI компоненты и графические объекты: коэффициент контрастности 3:1
 - Текст можно увеличивать до 200% без потери содержимого или функциональности
 - Интервал текста (межстрочный интервал, интервал между абзацами, интервал между буквами, интервал между словами) можно регулировать
 - Содержание не требует горизонтальной прокрутки при ширине 320px эквивалент
 - Переопределения интервала текста не вызывают потери содержимого или функциональности
 - Содержание при наведении/фокусе: отклоняемое, наводимое, постоянное

### 2. Управляемая
- **Доступная с клавиатуры**:
 - Все функции доступны через клавиатуру
 - Нет ловушек клавиатуры (фокус можно переместить только с помощью клавиатуры)
 - Видимый индикатор фокуса для всех интерактивных элементов
 - Логический порядок tab,following визуальному порядку
 - Ссылки пропуска навигации доступны (для обхода повторяющегося контента)
 - Клавиши доступа не используются (чтобы избежать конфликтов с вспомогательными технологиями)

- **Достаточно времени**:
 - Для каждого временного ограничения, установленного контентом, пользователи могут:
 - Выключить его
 - Отрегулировать его
 - Увеличить его
 - Исключения: события в реальном времени (LIVE-аукционы - Фаза 2+), необходимое время
 - Пользователи получают предупреждение перед истечением времени и возможность его продлить
 - Нет контента, предназначенного для вызова припадков или физических реакций

- **Припадки и физические реакции**:
 - Нет контента, который мигает более 3 раз в секунду
 - Общий порог вспышки: не более 3 общих вспышек
 - Красный порог вспышки: не более 3 красных вспышек

- **Навигабельная**:
 - Есть способ обойти повторяющиеся блоки контента (ссылки пропуска)
 - Заголовки страниц описательные и информативные
 - Порядок фокуса логичен и сохраняет смысл
 - Цель ссылки понятна из текста ссылки или контекста
 - Множество способов найти страницы (навигация, поиск, карта сайта)
 - Заголовки и метки описывают тему/назначение
 - Видимый индикатор фокуса
 - Горячие клавиши (если предоставляются) можно переназначить или выключить

### 3. Понятная
- **Читаемая**:
 - Язык страницы идентифицируем (атрибут lang на элементе html)
 - Язык частей идентифицируем (для многоязычного контента в Фазе 2+)
 - Текст читабелен и понятен (избегайте жаргона, объясняйте аббревиатуры)
 - Необычные слова определены (через глоссарий или всплывающую подсказку)
 - Аббревиатуры развернуты (при первом использовании)
 - Уровень чтения учитывается (стремление к четкому, простому языку)

- **Предсказуемая**:
 - При фокусе: не инициирует изменение контекста
 - При вводе: не инициирует изменение контекста, если ранее не объяснялось
 - Согласованные механизмы навигации
 - Согласованная идентификация функциональных элементов с одинаковой функцией

- **Ввод помощи**:
 - Ошибки идентифицируются и описываются в тексте
 - Предлагаются предложения по ошибкам, когда возможно
 - Предотвращение ошибок (Юридическое, Финансовое, Изменения данных): обратимые, проверенные, подтвержденные
 - Контекстуальная и доступная помощь
 - Инструкции предоставляются для сложных форм

### 4. Надёжная
- **Совместимая**:
 - Максимизировать совместимость с текущими и будущими пользовательскими агентами
 - Валидный HTML там, где это возможно (минимизировать ошибки валидации, влияющие на доступность)
 - Имя, роль, значение: для всех UI компонентов
 - Сообщения о статусе: программно определяемы через role или aria-live
 - Нет зависимости от конкретной технологии или браузера

## Требования к доступности, специфичные для платформы
### Веб-приложение (SPA/PWA)
- **Семантический HTML**:
 - Используйте правильные элементы HTML5: header, nav, main, section, article, aside, footer
 - Уровни заголовков (h1-h6) используются иерархически и последовательно
 - Списки (ul, ol, dl) используются для группировки связанных элементов
 - Таблицы используются только для табличных данных с правильными th, caption, scope
 - Формы используют элементы label правильно связанные с вводами

- **Использование ARIA**:
 - Используйте ARIA только тогда, когда нативный HTML недостаточен
 - Предпочитайте нативные элементы вместо ролей ARIA (например, используйте button вместо div role="button")
 - Аттрибуты ARIA валидируются на корректность
 - Живые регионы используются соответствующим образом для динамического контента (вежливые/настойчивые)
 - Модальные диалоги: захват фокуса, возврат фокуса к триггеру, фон инертный
 - Навигационные ориентиры: banner, navigation, main, complementary, contentinfo
 - Виджеты раскрытия: используйте паттерн раскрытия с aria-expanded

- **Навигация с клавиатуры**:
 - Все интерактивные элементы доступны и управляемы через клавиатуру
 - Пользовательские виджеты следуют практикам авторства ARIA
 - Ссылка "Пропустить основной контент" присутствует и функциональна
 - Фокус клавиатуры видим и чёткий (минимум 2px контраст, не только outline:none)
 - Управление фокусом: при открытии диалогов, меню и т.п. фокус перемещается соответствующим образом
 - Фокус возвращается к логическому месту после закрытия
 - Нет положительных значений tabindex (используйте 0 или -1 только при необходимости)

- **Цвет и контраст**:
 - Текст и интерактивные элементы соответствуют коэффициентам контрастности
 - UI компоненты (кнопки, поля ввода и т.д.) имеют коэффициент контрастности 3:1 относительно смежных цветов
 - Инструкции не зависят исключительно от цвета (например, "обязательные поля отмечены красным")
 - Учет дальтонизма: используйте узоры, иконки, текст в дополнение к цвету
 - Тёмный/светлый режим: обе темы соответствуют требованиям контрастности (подлежит оценке)

- **Типографика и читабельность**:
 - Базовый размер шрифта минимум 16px (или значение браузера по умолчанию)
 - Межстрочный интервал минимум 1.5 для абзацев
 - Интервал между абзацами минимум 2x размер шрифта
 - Шрифты выбраны для читабельности (беззасечные предпочтительны для UI)
 - Выравнивание текста: выравнивание по левому краю для языков LTR (центрирование/выравнивание по ширине избегаются для больших блоков)
 - Интервал между буквами и словами регулируется через пользовательскую таблицу стилей
 - Избегайте изображений текста; используйте реальный текст, который можно масштабировать и читать скринридерами

- **Формы и вводы**:
 - Каждое поле формы имеет связанную метку
 - Обязательные поля указываются текстом и/или aria-required (не только цветом)
 - Сообщения об ошибках связаны с вводами через aria-describedby или живой регион
 - Ограничения ввода описаны (min/max, pattern) доступным способом
 - Атрибуты автозаполнения используются соответствующим образом
 - Fieldsets и легенды используются для группировки связанных полей
 - Предотвращение ошибок: подтверждение для необратимых действий
 - Кнопки отправки и сброса различимы
 - Цели касания минимум 44x44 CSS пикселя (с отступами)

- **Медиа и непростой контент**:
 - Альтернативный текст: описательный, не избыточный, пустая строка для декоративного (alt="")
 - Сложные изображения: описывайте основную информацию
 - Инфографика: предоставьте таблицу данных или резюме
 - Иконки: доступные имена, SVG предпочтительно с title/desc
 - Аудио: контроль, регулировка громкости, отсутствие автозапуска
 - Видео: элементы управления, subtitles, транскрипты, аудио описание (Фаза 2+)
 - Анимированный контент: может быть приостановлен, остановлен, скрыт
 - Эмодзи: используется умеренно, с пониманием, что они могут быть прочитаны буквально

- **Адаптивность и сенсорный ввод**:
 - Содержание правильно перетекает при ширине 320px
 - Цели касания: минимум 44x44 dp с достаточным интервалом
 - Жесты: предпочтительно простой tap/click; сложные жесты имеют альтернативы
 - Движение как ввод: предоставляются альтернативы для движения устройства/движения пользователя
 - Указательные жесты: однопunktовые действия, где возможно

### Руководства по пользовательскому контенту
- **Описания объявлений**:
 - Поощряйте четкие, лаконичные описания
 - Предлагайте избегать чрезмерных аббревиатур или жаргона
 - Рекомендуйте размещать критически важную информацию в тексте (а не только в фотографиях)
 - Рекомендации предоставляются в интерфейсе: "Опишите животное четко: порода, возраст, состояние здоровья"

- **Руководства по фотографиям**:
 - Поощряйте фотографии, которые четко показывают животное
 - Предлагайте несколько ракурсов для скота (конституция, вымя/яички, боковые виды)
 - Предупреждайте о вводящих в заблуждение фотографиях (стоковые изображения, несвязанные животные)
 - Генерация альтернативного текста: система будет запрашивать описание или использовать предложение ИИ (Фаза 2+)

- **Коммуникация**:
 - Поощряйте четкое, уважительное общение
 - Рекомендации против домогательств или дискриминационного языка
 - Доступный механизм сообщения об оскорбительном контенте

## Тестирование и валидация доступности
### Автоматическое тестирование
- **Инструменты**: axe-core, Lighthouse, pa11y или аналогичные, интегрированные в пайплайн CI
- **Частота**: При каждом pull request и ночью против главной ветки
- **Правила**: WCAG 2.1 AA, Раздел 508, EN 301 549
- **Порог отказа**: Нет новых критических или серьезных нарушений; существующие нарушения требуют плана исправления
- **Ручной обзор**: Автоматические инструменты дополняются ручным тестированием

### Ручное тестирование
- **Навигация с клавиатуры**:
 - Последовательно переберите весь приложение через tab
 - Протестируйте все пользовательские виджеты (модальные окна, выпадающие списки, выбор даты)
 - Проверьте порядок фокуса и его видимость
 - Протестируйте ссылки пропуска и навигацию по ориентирам
- **Тестирование скринридером**:
 - Основное: NVDA (Windows) + Firefox или JAWS + IE/Edge
 - Вторичное: VoiceOver (macOS/iOS) + Safari
 - Протестируйте обычные пользовательские пути: регистрация, создание/поиск объявлений, раскрытие контактов
 - Проверяйте: объявление динамического контента, правильная маркировка, логический порядок чтения
- **Контрастность цвета**:
 - Проверки вручную с использованием проверяльщиков контрастности
 - Проверка текста поверх изображений/градиентов
 - Видимость индикатора фокуса
- **Увеличение и изменение размера текста**:
 - Тестирование при 200% зумме (нет потери содержимого/функциональности)
 - Тестирование только увеличения текста (настройка браузера)
 - Проверьте перетекание и читабельность
- **Мобильная доступность**:
 - Тестирование с TalkBack (Android) и VoiceOver (iOS)
 - Проверьте размер целей касания и интервалов
 - Тестирование изменения ориентации (портрет/ландшафт)
 - Проверьте адаптивные контрольные точки

### Пользовательское тестирование
- **Вовлекайте людей с ограниченными возможностями**:
 - Набирайте пользователей с различными нарушениями для тестирования удобства использования
 - Фокус-группы для обратной связи по функциям доступности
 - Рекомендуются платные тестовые сессии
- **Сценарии для тестирования**:
 - Регистрация и настройка профиля
 - Создание и редактирование профилей животных
 - Создание, модерирование и просмотр объявлений
 - Поиск и фильтрация результатов
 - Процесс раскрытия контактов
 - Рабочий процесс модератора (если применимо)
 - Доступ к помощи и поддержке

## Роли и ответственности
### Дизайнеры
- Убедитесь, что wireframes и макеты учитывают доступность с самого начала
- Указывайте порядки фокуса, альтернативный текст для изображений, доступные цветовые палитры
- Документируйте соображения доступности в системе дизайна
- Проверяйте цветовой контраст в макетах

### Разработчики
- Внедряйте семантический HTML и правильное использование ARIA
- Убедитесь в управляемости с клавиатуры всех интерактивных компонентов
- Валидируйте доступность в браузерах и вспомогательных технологиях
- Исправляйте ошибки доступности с тем же приоритетом, что и функциональные ошибки
- Используйте линтеры доступности в рабочем процессе разработки

### Создатели контента
- Предоставляйте значимый альтернативный текст для изображений, которые они загружают
- Пишите четкий, лаконичный и понятный текст
- Используйте заголовки правильно в более длинных описаниях
- Убедитесь, что связанный текст имеет смысл вне контекста

### QA/тестировщики
- Включайте тестовые случаи доступности в тестовые планы
- Выполняйте тестирование только с клавиатуры и скринридером
- Сообщайте о дефектах доступности с той же серьёзностью, что и о функциональных дефектах
- Проверяйте исправления и регрессии

### Владельцы продукта
- Приоритизируйте функции доступности в бэклоге
- Убедитесь, что критерии приемлемости доступности присутствуют в пользовательских историях
- Выделяйте время на тестирование доступности и исправление
- Отстаивайте доступность в дискуссиях со стейкхолдерами

## Исключения и альтернативный доступ
### Документированные исключения
- Любое исключение из WCAG 2.1 AA должно быть:
 - Документировано с обоснованием
 - Одобрено владельцем доступности/владельцем продукта
 - Сопровождаться планом эквивалентного обеспечения
 - Периодически пересматриваться
- Примеры потенциальных исключений:
 - Сложные исторические визуализации данных (может предоставить табличную альтернативу)
 - Встроенные сторонние элементы с ограниченным контролем (может предоставить ссылку на доступную версию)
 - Функции совместной работы в реальном времени (может иметь асинхронную альтернативу)

### Альтернативные методы доступа
- Если определённые функции нельзя сделать доступными, обеспечьте эквивалентное обеспечение:
 - Телефонная поддержка для выполнения действий
 - Электронная почта-based workflow
 - Помощь через посредника (с учётом конфиденциальности)
 - Примечание: Альтернатива должна обеспечивать эквивалентную возможность, а не уступающий сервис

## Дорожная карта доступности
### MVP (Фаза 1)
- Семантический HTML и правильная структура заголовков
- Навигация с клавиатуры для всей основной функциональности
- Достаточная цветовая контрастность (WCAG AA)
- Доступные формы с правильной маркировкой
- Управление фокусом для модальных окон и меню
- Ссылка "Пропустить основной контент"
- Альтернативный текст для всех значимых изображений
- Адаптивный дизайн, который правильно перетекает при ширине 320px
- Цели касания минимум 44x44px
- Базовое тестирование скринридером

### Фаза 2 (Рост)
- Расширенные паттерны ARIA для сложных виджетов
- Живые регионы для динамических обновлений контента
- Subtitles для видео контента (если добавлен)
- Улучшенные видимые индикаторы фокуса
- Настраиваемый интервал текста (через настройки пользователя)
- Учёт жестового языка для видео контента (Фаза 2+)
- Расширенное тестирование скринридером с несколькими инструментами/языками
- Опубликовано заявление о доступности с контактом для обратной связи

### Фаза 3 (Зрелость)
- Параметры персонализации (темы контраста, размеры шрифтов и т.д.)
- Расширенные горячие клавиши (с избеганием конфликтов)
- Реализация Руководств по авторству WAI-ARIA 1.2
- Регулярные аудиты доступности третьей стороной
- Рассмотрение сертификации доступности (VPAT и т.д.)
- Инклюзивные практики дизайна интегрированы в жизненный цикл
- Плановое тестирование пользователей с разнообразными группами ограниченных возможностей
