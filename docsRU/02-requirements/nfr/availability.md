# NFR: Доступность (NFR-AVAIL)

Ссылается из `specs/14-payment-domain.md`, `specs/15-api-gateway-domain.md` и др.

## Цели (MVP, Фаза 1)
- **Аптайм:** 99.5% в месяц (без анонсированного обслуживания) — макс. ~3.6 ч простоя/мес.
- **Плановое обслуживание:** анонс ≥24 ч заранее; окно вне пиков.
- **RPO (потеря данных):** ≤ 24 ч в MVP (ежедневный `pg_dump`); ужесточается WAL-архивированием при необходимости.
- **RTO (восстановление):** ≤ 4 ч для полного восстановления на одной VM MVP.

## Механизмы (MVP)
- Stateless API за reverse proxy; ≥2 реплики `api`, чтобы краш/редеплой был не фатален.
- Compose `restart: unless-stopped` + healthcheck (`/health/live`, `/health/ready`); нездоровые контейнеры перезапускаются.
- Грациозная деградация: сбой внешних провайдеров (SMS/email/карты/платежи) не должен ронять основные потоки —
  очередь, ретраи с backoff, понятная ошибка (см. `error_handling/standard_error_format.md`).
- Идемпотентность на write-путях с внешними провайдерами (платежи) для устойчивости к ретраям.
- Ежедневные off-box бэкапы (Yandex Object Storage); протестированное восстановление (см. `06-operations/deployment-mvp.md`).

## Фаза 2+
Read-реплики, мульти-зональный Kubernetes, автоматический failover (реплика→primary), cross-region бэкапы, цель
99.9%+ — см. `deployment-diagram.md` (Target State) и `06-operations/disaster-recovery-plan.md`.

## Верификация
- Синтетический uptime-монитор на `/health/ready`.
- Ежеквартальная тренировка восстановления из бэкапа.
- Chaos-проверка: убить одну реплику `api` под нагрузкой → без видимого простоя.
