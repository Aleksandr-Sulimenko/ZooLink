# Требования к безопасности нефункционального характера: ZooLink

## Цель
Определяет требования к безопасности для защиты данных пользователей, предотвращения несанкционированного доступа, обеспечения соблюдения нормативов (ФЗ-152, принципы GDPR) и поддержания целостности платформы.

## Область применения
Применяется ко всем компонентам системы: backend API, frontend приложения, базы данных, файловое хранилище и интеграции.

## Основные принципы безопасности
1. **Защита в глубине**: Многоуровневые элементы контроля безопасности.
2. **Наименьшие привилегии**: Пользователи и сервисы имеют только те разрешения, которые им необходимы.
3. **Безопасный отказ**: По умолчанию запрещать доступ в случае ошибки.
4. **Полное посредничество**: Каждый запрос на доступ проверяется на авторизацию.
5. **Экономия механизма**: Держать механизмы безопасности простыми и небольшими.
6. **Открытый дизайн**: Безопасность не зависит от секретности реализации.
7. **Психологическая приемлемость**: Меры безопасности не должны чрезмерно препятствовать удобству использования.

## Требования к аутентификации
### Политика паролей (если используется аутентификация по паролю)
- Минимальная длина: 12 символов
- Требовать сочетание заглавных, строчных букв, цифр и специальных символов
- Предотвращать повторное использование последних 5 паролей
- Блокировать учетную запись после 5 неудачных попыток на 15 минут
- Хранить пароли с использованием bcrypt с фактором стоимости ≥12
- Никогда не хранить и не передавать пароли в открытом тексте

### Многофакторная аутентификация (MFA)
- **Планируется для Фазы 2+**: 
 - Опциональная TOTP (Google Authenticator и т.д.) для всех пользователей
 - Обязательная для роли ADMIN и пользователей с повышенными привилегиями
- **MVP**: Не требуется, но инфраструктура подготовлена (поле MFA в схеме пользователя)

### Управление сессиями
- Токены сессий (JWT) с коротким сроком жизни: 15 минут для токена доступа
- Ротация токена обновления: выдается новый токен обновления при каждом использовании, старый аннулируется
- Срок жизни токена обновления: 7 дней
- Безопасные флаги cookie: HttpOnly, Secure, SameSite=Strict
- Инвалидация сессии при изменении пароля, изменении роли или явном выходе
- Ограничение одновременных сессий: 5 активных сессий на пользователя (настраиваемо)
- Хранилище сессий: зашифровано в Redis с TTL, соответствующим токену обновления

### Безопасность API
- Все эндпоинты API требуют действительной аутентификации (кроме публичной регистрации/входа)
- Ограничение скорости: 
 - Эндпоинты аутентификации: 5 попыток за 15 минут на IP
 - Общий API: 100 запросов в минуту на пользователя (всплеск 200)
 - Действия модерации: 10 действий в минуту на модератора
 - Раскрытие контактов: 10 раскрытий в час на пользователя (домашнее животное), 5 в час (скот)
- Валидация ввода: 
 - Все входные данные проверяются по строгим схемам (подход белого списка)
 - Защита от SQL-инъекций (через ORM/параметризованные запросы)
 - Защита от XSS (кодирование вывода во frontend)
 - Защита от CSRF (SameSite cookie и double-submit cookie для операций, изменяющих состояние)
 - Журналирование аудита: 
 - Все события аутентификации (успех/неудача)
 - Все попытки эскалации привилегий
 - Все доступы и модификации данных привилегированными ролями
 - Журналы исключают конфиденциальные данные (пароли, токены, полная PII)

## Требования к авторизации
### Управление доступом на основе ролей (RBAC)
- Четкое разделение между ролями USER, MODERATOR, ADMIN
- Разрешения определены на роль и ресурс (см. детали в документах доменов)
- Динамические разрешения: некоторые разрешения основаны на праве собственности (например, может редактировать собственный профиль)
- Проверки разрешений выполняются на уровне API шлюза и уровня сервиса
- Принцип наименьших привилегий: начинать без разрешений, предоставлять только необходимое

### Управление доступом к данным
- Пользователи могут иметь доступ только к:
 - Собственному профилю и животным
 - Объявлениям, которые они создали (если ОПУБЛИКОВАНО, то публично с ограничениями)
 - Публичным объявлениям (только статус ОПУБЛИКОВАНО)
 - Очереди модерации и действиям только если роль MODERATOR или ADMIN
 - Авторизация на уровне объекта: 
 - Перед возвратом ресурса проверять, имеет ли запрашивающий пользователь право доступа к конкретному экземпляру
 - Например: пользователь может обновлять только свои собственные объявления, а не чужие
 - Безопасные прямые ссылки на объекты: использовать косвенные ссылки (UUID) и проверять принадлежность

## Требования к защите данных
### Шифрование данных в состоянии покоя
- **База данных**: 
 - Включить Transparent Data Encryption (TDE), если используется управляемая служба PostgreSQL
 - Для самов управлением: рассмотреть шифрование на уровне файловой системы или pgcrypto для чувствительных столбцов
 - Чувствительные данные для шифрования: 
 - Хеши номера телефона (уже хешированы)
 - Токены обновления (зашифрованы)
 - OAuth токены (зашифрованы)
 - Адреса электронной почты (считаются PII, шифровать, если хранится долгосрочно)
- **Хранилище файлов**:
 - Шифрование на стороне сервера (SSE-S3 или SSE-KMS) для объектного хранилища
 - Шифрование на стороне клиента не требуется для MVP (изображения не являются конфиденциальными)
 - Будущее рассмотрение: зашифрованное хранилище для документов (Контракты, Medical records) в Фазе 2+
- **Резервные копии**: 
 - Зашифрованные резервные копии с отдельным управлением ключами
 - Период хранения согласуется с политикой хранения данных (например, 30 дней для активных, 1 год для аудитов)

### Шифрование данных в передаче
- **TLS 1.2+** обязателен для всех внешних коммуникаций
- Перенаправление HTTPS для всех HTTP-запросов
- Надежные наборы шифров: приоритет TLS_ECDHE_* с PFS
- Заголовок HSTS с max-age=31536000 (1 год) и includeSubDomains
- Внутренняя коммуникация сервисов: 
 - Внутри одного сегмента сети: может разрешаться открытый текст, если физически изолировано
 - Межсетевое или через общедоступные сети: всегда TLS

### Минимизация и хранение данных
- Собирать только необходимые данные для заявленной цели (см. модели доменов)
- Личные данные (PII): 
 - Номер телефона (хеширован для поиска)
 - Электронная почта (если предоставлена)
 - Имя
 - Город (для geo-поиска)
 - IP-адрес (журналируется для безопасности, хранится 30 дней)
- График хранения данных:
 - Данные пользователя: сохраняются до удаления учетной записи + 30 дней льготного периода
 - Данные объявлений: сохраняются до удаления/завершения + 60 дней
 - Журналы модерации: хранятся 1 год
 - Аудит-журналы: хранятся 2 лет (по требованиям соответствия)
 - Данные аналитики: агрегируются и анонимизируются через 90 дней
- Анонимизация/Псевдонимизация: 
 - Для аналитики: удалить или хешировать PII перед хранением
 - Для тестирования/разработки: использовать синтетические или маскированные данные
 - Право на стирание по ГДПР/ФЗ-152: реализовать рабочие процессы удаления в Фазе 2+

## Управление уязвимостями
### Сканирование зависимостей
- Автоматическое сканирование уязвимостей для:
 - Пакетов Node.js/npm (с использованием npm audit или аналогичного)
 - Docker-образов (с использованием Trivy или аналогичного)
 - Инфраструктуры как кода (сканирование Terraform/CloudFormation)
 - Частота: при каждом pull request и еженедельно для главной ветки
 - Критические уязвимости: должны быть исправлены в течение 48 часов
 - Высокие уязвимости: в течение 1 недели
 - Средние/Низкие: решаются в следующем регулярном цикле

### Тестирование безопасности
- **Тесты на проникновение**: 
 - Проводятся передmajor релизами (Фаза 2+, 3+)
 - Выполняются квалифицированной третьей стороной или внутренней красной командой
 - Обхват: сеть, приложение, API, аутентификация
- **Статическое тестирование безопасности приложений (SAST)**:
 - Интегрировано в пайплайн CI (например, SonarQube, Semgrep)
 - Правила: OWASP Top 10, CWE/SANS Top 25
- **Динамическое тестирование безопасности приложений (DAST)**:
 - Планируется регулярно (например, еженедельно) в среде staging
 - Инструменты: OWASP ZAP, Nikto
- **Проверка зависимостей**: 
 - Как часть пайплайна сборки, помечать известные уязвимые библиотеки

### Реагирование на инциденты
- **Мониторинг**: 
 - Оповещения в реальном времени о:
 - Множестве неудачных попыток входа
 - Событиях эскалации привилегий
 - Необычных паттернах доступа к данным
 - Оповещениях WAF (Web Application Firewall)
- **План реагирования**: 
 - Идентификация: автоматические оповещения + ручная триаж
 - Сдерживание: изолировать затронутые системы, отозвать скомпрометированные токены
 - Устранение: удалить вредоносное ПО, патчить уязвимости
 - Восстановление: восстановить из чистых резервных копий, мониторинг на повторение
 - Извлеченные уроки: пост-инцидентный отчет и обновление контролов
 - План коммуникации: 
 - Внутренний: команда реагирования на инциденты, управление
 - Внешний: затронутые пользователи, регуляторы (при нарушении персональных данных по ФЗ-152/GDPR)
 - Сроки: уведомить регулятора в течение 72 часов с момента осознания (ФЗ-152), пользователей без необоснованной задержки

## Требования к соответствию
### ФЗ-152 (Персональные данные)
- Законное основание для обработки: согласие (при регистрации) и законный интерес (для предоставления услуги)
- Права субъекта данных реализованы через:
 - Право на доступ: экспорт персональных данных в машиночитаемом формате (Фаза 2+)
 - Право на исправление: пользователи могут редактировать свой профиль
 - Право на стирание: удалить учетную запись и связанные данные (Фаза 2+)
 - Право на ограничение обработки: пользователи могут деактивировать учетную запись
 - Право на возражение: пользователи могут отказаться от необязательной обработки (например, аналитики)
 - Роль ответственного за защиту данных (DPO) assigned для надзора за соблюдением
 - Политика конфиденциальности и Условия использования опубликованы и связаны в подвале
 - Механизмы согласия: гранулярные согласия для различных видов обработки (маркетинг, аналитика и т.д.)

### Принципы GDPR (для будущего расширения в ЕС)
- Аналогично ФЗ-152 с дополнительными соображениями:
 - Механизмы передачи данных для трансграничных передач (если применимо)
 - Конфиденциальность по проекту и по умолчанию: интегрирован в жизненный цикл разработки
 - Оценки воздействия на защиту данных (DPIA) для обработки высокого риска
 - Журналы видов обработки данных (ROPA) поддерживаются

### Отраслевые специфические
- **Ветеринарная/Благосостояние животных**: 
 - Нет содействия незаконной торговле животными (например, исчезающие виды)
 - Модераторы обучены замечать проблемы благосостояния в объявлениях
 - Сотрудничество с органами власти по законному запросу
- **Финансовая** (для будущей монетизации): 
 - Соответствие PCI DSS при непосредственной обработке платежей картой (планируется использовать сертифицированные шлюзы платежей)
 - Рассмотрение AML/KYC для транзакций высокой стоимости (скот)

## Надежные методы разработки
### Обучение
- Все разработчики проходят основное обучение по безопасности (OWASP Top 10, безопасное кодирование)
- Ежегодное обновление обучения
- Специализированное обучение для аутентификации, криптографии и обработки данных

### Моделирование угроз
- Проводится для каждой основной функции перед разработкой
- Использует методологию STRIDE или PASTA
- Результаты информируют требования безопасности и тест-кейсы

### Рецензирование кода
- Вопросы безопасности являются частью чеклиста рецензирования pull-запросов
- Ищите: недостатки внедрения, обход аутентификации, небезопасные прямые ссылки на объекты, раскрытие конфиденциальных данных
- Использование средств контроля безопасности (linters) в пайплайне CI

### Управление зависимостями
- Частный реестр npm или прокси для одобренных пакетов
- Автоматические обновления для патчей безопасности
- Мониторинг базы уязвимостей

### Разделение окружений
- Отдельные окружения: Разработка, Тестирование, Staging, Producción
- Нет данных продакшена в непродукционных окружениях (или маскированные/anonymized, если используются)
- Строгие элементы управления доступом к окружению production
- Инфраструктура как код (IaC) для воспроизводимых окружений

## Неотрицаемость и аудит
### Журналирование аудита
- Неизменяемый аудиторский след для событий, связанных с безопасностью:
 - Аутентификация (успех/неудача, попытки MFA)
 - Авторизация (проверки разрешений, изменения ролей)
 - Доступ к данным (чтение/запись/удаление привилегированными пользователями)
 - Изменения конфигурации (настройки системы, переключатели функций)
 - События интеграции (использование ключей API, вызовы третьих сторон)
 - Формат журнала: JSON со стандартными полями (timestamp, тип события, актер, действие, результат, детали)
 - Хранение журналов: централизованное, защищенное от подделки (запись один раз или подписанные журналы)
 - Хранение журналов: минимум 1 год для журналов безопасности, 2 года для аудиторских журналов
 - Мониторинг журналов: оповещения типа SIEM для корреляций (например, неудачный вход followed by success с другого IP)

### Цифровые подписи (разумное будущее рассмотрение)
- Для критически важных документов (контракты, медицинские сертификаты) в Фазе 2+:
 - Рассмотреть цифровые подписи для непротиворечивости
 - Инфраструктура для управления ключами и валидации

## Тестирование и валидация безопасности
### Регулярные сканирования
- Ежемесячные сканирования уязвимостей производственной инфраструктуры
- Квартальные тесты на проникновение (или в соответствии с циклом релизов)
- Ежемесячные обновления правил веб-приложенного межсетевого экрана (WAF)

### Проверки соответствия
- Ежегодный обзор по соблюдению требований ФЗ-152
- Внешний аудит (если требуется регуляторами или партнерами) облегчается поддерживаемой документацией

### Программа вознаграждения за обнаружение ошибок
- Рассмотреть запуск программы ответственного раскрытия после MVP (Фаза 2+)
- Четкая политика: область применения, вознаграждения, безопасная гавань, канал коммуникации

## Исключения и принятие риска
- Любое исключение из этих требований должно быть задокументировано, утверждено офицером по информационной безопасности и периодически пересматриваться.
- Известные риски, принятые для MVP с планами по снижению:
 - Ограниченная доступность MFA (планируется для Фазы 2+)
 - Отсутствие специализированного WAF на MVP (надежда на защиту на уровне приложения и базовую защиту поставщика облака)
 - Базовый мониторинг безопасности (оповещения о сбоях аутентификации) вместо полной SIEM (планируется для Фазы 2+)
