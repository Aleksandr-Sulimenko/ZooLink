# Спецификация: Домен администрирования

## Результат
Предоставить административные функции для управления платформой, включая модерацию пользователей, модерацию объявлений, управление справочными данными и конфигурацию системы. Обеспечить соответствие российским нормам относительно модерации контента и обработки данных.

## Область и границы
**Включает:**
- Управление пользователями: просмотр пользователей, изменение ролей, бан/разбан, проверка личности
- Модерация объявлений: очередь на проверку, одобрение/отклонение объявлений, добавление причин отклонения
- Управление справочными данными: виды, породы, города, типы объявлений, состояния здоровья
- Конфигурация системы: правила модерации, ограничения частоты запросов, переключатели функций
- Журналы аудита: отслеживание действий модерации, действий пользователей, изменений системы
- Интеграция со всеми другими доменами (Идентификация, Животное, Рынки, Подбор)
- Соответствие российскому законодательству относительно модерации информации и 152-ФЗ

**Исключает:**
- Финансовые/административные системы выставления счетов - отложено на этап 2
- Расширенная аналитика/дашборд - отложено
- Инструменты настройки производительности системы - отложено
- Юридические/инструменты автоматизации соответствия - отложено

## Ограничения
- **Юридическое:** Должно соответствовать Федеральному закону РФ "О информации, информационных технологиях и защите информации" (149-ФЗ) относительно модерации контента, и 152-ФЗ для обработки персональных данных в административных действиях.
- **Безопасность:** Административные действия должны быть аудированы и ограничены только авторизованными ролями. Защита от повышения привилегий.
- **Производительность:** Очередь модерации должна загружаться быстро (<2s) даже при больших объемах.
- **Масштабируемость:** Поддержка рабочего процесса модерации для 1000+ ежедневных действий.
- **Технологии:** Должно соответствовать выбранному стеку (NestJS, TypeScript, PostgreSQL).
- **Удобство использования:** Интерфейс должен быть понятен для модераторов (может не быть техническими экспертами).

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Очередь модерации загружается за <2s с 10k ожидающими элементами (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Административные действия полностью аудированы; обработка персональных данных следует 152-ФЗ; модерация контента следует 149-ФЗ (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Интерфейс должен быть понятен для модераторов; следует рекомендациям WCAG 2.1 AA (см. docs/02-requirements/nfr/accessibility.md)

## User Stories

### Управление администрированием и модерацией
**UC-AD-01:** Как модератор, я хочу легко управлять пользователями, чтобы я мог поддерживать безопасную и соответствующую требованиям платформу.
- Критерии приёмки:
  - Управление пользователями загружается за <2с
  - Поиск и фильтрация пользователей по роли, статусу, дате регистрации
  - Изменение ролей в один клик (пользователь в модератора, модератор в администратора и т.д.)
  - Бан/разбан пользователей с необязательной причиной и длительностью
  - Проверка личности с загрузкой документов (будущее улучшение)
  - Система уведомлений для пользователей, когда против них предпринимаются действия
  - Журнал аудита всех действий модерации для подотчётности

**UC-AD-02:** Как модератор, я хочу эффективно модерировать объявления, чтобы я мог обеспечить качество, соответствие требованиям и безопасность на рынке.
- Критерии приёмки:
  - Очередь модерации загружается за <2с с чёткими индикаторами статуса (ожидает, одобрено, отклонено)
  - Опции массовой модерации для похожих объявлений
  - Чёткий чек-лист соответствия требованиям к объявлениям (специфичным для вида, для скота, для домашних животных)
  - Возможность запросить дополнительную информацию у продавца до принятия решения
  - Шаблоны комментариев для распространённых причин отклонения
  - Путь эскалации для сложных случаев на рассмотрение администратора
  - Статистика по пропускной способности и точности модерации

**UC-AD-03:** Как администратор, я хочу управлять справочными данными, чтобы я мог обеспечить точность и согласованность на всей платформе.
- Критерии приёмки:
  - Управление справочными данными загружается за <2с
  - CRUD-операции для видов, пород, городов, типов объявлений, состояний здоровья
  - Правила валидации для справочных данных (например, порода должна принадлежать виду)
  - Массовый импорт/экспорт справочных данных (CSV/JSON)
  - Версионирование и история изменений для справочных данных
  - Система уведомлений для пользователей, когда изменения справочных данных затрагивают их объявления

> **Объём реализации (round-9, нормативно — Admin Slice 1).** CRUD доступен только для **управляемых
> справочников, существующих в `database_schema.sql`: `species`, `breeds`, `cities`** (совпадает с
> rbac-matrix.md «Reference data (species, breeds, cities) = ADMIN C/R/U/D»). «Типы объявлений» и
> «статусы животных/здоровья» — это **фиксированные CHECK-перечисления БД** (`listings.listing_type`,
> `listings.status`, жизненный цикл животного), а не CRUD по строкам: их изменение = изменение схемы/ADR.
> Traits, health-certifications и genetic-markers **не имеют управляемой таблицы** и отложены до Фазы 2
> (последний — за `feature_toggles.genetics_portal`). Идентификаторы справочников — **INT** (конвенция
> типов id). Массовый импорт/экспорт и версионирование/история отложены (в MVP нет колонок
> `version`/`deactivated_at`; мягкое удаление = `is_active`); каждое изменение пишется в audit_log.
> ПОЧЕМУ: контракт обязан совпадать с источником истины (схема + RBAC); ПОЧЕМУ ТАК ЛУЧШЕ: исключает
> эндпоинты, которых не может быть, удерживает рамки MVP и оставляет чистый путь добавить справочник,
> когда у него появится таблица. См. `api-contracts/admin-api.yaml` (согласовано).

**UC-AD-04:** Как администратор, я хочу отслеживать активность системы и безопасность, чтобы я мог обнаруживать потенциальные проблемы и реагировать на них.
- Критерии приёмки:
  - Просмотрщик журнала аудита загружается за <2с
  - Фильтрация и поиск журналов аудита по типу действия, пользователю, диапазону дат
  - Оповещения в реальном времени о подозрительной активности (множественные неудачные входы, быстрое создание объявлений)
  - Экспорт журналов аудита для внешнего анализа (CSV/JSON)
  - Представление дашборда с ключевыми метриками модерации и пользователей
  - Возможность приостановить определённые конечные точки API или функции во время обслуживания

**UC-AD-05:** Как пользователь, заботящийся о безопасности платформы, я хочу понимать политики модерации и сообщать о проблемах, чтобы я мог вносить вклад в безопасное сообщество.
- Критерии приёмки:
  - Доступ к руководствам платформы и стандартам сообщества
  - Чёткий процесс сообщения о неприемлемом контенте или пользователях
  - Обновления статуса по отправленным жалобам (на рассмотрении, приняты меры, мер не принято)
  - Защита от мести за добросовестные сообщения
  - Образовательные ресурсы по распознаванию мошенничества и поддельных объявлений
  - Механизм обратной связи для пользователей с предложениями по улучшению модерации

## Предыдущие решения
- Функционал администрирования будет доступен через отдельную панель администратора (изначально та же WebApp, но с доступом на основе ролей).
- Модераторы представляют специальную роль в домене идентификации с разрешениями на модерацию объявлений и пользователей.
- Роль администратора имеет полный доступ к системе включая управление справочными данными.
- Все действия модерации требуют причины и логируются для аудиторного следа.
- Справочные данные (виды, породы и т.д.) управляются через интерфейсы CRUD в панели администратора.
- Мы реализуем управление доступом на основе ролей (RBAC) с детализированными разрешениями.
- Административные действия, изменяющие данные (бан пользователей, отклонение объявлений), требуют явного подтверждения.

## Разбивка задач
1. **Бэкенд (NestJS)**
   - [ ] Создать модуль `admin`
   - [ ] Создать AdminController для конечных точек модерации пользователей
   - [ ] Создать AdminService для бизнес-логики (бан пользователей, изменение ролей и т.д.)
   - [ ] Создать ModerationController для конечных точек модерации объявлений
   - [ ] Создать ModerationService для логики рабочего процесса модерации
   - [ ] Создать ReferenceDataController для управления видами, породами и т.д.
   - [ ] Создать AuditService для ведения журнала действий администратора
   - [ ] Внедрить защитники на основе ролей и системы разрешений
   - [ ] Настроить индексы базы данных для очередей модерации
   - [ ] Написать модульные и интеграционные тесты для потоков работы админа/модерации
   - [ ] Создать документацию OpenAPI для административных конечных точек

2. **Фронтенд (React)**
   - [ ] Создать маршруты панели администратора: /admin/users, /admin/moderation, /admin/reference-data
   - [ ] Реализовать таблицу управления пользователями с поиском, фильтрацией, изменением ролей, действиями бан/разбан
   - [ ] Реализовать очередь модерации с карточками объявлений, действий одобрения/отклонения, вводом причины
   - [ ] Создать интерфейсы управления справочными данными (CRUD для видов, пород и т.д.)
   - [ ] Реализовать просмотр журнала аудита
   - [ ] Создать макет панели администратора с навигацией и видимостью меню на основе ролей
   - [ ] Написать модульные и end-to-end тесты для потоков работы администратора

3. **Инфраструктура**
   - [ ] Убедиться, что схема базы данных поддерживает сущности админа/модерации и журналы аудита
   - [ ] Настроить схему Prisma для сущностей Admin, ModerationAction, AuditLog
   - [ ] Настроить эффективное индексирование для очередей модерации (по статусу, created_at)
   - [ ] Реализовать ограничение частоты запросов для административных конечных точек для предотвращения злоупотреблений
   - [ ] Добавить заголовки безопасности и строгую CSP для панели администратора

## Критерии верификации
- [ ] Модульные тесты >90% покрытия для модуля админа (backend)
- [ ] Интеграционные тесты покрывают: модерацию пользователей (бан, изменение роли), модерацию объявлений (одобрение/отклонение), CRUD справочных данных, журналирование аудита
- [ ] End-to-end тесты покрывают: модератор входит в систему, просматривает очередь, одобряет/отклоняет объявление, просматривает журнал аудита
- [ ] Ручное тестирование: проверка рабочего процесса модерации, полноты аудиторного следа, доступа на основе ролей
- [ ] Производительность: очередь модерации загружается за <2s с 10k ожидающими элементами
- [ ] Соответствие: административные действия полностью аудированы; обработка персональных данных следует 152-ФЗ; модерация контента следует 149-ФЗ
- [ ] Документация: спецификация OpenAPI сгенерирована и доступна
- [ ] Трассируемость НФТ: проверить, что требования производительности, безопасности и доступности корректно учтены и документированы
