# Спецификация: Дорожная карта внедрения

## Результат
Предоставить поэтапный план внедрения для ZooLink, который поставляет ценность на ранних этапах, управляет рисками и позволяет обучаться и адаптироваться. Дорожная карта должна соответствовать объему MVP, определенному в кратком описании проекта, и подготовиться к будущему расширению.

## Область и границы
**Включает:**
- Разбиение на этапы (MVP, Этап 2, Этап 3 и т.д.)
- Приоритизация функций внутри каждого этапа
- Снижение рисков и цели обучения
- Распределение ресурсов и предложения по структуре команды
- Критерии успеха для каждого этапа
- Зависимости между функциями и этапами

**Исключает:**
- Детальная оценка задач (story points) - отложено до спланирования спринта
- Конкретные расписания спринтов - отложено до гибкого планирования
- Назначения отдельных разработчиков - отложено до лидеров команды
- Бюджет и финансовое планирование - отложено до продуктового управления

## Ограничения
- **Фокус на MVP:** Первый этап должен deliver ключевое ценностное предложение: маркетплейс домашнего и сельскохозяйственного скота с аутентификацией и модерацией.
- **Соответствие нормативным актам:** Требования российского законодательства (152-ФЗ, идентификация животных) должны быть выполнены в MVP.
- **Техническая фундаментальная основа:** Архитектура должна поддерживать будущие этапы без значительной переработки.
- **Управление рисками:** Эржно устранить наибольшие риски на ранних этапах (безопасность, производительность гео-поиска, рабочий процесс модерации).
- **Обучение:** Каждый этап должен валидировать ключевые бизнес-гипотезы.
- **Масштабируемость:** Фундамент должен позволять рост до 100k+ пользователей.

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Фундамент должен позволять рост до 100k+ пользователей; каждый этап должен валидировать ключевые бизнес-гипотезы включая производительность (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Управление рисками: устранение наибольших рисков на ранних этапах включая безопасность; соблюдение российских законов о персональных данных (152-ФЗ) в MVP (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Каждый этап должен валидировать ключевые бизнес-гипотезы включая доступность (см. docs/02-requirements/nfr/accessibility.md)

## Предыдущие решения
- Из краткого описания проекта: MVP - только веб-версия с адаптивностью для мобильных устройств; нативные приложения отложены на этап 2.
- Из краткого описания проекта: Чат отложен на этап 2.
- Из краткого описания проекта: Жесткое разделение между маркетплейсами домашнего скота и сельскохозяйственного скота (отдельные UIs/потоки).
- Из решения о стеке технологий (ADR 0001): Бэкенд = NestJS/TypeScript/PostgreSQL, Фронтенд = React/Vite/Tailwind.
- Архитектура должна следовать DDD с ограниченными контекстами, соответствующими доменам, идентифицированным выше.

## Разбивка задач
### Этап 1: MVP (Минимально жизнеспособный продукт)
**Цель:** Валидация основной концепции маркетплейса для торговли домашним и сельскохозяйственным скотом на российском рынке с базовой аутентификацией и модерацией.

**Функции:**
1. **Домен идентификации** (Полный)
   - Регистрация по телефону (верификация через SMS)
   - Вход через OAuth (Google, Apple, Telegram, VK)
   - Базовое управление профилем пользователя
   - Аутентификация через JWT
   - Управление доступом на основе ролей (пользователь, заводчик, фермер, модератор, администратор)
   
2. **Домен животного** (Основной)
   - Сущность животного с базовыми атрибутами (вид, порода, имя, дата рождения, пол и т.д.)
   - Отслеживание чипа идентификации (для домашних животных)
   - Отслеживание бирки для ушей/паспорта (для сельскохозяйственного скота)
   - Связь собственности с Пользователем
   
3. **Домены маркетплейса** (Домашний скот и Сельскохозяйственный скот - Основные)
   - Создание, редактирование, удаление объявлений
   - Гео-поиск (поиск по радиусу, 1-100 км)
   - Базовая фильтрация (вид, порода, цена, тип объявления)
   - Процесс пре-модерации (объявления невидимы до одобрения)
   - Загрузка медиа (изображения через предварительно подписанные URL в S3)
   - Базовое отображение объявлений (карточки, детальные виды)
   
4. **Домен подбора** (Базовый)
   - Профили животных для разведения
   - Базовый поиск/фильтрация для поиска партнеров по разведению
   - Объявления о разведении (тип объявления = "разведение")
   
5. **Домен администрирования** (Основной)
   - Очередь модерации для объявлений
   - Одобрение/отклонение объявлений с указанием причин
   - Базовое управление пользователями (просмотр, изменение роли, блокировка)
   - Управление справочными данными (виды, породы)
   - Журнал аудита действий модерации
   
6. **Техническая основа**
   - Трехслойная архитектура фронтенда (View→Domain→System)
   - Структура бэкенда, выровненная по DDD (модули по доменам)
   - Схема PostgreSQL с ORM Prisma
   - Redis для сессий/кеширования
   - Базовая валидация API и обработка ошибок
   - Docker compose для локальной разработки
   - Конвейер CI/CD (GitHub Actions)
   - Базовый мониторинг/логирование

**Критерии успеха:**
- Ключевой пользовательский поток работает: зарегистрироваться → создать животное → создать объявление → искать объявления → просматривать детали объявления
- Рабочий процесс модерации функционален: модераторы могут одобрять/отклонять объявления
- Гео-поиск возвращает точные результаты в указанном радиусе
- Система обрабатывает 100 одновременных пользователей с приемлемым временем ответа (<2s для ключевых потоков)
- Базовое автоматическое тестовое покрытие: unit тесты >80% для критических путей
- Проверка соответствия: базовое соблюдение 152-ФЗ (минимальное хранение PII, потоки согласия)
- Uptime: ≥99.0% месячный
- RPO: ≤4 часа (максимально tolerable данные потери)
- RTO: ≤2 часа (время восстановления после сбоя)
- Уровень надежности: uptime > 99.0%, RPO < 2 часа, RTO < 1 час для MVP этапа


**Риски, которые необходимо устранить:**
- Безопасность: Безопасность верификации по телефону, ограничение частоты запросов на SMS
- Производительность: Эффективность гео-поиска с ростом данных
- Модерация: Узкие места в ручном рабочем процессе (для улучшения с помощью ML لاحقًا)
- Нормативные акты: Понимание и правильное внедрение требований 152-ФЗ

### Этап 2: Улучшенные функции и полировка
**Цель:** Улучшить пользовательский опыт, добавить функции вовлечения, подготовиться к нативным приложениям.

**Функции:**
1. **Социальные функции** (Чат)
   - Базовый обмен сообщениями между пользователями (обмен контактами после модерации эволюционирует в реальный чат)
   - Push-уведомления (для веба и будущих нативных приложений)
   
2. **Улучшения пользовательского опыта**
   - Расширенные страницы профиля (показ животных, объявлений, значков)
   - Сохраненные поиски и избранное
   - Продвижение/подчеркивание объявлений (простые рекомендуемые объявления)
   - Улучшенный поиск (автодополнение, сохраненные локации)
   - Базовые оценки и отзывы пользователей
   
3. **Мобильный опыт**
   - Улучшения Progressive Web App (офлайн кеширование, приглашения к установке)
   - Подготовка к переходу на React Native
   
4. **Расширенные функции маркетплейса**
   - История редактирования объявлений
   - Повторное размещение/дублирование объявлений
   - Управление запасами для продавцов с несколькими животными
   - Расширенная фильтрация (диапазоны дат, больше атрибутов)
   
5. **Инструменты администрирования и модерации**
   - Массовые действия модерации
   - Метрики производительности модераторов
   - Базовое обнаружение спама (поддержка простого ML)
   - Расширенные журналы аудита
   
6. **Технические улучшения**
   - Оптимизация производительности (ленивая загрузка, разделение кода, оптимизация запросов)
   - Улучшенный мониторинг и оповещение
   - Улучшение индексов базы данных
   - Улучшения CI/CD (средовые окружения для тестирования, канареечные релизы)
   - Feature flags для рискованных изменений

**Критерии успеха:**
- Метрики удержания пользователей показывают улучшение (DAU/WAU)
- Коэффициент конверсии объявлений (просмотры → контакты) улучшается
- Эффективность модерации увеличивается (время на проверку уменьшается)
- Система обрабатывает 500 одновременных пользователей
- Тестовое покрытие улучшается: unit тесты >90%, интеграционные тесты >80%
- Баллы PWA хорошо показывают себя в Lighthouse (>90 производительность, >95 доступность)
- Уровень надежности: uptime > 99.5%, RPO < 1 час, RTO < 30 минут


### Этап 3: Расширение и масштабирование
**Цель:** Масштабирование до большей базы пользователей, добавление сложных функций, подготовка к интернационализации.

**Функции:**
1. **Расширенный подбор и разведение**
   - Базовая оценка генетической совместимости
   - Отслеживание и визуализация родословной
   - Интеграция медицинских записей (с ветеринарными системами через API)
   - Контракты и соглашения на разведение
   
2. **Географическое расширение**
   - Поддержка нескольких языков (фреймворк i18n)
   - Адаптация регионального соответствия (Казахстан, Беларусь и т.д.)
   - Поддержка нескольких валют
   
3. **Новые вертикали**
   - Маркетплейс ветеринарных услуг (записи, телемедицина)
   - Сервисы для животных (груминг, передержка, дрессировка)
   - Интеграция с приютами для защиты и удочерения животных
   
4. **Функции монетизации**
   - Обработка платежей (внутриприложение транзакции за услуги/рекомендуемые объявления)
   - Модельные модели (премиум-уровни для продавцов)
   - Платформа рекламы (таргетированная реклама)
   
5. **Расширенная аналитика**
   - Дашборд для администраторов и продвинутых пользователей
   - Рынковые тренды и инсайты по ценообразованию
   - Аналитика поведения пользователей
   
6. **Масштабирование и производительность**
   - Масштабирование базы данных (реплики только для чтения, секционирование)
   - Расширенные стратегии кеширования
   - Рассмотрение микросервисов для доменов с высокой нагрузкой
   - Глобальная CDN для доставки медиа
   - Нагрузочное тестирование и инженерия хаоса
**Критерии успеха:**
- Система масштабируется до 50k+ активных пользователей
- Готовность к международному расширению (правовая рамка для соответствия)
- Новые вертикали показывают раннее закрепление
- Пути монетизации валидированы
- Производительность поддерживает <2s ключевых потоков в масштабе
- Уровень надежности: uptime > 99.9%, RPO < 30 минут, RTO < 15 минут

- RTO: ≤30 минут

## Зависимости и снижение рисков
- **Технический долг:** Выделять 20% каждого спринта на рефакторинг и уменьшение долга
- **Изменения в регуляторике:** Поддерживать юридическое консультирование для обновлений 152-ФЗ и связанных законов
- **Производительность:** Проводить регулярное нагрузочное тестирование; устанавливать бюджеты производительности
- **Безопасность:** Регулярное тестирование на проникновение; программа поиска уязвимостей
- **Рост команды:** Документировать архитектуру и материалы для ввода в должность на ранних этапах

## Метрики успеха по этапам
- **MVP:** Коэффициент активации пользователей (>30% зарегистрированных пользователей создают объявление), SLA модерации (<48h время проверки)
- **Этап 2:** Метрики вовлеченности (ежедневно активные пользователи, коэффициент возврата к объявлениям), принятие чата, если построен
- **Этап 3:** Метрики расширения (новая демография пользователей, выручка на пользователя, если монетизирован), время безотказной работы системы (>99.9%)

## Приложение: Критерии выпуска. Определение готовности (Definition of Done)
Каждая функция должна соответствовать следующим критериям перед тем, как будет считаться завершенной:
- [ ] Спецификация обновлена или создана (если новая)
- [ ] Код реализован в соответствии со спецификацией
- [ ] Модульные тесты написаны и проходят (>90% покрытия для нового кода)
- [ ] Интеграционные тесты написаны и проходят (где применимо)
- [ ] Ручное тестирование завершено (исследовательское тестирование счастливого пути и граничных случаев)
- [ ] Документация обновлена (API документация, руководства пользователя при необходимости)
- [ ] Документация обновлена на английском и русском языках (если применимо)
- [ ] Код просмотрен минимум одним другим разработчиком
- [ ] В новом коде отсутствуют критические или высокой степени уязвимости безопасности
- [ ] Показатели производительности соответствуют (если применимо)
- [ ] Проверки соответствия завершены (для функций правовой/безопасной природы)
- [ ] Трассируемость НФТ: проверить, что требования производительности, безопасности и доступности корректно учтены и документированы
