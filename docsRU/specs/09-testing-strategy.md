# Спецификация: Стратегия тестирования

## Результат
Определить комплексную стратегию тестирования для ZooLink, которая обеспечивает высокое качество кода, надежность и поддерживаемость. Достичь тестового покрытия ≥ 90–95% с акцентом на модульные, интеграционные и end-to-end тесты, где применимо. Внедрить практики TDD/BDD там, где возможно.

## Область и границы
**Включает:**
- Модульное тестирование для компонентов фронтенда и бэкенда
- Интеграционное тестирование для API endpoints и взаимодействия сервисов
- End-to-end (E2E) тестирование для критических пользовательских потоков
- Измерение и отчетность по тестовому покрытию
- Руководства по разработке через тестирование (TDD) и разработке через поведение (BDD)
- Настройка тестовой среды и интеграция с CI/CD
- Стратегии мокирования внешних зависимостей

**Исключает:**
- Тестирование производительности (нагрузка, стресс) - отложено на этап 2
- Тестирование безопасности на проникновение - отложено на этап 2
- Тестирование доступности (a11y) - отложено, но учитывается в дизайне компонентов
- Тестирование удобства использования - отложено для продуктовой команды
- Визуальное регрессионное тестирование - отложено на этап 2

## План нагрузочного и стресс-тестирования (k6/Locust)

Для оценки производительности системы под нагрузкой на этапе 2 и далее рекомендуется использовать k6 или Locust для симуляции реалистичных сценариев использования.

### Рекомендуемые сценарии
1. **Всплеск регистраций пользователей** – 1000 новых пользователей в течение 5 минут (симуляция пиковой нагрузки во время маркетинговых кампаний)
2. **Одновременное создание объявлений** – 500 активных пользователей одновременно создают объявления
3. **Гео-поиск под нагрузкой** – 1000 запросов/сек при более чем 100k объявлений в базе данных
4. **Смешанная нагрузка** – комбинация просмотра объявлений, поиска, создания объявлений и аутентификации

### Целевые метрики
- Задержка 95-го перцентиля < 2s для критических пользовательских потоков
- Коэффициент ошибок < 0.1%
- Стабильность: система остается отзывчивой во время одночасового непрерывного нагрузочного теста
- Потребление ресурсов: CPU < 70%, RAM < 80% при пиковой нагрузке

### Инструменты
- **k6** – предпочтителен для скриптового тестирования на JavaScript/TypeScript, хорошая интеграция с CI
- **Locust** – удобен для сценарного тестирования на Python, распределенное тестирование


## Ограничения
- **Цель по покрытию:** ≥ 90–95% общего покрытия кода (исключения только для тривиального кода вроде геттеров/сеттеров)
- **Технологии:** Должно соответствовать выбранному стеку (NestJS/Jest для бэкенда, React Testing Library/Jest/Vitest для фронтенда, Cypress/Playwright для E2E)
- **Скорость:** Модульные тесты должны выполняться быстро (<5s для набора unit тестов бэкенда, <10s для набора unit тестов фронтенда)
- **Надежность:** Тесты должны быть детерминированными и не нестабильными (flaky)
- **Поддерживаемость:** Тесты должны быть легкими для написания, чтения и модификации
- **Изоляция:** Модульные тесты должны изолировать тестируемый unit с помощью моков/шпионов
- **CI/CD:** Тесты должны запускаться при каждом pull request и слиянии в main

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Модульные тесты должны выполняться быстро (<5s для набора unit тестов бэкенда, <10s для набора unit тестов фронтенда); весь E2E набор <5m (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Следует рекомендациям по тестированию безопасности; обработка тестовых данных соответствует 152-ФЗ (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Тестирование доступности (a11y) отложено, но учитывается в дизайне компонентов (см. docs/02-requirements/nfr/accessibility.md)

## Пользовательские истории

### Тестирование и обеспечение качества
**UC-TS-01:** Как разработчик, я хочу легко писать и запускать тесты, чтобы обеспечивать качество кода и выявлять ошибки на раннем этапе.
- Критерии приёмки:
  - Настройка и конфигурация тестов хорошо документированы
  - Тестовые утилиты и хелперы доступны для типовых задач
  - Команды для запуска тестов просты и быстры
  - Падения тестов дают понятные и практичные сообщения об ошибках
  - Отчёты о покрытии тестами легко понять и получить к ним доступ

**UC-TS-02:** Как разработчик, я хочу доверять надёжности своих тестов, чтобы уверенно рефакторить и изменять код.
- Критерии приёмки:
  - Тесты детерминированы и не нестабильны (flaky)
  - Тесты корректно изолируют тестируемый unit
  - Зависимости тестов хорошо управляются и мокируются надлежащим образом
  - Тестовые данные реалистичны и покрывают граничные случаи
  - Стратегии очистки тестов предотвращают взаимное влияние тестов

**UC-TS-03:** Как разработчик, я хочу убедиться, что критические пользовательские потоки работают корректно, чтобы предотвращать регрессии в ключевой функциональности.
- Критерии приёмки:
  - End-to-end тесты покрывают критические пользовательские сценарии (регистрация, создание объявления, поиск, модерация)
  - E2E тесты надёжно выполняются в окружении CI/CD
  - E2E тесты предоставляют понятные скриншоты и логи при падении
  - Данные для E2E тестов корректно загружаются и очищаются
  - Набор E2E тестов выполняется в приемлемых временных рамках

**UC-TS-04:** Как разработчик, я хочу поддерживать высокое тестовое покрытие, чтобы обеспечить тестирование большинства путей кода.
- Критерии приёмки:
  - Unit тесты бэкенда достигают >90% покрытия для всех модулей
  - Unit тесты фронтенда достигают >85% покрытия для слоев domain и system
  - Интеграционные тесты покрывают все API endpoints с положительными и отрицательными кейсами
  - Пороги покрытия применяются в пайплайне CI/CD
  - Отчёты о покрытии показывают тренды во времени

**UC-TS-05:** Как архитектор или техлид, я хочу убедиться, что практики тестирования соблюдаются, чтобы поддерживать стандарты качества кода.
- Критерии приёмки:
  - Руководства по тестированию документированы и доступны
  - Ревью тестов является частью процесса ревью кода
  - Pre-commit хуки могут быть настроены для запуска соответствующих тестов
  - Документация по тестированию включает примеры и лучшие практики
  - Тестирование способствует общей поддерживаемости и надёжности

## Предыдущие решения
- **Бэкенд тестирования:**
  - Framework: Jest с supertest для тестирования API
  - Мокирование: Jest моки для внешних сервисов (SMS.RU, Unisender, Redis и т.д.)
  - База данных: Использовать SQLite in-memory или транзакционные откаты для тестов
  - Инструмент покрытия: Встроенный coverage Jest или nyc

- **Фронтенд тестирования:**
  - Framework: Jest с React Testing Library (или Vitest при переходе)
  - Мокирование: Mock service workers (MSW) для мокирования API вызовов
  - События пользователя: Testing Library user-event или Cypress для взаимодействия

- **End-to-End тестирования:**
  - Framework: Cypress (выбран из-за зрелости и простоты использования)
  - Альтернатива: Playwright рассматривался, но Cypress выбран из-за лучшего DX
  - Объем: Критические пользовательские потоки (регистрация, создание объявления, поиск, модерация)

- **Организация тестов:**
  - Бэкенд: `src/**/*.spec.ts` рядом с исходными файлами
  - Фронтенд: `src/**/*.test.tsx` или папки `__tests__`
  - E2E: директория `cypress/` или `e2e/` в корне проекта

- **Подход BDD/TDD:**
  - Поощрять написание тестов до реализации (TDD) там, где возможно
  - Использовать комментарии Given/When/Then в тестах для читаемости в стиле BDD
  - Сосредоточиться на тестировании поведения, а не деталей реализации

## Критерии верификации
- [ ] Unit тесты бэкенда достигают >90% покрытия для всех модулей
- [ ] Unit тесты фронтенда достигают >85% покрытия для слоев domain и system (слой view может иметь меньшее покрытие)
- [ ] Интеграционные тесты покрывают все API endpoints с положительными и отрицательными кейсами
- [ ] E2E тесты покрывают критические пользовательские потоки: регистрация, вход, создание объявления, поиск, модерация
- [ ] Тесты детерминированные: многократный запуск одного набора тестов дает одинаковый результат
- [ ] Тестовый набор выполняется быстро: unit тесты бэкенда <5s, unit тесты фронтенда <10s, весь E2E набор <5m
- [ ] CI/CD пайплайн падает при падении тестов и блокирует слияния
- [ ] Отчеты по покрытию показывают отсутствие значительного падения со временем
- [ ] Документация: руководства по тестированию понятны и соблюдаются contributors
- [ ] Трассируемость НФТ: проверить, что требования производительности, безопасности и доступности корректно учтены и документированы

## Тестовые данные, стратегия БД и CI (раунд 5, нормативно)

- **Тестовая БД — только PostgreSQL.** SQLite **непригоден** — схема опирается на PL/pgSQL-триггеры, JSONB,
  `pg_trgm`, GIN/partial-индексы. Integration/E2E — на **PostgreSQL 16** через Testcontainers (или GH Actions
  `services: postgres`), каждый тест в транзакции с rollback (или свежая ephemeral-БД на сьют).
- **Seed / fixtures (нормативно):**
  - Справочники + `moderation_reasons` + `notification_templates` — из **seed-миграций** (0010); продовые справочники
    (species/breeds/cities РФ) — отдельная seed-миграция.
  - Бизнес-сущности — **типизированные фабрики** (`@faker-js` с фиксированным seed для детерминизма): users, animals,
    listings **во всех состояниях** (DRAFT/PENDING_MODERATION/ACTIVE/EXPIRED/SOLD/DEACTIVATED) и структуры org.
  - **Генератор больших объёмов** (50k животных / 100k объявлений) для гео/perf-тестов.
- **Матрица негативных тестов инвариантов (покрыть правила уровня БД):** триггер ACTIVE⇒APPROVED; цикл родословной/
  само-родитель/пол-вид-DOB родителя; уникальность microchip/tattoo/inn/oauth/phone; один HQ на org; price≥0,
  quantity≥1, currency ISO-4217, язык nickname; append-only audit_log/moderation_decisions; rate-limits (auth,
  contact-reveal). Каждое — явный падающий тест-кейс.
- **Contract testing:** OpenAPI-контракты против реализации (schemathesis или dredd); опц. consumer-driven (Pact) фронт↔бэк.
- **Покрытие:** backend unit/integration ≥90%; критические E2E (register→verify→animal→listing→moderate→publish→
  search→contact-reveal) зелёные; enforce через `coverageThreshold` в CI.
- **CI-гейты (`.github/workflows`):** `ci.yml` (draft-шаблон) = lint + typecheck + проверка `prisma migrate deploy` +
  unit/integration на PG-service + **security-гейты** (`npm audit`, Trivy, Semgrep SAST, OWASP ZAP baseline на staging).
  `performance-tests.yml` — **draft** (k6), активируется после появления репо `./backend`; до этого не является рабочим гейтом.
