# Спецификация: Стратегия тестирования

## Результат
Определить комплексную стратегию тестирования для ZooLink, которая обеспечивает высокое качество кода, надежность и поддерживаемость. Достичь тестового покрытия ≥ 90–95% с акцентом на модульные, интеграционные и end-to-end тесты, где применимо. Внедрить практики TDD/BDD там, где возможно.

## Область и границы
**Включает:**
- Модульное тестирование для компонентов фронтенда и бэкенда
- Интеграционное тестирование для API endpoints и взаимодействия сервисов
- End-to-end (E2E) тестирование для критических пользовательских потоков
- Измерение и отчетность по тестовому покрытию
- Richtlinien для разработки через тестирование (TDD) и разработки через поведение (BDD)
- Настройка тестовой среды и интеграция с CI/CD
- Стратегии мокирования внешних зависимостей

**Исключает:**
- Тестирование производительности (нагрузка, стресс) - отложено на этап 2
- Тестирование безопасности на проникновение - отложено на этап 2
- Тестирование accessible (a11y) - отложено, но учитывается в дизайне компонентов
- Тестирование удобства использования - отложено для продуктовой команды
- Визуальное регрессионное тестирование - отложено на этап 2

## Ограничения
- **Цель по покрытию:** ≥ 90–95% общего покрытия кода (исключения только для тривиального кода вроде геттеров/сеттеров)
- **Технологии:** Должно соответствовать выбранному стеку (NestJS/Jest для бэкенда, React Testing Library/Jest/Vitest для фронтенда, Cypress/Playwright для E2E)
- **Скорость:** Модульные тесты должны выполняться быстро (<5s для набора unit тестов бэкенда, <10s для набора unit тестов фронтенда)
- **Надежность:** Тесты должны быть детерминированными и не нестабильными (flaky)
- **Поддерживаемость:** Тесты должны быть легкими для написания, чтения и модификации
- **Изоляция:** Модульные тесты должны изолировать тестируемый unit с помощью моков/шпионов
- **CI/CD:** Тесты должны запускаться при каждом pull request и слиянии в main

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Модульные тесты должны выполняться быстро (<5s для набора unit тестов бэкенда, <10s для набора unit тестов фронтенда); весь E2E набор <5m (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Следует рекомендациям по тестированию безопасности; обработка тестовых данных соответствует 152-ФЗ (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Тестирование доступности (a11y) отложено, но учитывается в дизайне компонентов (см. docs/02-requirements/nfr/accessibility.md)

## Предыдущие решения
- **Бэкенд тестирования:**
  - Framework: Jest с supertest для тестирования API
  - Мокирование: Jest моки для внешних сервисов (Twilio, Redis и т.д.)
  - База данных: Использовать SQLite in-memory или транзакционные откаты для тестов
  - Инструмент покрытия: Встроенный coverage Jest или nyc

- **Фронтенд тестирования:**
  - Framework: Jest с React Testing Library (или Vitest при переходе)
  - Мокирование: Mock service workers (MSW) для мокирования API вызовов
  - События пользователя: Testing Library user-event или Cypress для взаимодействия

- **End-to-End тестирования:**
  - Framework: Cypress (выбран из-за зрелости и простоты использования)
  - Альтернатива: Playwright рассматривался, но Cypress выбран из-за лучшего DX
  - Объем: Критические пользовательские потоки (регистрация, создание объявления, поиск, модерация)

- **Организация тестов:**
  - Бэкенд: `src/**/*.spec.ts` рядом с исходными файлами
  - Фронтенд: `src/**/*.test.tsx` или папки `__tests__`
  - E2E: директория `cypress/` или `e2e/` в корне проекта

- **Подход BDD/TDD:**
  - Поощрять написание тестов до реализации (TDD) там, где возможно
  - Использовать комментарии Given/When/Then в тестах для readability в стиле BDD
  - Сосредоточиться на тестировании поведения, а не деталей реализации

## Критерии верификации
- [ ] Unit тесты бэкенда достигают >90% покрытия для всех модулей
- [ ] Unit тесты фронтенда достигают >85% покрытия для слоев domain и system (слой view может иметь меньшее покрытие)
- [ ] Интеграционные тесты покрывают все API endpoints с положительными и отрицательными кейсами
- [ ] E2E тесты покрывают критические пользовательские потоки: регистрация, вход, создание объявления, поиск, модерация
- [ ] Тесты детерминированные: многократный запуск одного набора тестов дает одинаковый результат
- [ ] Тестовый набор выполняется быстро: unit тесты бэкенда <5s, unit тесты фронтенда <10s, весь E2E набор <5m
- [ ] CI/CD пайплайн падает при падении тестов и блокирует слияния
- [ ] Отчеты по покрытию показывают отсутствие значительного падения со временем
- [ ] Документация: руководства по тестированию понятны и соблюдаются contributors
- [ ] Трассируемость НФТ: проверить, что требования производительности, безопасности и доступности корректно учтены и документированы

## Нагрузочное и стресс-тестирование

Для обеспечения производительности системы под нагрузкой планируется внедрение нагрузочного и стресс-тестирования начиная с этапа 2.

### Инструменты
- **k6** - для нагрузочного тестирования API endpoints с поддержкой сценариев на JavaScript/TypeScript
- **Locust** - альтернативное решение для распределенного нагрузочного тестирования на Python
- **Artillery** - для комплексного тестирования включая WebSocket и сценарии поведения пользователей

### План тестирования
1. **Базовое нагрузочное тестирование (этап 2)**
   - Тестирование аутентификации: 100 RPS (requests per second) с пиком до 200 RPS
   - Тестирование создания объявлений: 50 RPS с пиком до 100 RPS
   - Тестирование гео-поиска: 30 RPS с пиком до 60 RPS
   - Целевые показатели latency: 95% запросов <1s, 99% запросов <2s

2. **Стресс-тестирование (этап 3)**
   - Постепенное увеличение нагрузки до точки отказа
   - Тестирование Recovery после пиковых нагрузок
   - Определение максимальной пропускной способности системы
   - Тестирование с аномальными паттернами трафика (spiky, surge)

3. **Регулярное тестирование производительности**
   - Интеграция нагрузочных тестов в CI/CD pipeline для регрессионного тестирования
   - Еженедельные запуски базовых нагрузочных сценариев
   - Мониторинг ключевых метрик производительности в продакшене

### Критерии успеха для нагрузочного тестирования
- Система должна поддерживать целевую нагрузку без ошибок 5xx
- Среднее время ответа не должно превышать целевые показатели более чем на 20%
- 95-й перцентиль времени ответа должен находиться в пределах SLA
- Система должна восстанавливаться после пиковых нагрузок без ручного вмешательства
- Использование ресурсов (CPU, Memory, DB connections) должно оставаться в безопасных пределах

### Отчетность и метрики
- Отслеживание следующих метрик: RPS, среднее время ответа, перцентили времени ответа (50th, 90th, 95th, 99th), коэффициент ошибок
- Сравнение результатов с baseline и предыдущими запусками
- Генерация отчетов в формате HTML/JUnit для интеграции с CI/CD
- Уведомления о деградации производительности через интеграцию с системами мониторинга
