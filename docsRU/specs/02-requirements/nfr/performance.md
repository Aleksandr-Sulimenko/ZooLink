# Требования к производительности нефункционального характера: ZooLink

## Цель
Определяет требования к производительности, чтобы обеспечить отзывчивый пользовательский опыт, масштабируемость и эффективное использование ресурсов при ожидаемых нагрузках.

## Область применения
Применяется ко всем компонентам системы: backend API, frontend приложения, запросы к базе данных, файловые операции и интеграции.

## Цели и метрики производительности
### Целевые значения времени отклика
- **API Endpoints**:
   - 95% запросов < 1s при нормальной нагрузке
   - 99% запросов < 2s при нормальной нагрузке
   - Максимум 95-го процентиля < 3s в пиковую нагрузку
- **Специфичные критические эндпоинты**:
   - Geo-поиск объявлений: < 1.5s для 95% запросов (<100km радиус)
   - Создание объявления (включая загрузку фото): < 2s
   - Аутентификация (вход/регистрация): < 2s
   - Получение профиля животного: < 500ms
   - Просмотр деталей объявления: < 1s
   - Загрузка очереди модерации: < 2s для 100 элементов
   - Автодополнение поиска: < 300ms
- **Frontend**:
   - First Contentful Paint (FCP): < 1.5s при подключении 3G
   - Largest Contentful Paint (LCP): < 2.5s при подключении 3G
   - Time to Interactive (TTI): < 3.5s при подключении 3G
   - Cumulative Layout Shift (CLS): < 0.1
   - First Input Delay (FID): < 100ms

### Пропускная способность и ёмкость
- **Конкурентные пользователи**: 
   - Поддержка 500 одновременных активных пользователей без ухудшения
   - Выдержка 1000 одновременных сессий (включая неактивные вкладки)
- **Запросов в секунду (RPS)**:
   - Устойчивая: 50 RPS в среднем
   - Пиковая: 200 RPS для 5-минутных всплесков
   - Всплеск: 500 RPS для 30-секундных скачков
- **Объём данных**:
   - Поддержка 100K всего объявлений (активных + archived) без ухудшения
   - Поддержка 50K активных профилей животных
   - Поддержка 20K зарегистрированных пользователей
   - Поддержка 1TB общего хранилища файлов (фото) с эффективным извлечением

## Использование ресурсов
- **CPU**: 
   - Среднее использование < 60% при нормальной нагрузке
   - Пиковое использование < 85% во время скачков трафика
- **Память**:
   - Среднее использование < 70% при нормальной нагрузке
   - Пиковое использование < 85% во время скачков трафика
   - Отсутствие утечек памяти (контролируется в периодах 24+ часов)
- **База данных**:
   - Использование пула соединений < 80% при нормальной нагрузке
   - Среднее время выполнения запроса < 100ms
   - Медленные запросы (>1s) < 0.1% от общего числа запросов
- **Хранилище**:
   - Занятость дискового пространства < 70% для размера роста и обслуживания
   - IOP suficientes для обработки пиковой нагрузки без очередей

## Мониторинг производительности и критерии приемки
### Ключевые показатели эффективности (KPIs)
- **Оценка Apdex**: Целевое значение ≥ 0.9 (удовлетворённые пользователи) для критических пользовательских путей
- **Частота ошибок**: < 0.5% запросов, приводящих к ошибкам 5xx
- **Доступность**: 
   - Целевое значение: 99.5% времени работы в месяц (исключая плановое обслуживание)
   - Максимальное время простоя: < 4 часов в месяц
- **Специфично для geo-поиска**:
   - 90% geo-запросов возвращают результаты за < 1.5s
   - 99% geo-запросов возвращают результаты за < 3s
   - Использование индекса: > 95% geo-запросов используют соответствующие индексы

### Измерение и профилирование
- **Инструменты мониторинга**:
   - Application Performance Monitoring (APM): Datadog, New Relic или аналогичные
   - Real User Monitoring (RUM): Для метрик производительности frontend
   - Инфраструктурный мониторинг: использование CPU, памяти, диска, сети
   - Мониторинг базы данных: производительность запросов, пул соединений, задержка репликации
   - Агрегация логов: стек ELK или аналогичный для корреляции и отладки
- **Подход к тестированию**:
   - Нагрузочное тестирование: Locust, k6 или JMeter для имитации пользовательской нагрузки
   - Стресс-тестирование: Для нахождения точек отказа и поведения восстановления
   - Тестирование выдержки (Soak): Для выявления утечек памяти и постепенного ухудшения
   - Тестирование скачков (Spike): Для оценки обработки внезапных всплесков трафика
   - Конфигурация: Тесты выполняются против среды staging, зеркальной production
- **Критерии приемки для релизов**:
   - Отсутствие регресса в критических метриках производительности (>10% ухудшения)
   - Все цели производительности достигнуты при тестовой нагрузке
   - Тесты производительности включены в пайплайн CI/CD для предотвращения регрессий

## Стратегии оптимизации производительности
### Оптимизации backend
- **База данных**:
   - Правильное индексирование по шаблонам запросов (geo-поиск, объявления по типу/виду/местоположению)
   - Использование пула соединений (HikariCP или аналогичный через Prisma)
   - Оптимизация запросов: избегать SELECT *, использовать лимиты, оптимизировать JOINs
   - Реплики чтения для операций, тяжёлых на чтение (geo-поиск, просмотр объявлений)
   - Кэширующий слой (Redis) для:
      - Справочных данных (породы, виды, города) - TTL 24ч
      - Дanych сессий - TTL, соответствующий токену обновления
      - Часто запрашиваемых профилей животных (кеш LRU)
      - Рассчитанных оценок соответствия (для популярных животных)
   - Таймаут соединения с БД: 30s
   - Таймаут оператора: 10s для рискованных запросов
- **API и сервисы**:
   - Пагинация для эндпоинтов списков (по умолчанию 20 элементов, максимум 100)
   - Эффективная сериализация: избегать перевыборки, использовать DTOs
   - Асинхронная обработка для некритических операций (уведомления по email, аналитика)
   - Сжатие запросов/ответов (gzip/brotli)
   - Поддержание соединений и HTTP/2 там, где это выгодно
   - Ограничение скорости для предотвращения degradation из-за злоупотребления
   - Паттерн circuit breaker для внешних интеграций
- **Файловые операции**:
   - Pre-signed URL для прямой загрузки/скачивания в объектное хранилище (без прокси через backend)
   - Многокомпонентная загрузка для больших файлов
   - Кеширование CDN для статических активов и часто запрашиваемых изображений
   - Оптимизация изображений: автоматическое изменение размера, сжатие, выбор формата (WebP)
   - Очистка временных файлов: автоматизированная задача для удаления незавершённых загрузок через 24ч
- **Стратегия кэширования**:
   - Паттерн cache-aside для кэширования на уровне приложения
   - Инвалидация кэша: TTL-based + явная инвалидация при обновлениях
   - Прогрев кэша для предсказуемых периодов высокой нагрузки
   - Метрики кэша: соотношение popадания/промаха, уровень вытеснения, использование памяти

### Оптимизации frontend
- **Оптимизация бандла**:
   - Разделение кода: route-based и component-based отложенная загрузка
   - Удаление неиспользуемых зависимостей (tree shaking)
   - Минимизация: JavaScript, CSS, HTML
   - Критический CSS: встраивание стилей above-the-fold
   - Оптимизация шрифтов: подмножества, предзагрузка, font-display: swap
- **Оптимизация рендеринга**:
   - Виртуальная прокрутка для длинных списков (объявления, животные, очередь модерации)
   - Библиотеки windowing для больших наборов данных
   - Использование request animation frame для анимаций
   - Debounce/throttle для обработчиков resize/scroll
   - CSS containment для сложных компонентов
- **Оптимизация сети**:
   - Мультиплексирование HTTP/2 (если поддерживается хостингом)
   - Приоритезация ресурсов: предзагрузка критических активов
   - Предварительная выборка DNS для сторонних доменов
   - Предварительная выборка/предварительный рендер для предсказуемой навигации
   - Service worker для кеширования PWA в офлайне и фоновой синхронизации
- **Оптимизация изображений**:
   - Отзывчивые изображения: атрибуты srcset и sizes
   - Отложенная загрузка: изображения ниже сгиба
   - Заполнители: низкокачественные заполнители изображений (LQIP) или skeleton экраны
   - Подача формата: WebP с fallback'ами
   - Указание размеров для предотвращения сдвига макета
- **Управление состоянием**:
   - Минимизация ререндеров: useMemo, useCallback, React.memo
   - Нормализация формы состояния для предотвращения ненужных обновлений
   - Выборочные подписки в хранилищах состояния
   - Неизменяемые обновления там, где это выгодно

### Архитектурные соображения
- **Паттерны масштабируемости**:
   - Горизонтальное масштабирование: stateless backend сервисы за балансировщиком нагрузки
   - Шардирование базы данных: рассматривается для Фазы 2+ если один екземпляр недостаточен
   - Микросервисы: не для MVP; модульный монолит с чёткими границами
   - Событийно-ориентированный: использование очереди сообщений (RabbitMQ/Amazon SQS) для non-real-time процессов
- **Специфично для geo-поиска**:
   - Рассматривается расширение PostGIS для Фазы 2+ для продвинутых пространственных запросов
   - Текущий MVP: использование PostgreSQL с GiST индексом на geography point
   - Аппроксимация Geohash для фильтрации перед точным расчётом
   - Кеширование на основе тайлов для популярных регионов (хранение предвычисленных результатов)
- **Доставка контента**:
   - CDN для статических активов (JS, CSS, изображения, шрифты)
   - Региональное размещение рассматривается для Фазы 2+ если пользовательская база сосредоточена географически
   - Вычисления на краю для аутентификации и операций чтения, тяжёлых на чтение (будущее)

## Сценарные тесты производительности
### Базовое нагрузочное тестирование
- 100 одновременных пользователей
- Смесь активностей: просмотр (60%), поиск (20%), просмотр объявлений (15%), создание контента (5%)
- Продолжительность: 30 минут
- Критерий успеха: 95% запросов < 2s, частота ошибок < 0.5%

### Тестирование пиковой нагрузки
- 500 одновременных пользователей
- Та же смесь, как в базовом
- Продолжительность: 15 минут
- Критерий успеха: 95% запросов < 3s, частота ошибок < 1%, отсутствие сбоев системы

### Стресс-тест
- Плавное увеличение с 100 до 1000 одновременных пользователей за 10 минут
- Удержание на пике в течение 5 минут
- Плавное снижение за 5 минут
- Критерий успеха: Система degradation gracefully, восстановление после уменьшения нагрузки, отсутствие потери данных

### Тест выдержки (Soak)
- 200 одновременных пользователей при постоянной нагрузке
- Продолжительность: 4 часа
- Критерий успеха: Отсутствие утечек памяти (<5% рост памяти), стабильная производительность, отсутствие истощения ресурсов

### Тест скачков (Spike)
- Базовый уровень 100 одновременных пользователей
- Всплеск до 1000 одновременных пользователей на 5 минут
- Возврат к базовому уровню
- Критерий успеха: Выдерживает скачок без сбоев, восстановление до базовой производительности в течение 2 минут

### Специфичный тест geo-поиска
- 100 одновременных пользователей выполняющих geo-запросы
- Смесь радиусов: 1km (30%), 10km (40%), 50km (20%), 100km (10%)
- Смесь плотности результатов: городские (высокая плотность) и сельские (низкая плотность)
- Критерий успеха: 95% запросов < 1.5s, 99% < 3s, правильное использование индекса

## Дорожная карта оптимизации производительности
### MVP (Фаза 1)
- Базовое индексирование по полям запросов
- Кэширование Redis для справочных данных и сессий
- Оптимизация изображений и CDN для статических активов
- Frontend code splitting и отложенная загрузка
- Пагинация на эндпоинтах списков
- Пул соединений с базой данных

### Фаза 2 (Рост)
- Реплики чтения для просмотра объявлений и поиска
- Оптимизация запросов на основе логов продакшена
- Продвинутые стратегии кэширования (рассчитанные оценки соответствия, популярные профили животных)
- Соблюдение бюджета производительности frontend
- CDN для изображений, загруженных пользователями (через сервис изменения размера/кеширования)
- Оценка PostGIS для улучшенного geo-поиска

### Фаза 3 (Зрелость)
- Шардирование базы данных при необходимости (по географии или типу объявления)
- Микросервисы для компонентов высокой масштабируемости (сопоставление, уведомления)
- Продвинутые стратегии CDN (оптимизация изображений на краю, потоковая передача видео)
- Машинное обучение для предсказания кэширования и предвычисления
- Реальное время аналитики производительности и автоматическая оптимизация
