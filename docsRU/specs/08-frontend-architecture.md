# Спецификация: Архитектура фронтенда

## Результат
Определить архитектуру фронтенда siguiendo трехслойной модели (View → Domain → System), как указано в требованиях. Обеспечить сильную связанность со слоем Domain и слабую связанность с внешними зависимостями. Подготовить фронтенд к масштабируемости и поддерживаемости с использованием современных практик, таких как feature-sliced design или доменно-ориентированная структура.

## Область и границы
**Включает:**
- Структура фронтенда, организованная в три слоя: View (Презентация), Domain (Бизнес-логика), System (Инфраструктура/Приложение)
- Использование React с TypeScript
- Решение для управления состоянием (React Context API или Zustand)
- Библиотека UI (Headless UI с Tailwind CSS)
- Организация папок feature-sliced или domain-driven
- Подготовка к росту: модульность, переиспользуемость и четкое разделение ответственности
- Интеграция с backend API через слой System

**Исключает:**
- Конфигурация инструментов сборки (Vite) - покрывается решением о стеке технологий
- Детали стилизации (конфигурация Tailwind) - покрывается решением о стеке технологий
- Конкретные библиотеки UI компонентов помимо headless UI - отложено
- Настройка тестирования (Jest, React Testing Library, Cypress) - отложено к спецификации тестирования
- Оптимизация производительности (разделение кода, ленивая загрузка) - отложено к спецификации производительности

## Ограничения
- **Архитектура:** Должно следовать трехслойной модели: View, Domain, System.
- **Связанность:** Слой Domain не должен зависеть от слоев View или System. Слой System обрабатывает внешние зависимости (API, хранилище и т.д.).
- **Технологии:** Должно соответствовать выбранному стеку (React, TypeScript, Vite, Tailwind, Headless UI).
- **Удобство использования:** Интерфейс должен быть доступным и удобным для нетехнических массовых пользователей (владельцев домашних животных).
- **Производительность:** Время начальной загрузки < 3s на 3G, время до интерактивности < 5s.
- **Масштабируемость:** Структура должна поддерживать добавление новых функций без значительной рефакторинга.
- **Поддерживаемость:** Код должен быть легко понятен и модифицируем разработчиками, знакомыми со стеком.

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Время начальной загрузки < 3s на 3G, время до интерактивности < 5s (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Следует лучшим практикам безопасности для frontend-приложений (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Интерфейс должен быть доступным и удобным для нетехнических массовых пользователей (владельцев домашних животных); следует рекомендациям WCAG 2.1 AA (см. docs/02-requirements/nfr/accessibility.md)

## Пользовательские истории

### Пользовательский опыт фронтенда
**UC-FE-01:** Как пользователь, я хочу, чтобы приложение загружалось быстро и было отзывчивым, чтобы пользоваться платформой без раздражения.
- Критерии приёмки:
  - Время начальной загрузки < 3s на 3G
  - Время до интерактивности < 5s
  - Плавные переходы и анимации
  - Мгновенная обратная связь на действия пользователя (нажатия кнопок, отправка форм)
  - Индикаторы загрузки для асинхронных операций
  - Границы ошибок (error boundaries) для предотвращения сбоев и показа понятных сообщений

**UC-FE-02:** Как пользователь, я хочу, чтобы интерфейс был интуитивным и удобным для навигации, чтобы находить нужное без путаницы.
- Критерии приёмки:
  - Единообразный паттерн навигации по всему приложению
  - Чёткая визуальная иерархия и группировка связанных элементов
  - Описательные подписи и подсказки (placeholder) в формах
  - Хлебные крошки для глубокой навигации
  - Функция поиска с авто-подсказками, где это уместно
  - Всплывающие подсказки для сложных функций

**UC-FE-03:** Как пользователь, я хочу, чтобы приложение было доступным, чтобы пользоваться им независимо от моих возможностей или устройства.
- Критерии приёмки:
  - Соответствие WCAG 2.1 AA
  - Интерфейс с навигацией с клавиатуры
  - Подписи и ориентиры (landmarks), дружественные к программам чтения с экрана
  - Достаточный контраст цвета для текста и иконок
  - Адаптивный дизайн для мобильных устройств и планшетов
  - Регулируемый размер текста и масштабируемый UI

**UC-FE-04:** Как пользователь, я хочу чувствовать себя в безопасности и контролировать свою личную информацию, чтобы доверять платформе свои данные.
- Критерии приёмки:
  - Понятные политика конфиденциальности и условия использования
  - Минимальный сбор данных и прозрачное использование
  - Понятные запросы разрешений (местоположение, уведомления)
  - Возможность просмотреть и удалить персональные данные
  - Безопасная обработка токенов аутентификации и паролей
  - Видимые индикаторы безопасности (HTTPS, значки защищённости)

**UC-FE-05:** Как пользователь, я хочу, чтобы приложением было приятно и удобно пользоваться, чтобы возвращаться к нему регулярно.
- Критерии приёмки:
  - Продуманные микро-взаимодействия и приятные детали
  - Единообразный и привлекательный визуальный дизайн
  - Вовлекающие пустые состояния и сообщения об ошибках
  - Персонализация, где уместно (сохранённые поиски, избранное)
  - Возможность работы офлайн для критичных функций (будущее улучшение)
  - Регулярные обновления и улучшения на основе обратной связи пользователей

## Предыдущие решения
- Технологический стек (ADR 0001): React с TypeScript, Vite, Tailwind CSS, Headless UI.
- Управление состоянием: React Context API для простоты в MVP (может мигрировать на Zustand при необходимости).
- Структура папок: Мы будем использовать доменно-ориентированный подход, где функции группируются по домену (например, `identity`, `animal`, `pet-marketplace`) и внутри каждой функции мы организуем по слоям (view, domain, system).
- Рассматривалась альтернатива: Feature-sliced design (FSD), но выбрано доменно-ориентированное для более сильного согласования с backend DDD.
- Коммуникация с API: Абстрагирована через сервисные классы в слое System.
- Управление состоянием: Логика домена (варианты использования) будут чистыми функциями или классами, которые можно тестировать независимо.

## Разбивка задач
1. **Структура папок**
   - [ ] Определить и создать следующую структуру в `frontend/src/`:
     ```
     src/
       app/                    # Поставщики приложений, маршруты, глобальные стили
       features/               # Фичи, специфичные для домена (identity, animal, и т.д.)
         identity/             # Фича идентификации
           view/               # Компоненты UI (страницы, формы, кнопки)
           domain/             # Бизнес-логика ( варианты использования, сущности, валидация)
           system/             # Инфраструктура (сервисы API, хранилище, состояние)
         animal/               # Фича животного (та же подструктура)
         pet-marketplace/
         livestock-marketplace/
         matching/
         admin/
         shared/               # Общие компоненты, хуки, утилиты между фичами
       lib/                    # Общие библиотеки (клиент API, утилиты, константы)
       styles/                 # Конфигурация Tailwind CSS, глобальные стили
       types/                  # Общие типы TypeScript
     ```

2. **Слой View (Презентация)**
   - [ ] Создавать презентационные компоненты, которые являются "тупыми" и получают данные через props
   - [ ] Компоненты страниц, которые маршрутизируют данные из слоев domain/system
   - [ ] Компоненты должны быть переиспользуемыми и настраиваемыми через props
   - [ ] Стилизация с использованием Tailwind CSS и primitives Headless UI
   - [ ] Компоненты в `src/features/*/view/`

3. **Слой Domain (Бизнес-логика)**
   - [ ] Содержать бизнес-сущности (классы/интерфейсы TypeScript)
   - [ ] Содержать варианты использования (специфичные для приложения бизнес-правила)
   - [ ] Содержать логику валидации
   - [ ] Чистые функции везде, где возможно (легко тестируются)
   - [ ] Не должны импортировать из слоев view или system (кроме типов)
   - [ ] Логика домена в `src/features/*/domain/`

4. **Слой System (Инфраструктура/Приложение)**
   - [ ] Обрабатывать внешние зависимости: сервисы API, браузерное хранилище, кеширование
   - [ ] Управление состоянием (провайдеры React Context или хранилища Zustand)
   - [ ] Хуки, мосты между domain и view
   - [ ] Сервисные классы, которые обрабатывают коммуникацию с backend
   - [ ] Слой system в `src/features/*/system/`

5. **Общий слой**
   - [ ] Создать общие компоненты, хуки и утилиты в `src/features/shared/`
   - [ ] Создать общий клиент API, константы и типы в `src/lib/`
   - [ ] Создать общие стили и тему в `src/styles/`

6. **Управление состоянием**
   - [ ] Решить между React Context API и Zustand для MVP (начнем с Context)
   - [ ] Создать глобальные хранилища для: состояния аутентификации, состояния пользователя, возможно, ожидающих действий
   - [ ] Состояние, специфичное для фичи, может быть colocated в системном слое фичи

7. **Коммуникация с API**
   - [ ] Создать типизированный клиент API (используя fetch или axios) в `src/lib/api-client.ts`
   - [ ] Создать сервисные классы для каждого backend модуля в системном слое каждой фичи
   - [ ] Использовать интерсепторы для добавления токенов аутентификации, обработки ошибок

8. **Маршрутизация**
   - [ ] Определить маршруты в `src/app/routes.tsx` (или аналог)
   - [ ] Защищать маршруты на основе аутентификации и роли (используя гвардейцы слоя system)

## Критерии верификации
- [ ] Структура папок следует доменно-ориентированной, трехслойной модели
- [ ] Компоненты слоя View являются презентационными и не содержат бизнес-логики
- [ ] Слой Domain содержит чистую бизнес-логику и может быть протестирован без DOM или API
- [ ] Слой System обрабатывает все внешние зависимости (API, хранилище и т.д.)
- [ ] Нет циклических зависимостей между слоями (использовать madge или аналогичный для проверки)
- [ ] Код типизирован с TypeScript (нет `any`, если абсолютно необходимо)
- [ ] Компоненты переиспользуемы и компонуемы
- [ ] Управление состоянием централизовано и предсказуемо
- [ ] Сервисы API согласованы и обрабатывают ошибки единообразно
- [ ] Юнит-тесты для слоя domain (бизнес-логика) достигают >90% покрытия
- [ ] Тесты компонентов (слой view) достигают разумного покрытия (>70%)
- [ ] E2E-тесты покрывают критические пользовательские потоки (вход, создание объявления, поиск)
- [ ] Производительность: размер бандла проанализирован, реализована ленивая загрузка для больших фич
- [ ] Документация: Storybook или аналогичный для библиотеки компонентов (отложено на этап 2)
- [ ] Трассируемость НФТ: проверить, что требования производительности, безопасности и доступности корректно учтены и документированы
