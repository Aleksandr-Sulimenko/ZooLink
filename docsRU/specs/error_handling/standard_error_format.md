# Стандартный формат ответа об ошибке

## Обзор
Этот документ определяет стандартный формат JSON-ответа об ошибке для всех конечных точек API в системе ZooLink, а также диапазоны кодов ошибок, конкретные коды ошибок для распространенных отказов валидации и использование корреляционного идентификатора для трассировки запросов.

## Стандартный формат JSON-ответа об ошибке
Все ответы об ошибках от API должны следовать этой структуре:

```json
{
  "error": {
    "code": "<КОД_ОШИБКИ>",
    "message": "<Читаемое человеком сообщение об ошибке>",
    "details": [                 // Необязательно: Массив конкретных деталей ошибки
      {
        "field": "<имя_поля>",   // Необязательно: Для ошибок валидации, поле, которое не прошло проверку
        "code": "<ПОДКОД_ВАЛИДАЦИИ>",
        "message": "<Сообщение об ошибке поля>"
      }
    ],
    "correlationId": "<уникальный_идентификатор_запроса>", // Для трассировки запросов между сервисами
    "timestamp": "<ISO_8601_метка_времени>",               // Время возникновения ошибки
    "path": "<путь_конечной_точки_API>"                    // Конечная точка, которая вызвала ошибку
  }
}
```

### Описание полей
- **error.code**: Код ошибки верхнего уровня из определенных диапазонов (см. ниже)
- **error.message**: Общее описание ошибки (подходит для отображения пользователям)
- **error.details**: Необязательный массив, предоставляющий гранулярные детали, особенно для ошибок валидации
- **error.correlationId**: Уникальный идентификатор запроса, используемый для трассировки в журналах
- **error.timestamp**: Метка времени возникновения ошибки (UTC, формат ISO 8601)
- **error.path**: Путь конечной точки API, которая вызвала ошибку

## Диапазоны кодов ошибок
| Диапазон | Описание | Примеры использования |
|----------|----------|-----------------------|
| 4000-4999 | Ошибки клиента | Ошибки валидации, проблемы аутентификации, неверные запросы |
| 5000-5999 | Ошибки сервера | Внутренние ошибки сервера, недоступность servicioа, сбои базы данных |

## Распространенные коды ошибок

### Ошибки клиента (4000-4999)
| Код | Сообщение | Детали | Сценарий |
|-----|-----------|--------|----------|
| 4000 | Неверный запрос | Ошибки валидации в полезной нагрузке запроса | Неправильный JSON, отсутствие обязательных полей |
| 4001 | Не авторизован | Требуется аутентификация | Отсутствующий или неверный токен аутентификации |
| 4002 | Запрещено | Недостаточно прав | Пользователь не имеет роли/разрешения для выполнения действия |
| 4003 | Не найдено | Ресурс не найден | Запрошенное животное/листинг/пользователь не существует |
| 4004 | Метод не разрежен | HTTP-метод не поддерживается | Использование POST на конечной точке только для чтения |
| 4005 | Конфликт | Конфликт состояния ресурса | Попытка продать животное, уже помеченное как проданное |
| 4006 | Не прошла валидация | Ошибки валидации ввода | См. подкоды валидации ниже |
| 4007 | Превышен лимит скорости | Слишком много запросов | Пользователь превысил допустимую частоту запросов |
| 4008 | Листинг истек | Листинг больше не активен | Попытка приобрести истекший листинг |
| 4009 | Недостаточно средств | Требуется оплата | Пользователь пытается совершить покупку без достаточного баланса или способа оплаты |
| 4010 | Недействительный радиус геопоиска | Радиус вне допустимых пределов | Запрошенный радиус поиска <1 км или >100 км |

### Ошибки сервера (5000-5999)
| Код | Сообщение | Детали | Сценарий |
|-----|-----------|--------|----------|
| 5000 | Внутренняя ошибка сервера | Неожиданное условие | Универсальная заглушка для неперехваченных исключений |
| 5001 | Сервис недоступен | Временная перегрузка или técnicas обслуживания | Зависание зависимости от стороннего сервиса |
| 5002 | Ошибка базы данных | Операция с базой данных не удалась | Таймаут подключения, нарушение ограничения |

| 5004 | Ошибка очереди сообщений | Публикация события не удалась | Проблемы с подключением к RabbitMQ/Kafka |
| 5005 | Ошибка внешнего сервиса | Сбой стороннего API | Шлюз SMS, сервис геокодирования недоступен |
| 5006 | Ошибка конфигурации | Отсутствующая или неверная конфигурация | Требуемая переменная окружения не установлена |
| 5007 | Не реализовано | Функция еще не доступна | Конечная точка существует, но логика не реализована |
| 5008 | Таймаут | Обработка запроса заняло слишком много времени | Таймаут верхнего уровня или долгая операция |
| 5009 | Конфликт одновременности | Обнаружено одновременное изменение | Неудача оптимистической блокировки при обновлении |

## Использование корреляционного идентификатора
### Генерация
- каждому входящему запросу должен быть назначен уникальный корреляционный идентификатор на точке входа (шлюз API или первый микросервис).
- Формат: UUIDv4 (например, "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8")
- Если входящий запрос уже содержит корреляционный идентификатор (через заголовок `X-Correlation-ID`), он должен быть передан; в противном случае, генерируется новый.

### Распространение
- корреляционный идентификатор должен быть включен в:
  - Все исходящие HTTP-запросы к другим сервисам (в виде заголовка `X-Correlation-ID`)
  - Все записи журнала, связанные с запросом
  - Ответы об ошибках (в поле `error.correlationId`)
  - Метрики и трассировки (если используются инструменты APM)

### Хранение и ведение журнала
- Записи журнала должны включать корреляционный идентификатор в структурированном формате (например, журналы в формате JSON) для обеспечения фильтрации и трассировки.
- Пример записи журнала:
  ```json
  {
    "timestamp": "2026-06-13T10:30:00.000Z",
    "level": "ERROR",
    "message": "Не удалось проверить цену листинга",
    "correlationId": "a1b2c3d4-e5f6-7890-g1h2-i3j4k5l6m7n8",
    "service": "listing-service",
    "endpoint": "/listings",
    "errorCode": "4006-001"
  }
  ```

### Преимущества
- Обеспечивает сквозную трассировку запроса между несколькими сервисами.
- Упрощает отладку, позволяя операторам находить все журналы, связанные с одним действием пользователя.
- Облегчает мониторинг и оповещение о частоте ошибок на основе корреляционного идентификатора (если необходимо).

## Рекомендации по реализации
1. **Подход через промежуточное ПО**: Реализуйте форматирование ошибок и внедрение корреляционного идентификатора как промежуточное ПО на входе в API шлюзе или точке входа в сервис.
2. **Единообразное использование**: Все сервисы должны придерживаться этого формата; нельзя использовать ad-hoc ответы об ошибках.
3. **Локализация**: Поле `message` должно быть локализовано на основе заголовка `Accept-Language` (см. спецификации локализации).
4. **Безопасность**: Избегайте раскрытия конфиденциальной информации в сообщениях об ошибках (например, данные подключения к базе данных).
5. **Тестирование**: Юнит- и интеграционные тесты должны проверять, что ответы об ошибках соответствуют этому формату.
6. **Документация**: Включите примеры ответов об ошибках в документацию OpenAPI/Swagger для каждой конечной точки.
