# Спецификация: Сервис гео-поиска

## Результат
Обеспечить эффективные возможности гео-поиска для поиска животных и объявлений в указанном радиусе (1-100 км) от местоположения пользователя. Поддерживать точные расчеты расстояния и фильтрацию, чтобы обеспечить поиск на основе местоположения во всех доменах рынка (Домашние животные, Скот, Подбор).

> ⚠️ **Решение MVP (зафиксировано):** в MVP используются колонки `lat`/`lng` + **формула Haversine с предфильтром bounding box** (без расширений), по [ADR-0009](../04-decisions/0009-mvp-vs-target-architecture.md) и `storage.md`. **PostGIS** (и альтернатива `earthdistance`/`ll_to_earth`) — **Фаза 2+**, а не открытый вопрос MVP. Упоминания PostGIS ниже — это Target-вариант.

## Область и границы
**Включает:**
- Расчет расстояния с использованием формулы Хаверсин или расширения PostGIS
- Поиск по радиусу: найти все точки в пределах X километров от заданной широты/долготы
- Интеграция с сущностями Объявлений (Рынок домашних животных, Рынок скота)
- Интеграция с сущностями Животных (для домена Подбора)
- Поддержка как точного соответствия местоположения, так и поиска по радиусу
- Возможность комбинировать гео-поиск с другими фильтрами (вид, порода, цена и т.д.)
- Оптимизация производительности для больших наборов данных (100k+ записей)
- Конфигурация ограничений радиуса поиска (мин 1км, макс 100км согласно требованиям)

**Исключает:**
- Маршруты/направления - отложено
- Автозаполнение мест/геокодинговое обслуживание (обрабатывается внешним API Карт)
- Сложные поиски по полигонам (например, поиск внутри границ города) - отложено
- Отслеживание местоположения в реальном времени - отложено
- Офлайн карты/кэширование - отложено

## Ограничения
- **Юридическое:** Должен использоваться соответствующий геокодинговый сервис (API Яндекс.Карт для MVP согласно решению о стеке технологий).
- **Точность:** Расчеты расстояния должны быть достаточно точными для доверия пользователей (ошибка в пределах 100м допустима).
- **Производительность:** Запросы гео-поиска должны выполняться за <1s для 95% запросов при ожидаемой нагрузке.
- **Масштабируемость:** Должен эффективно обрабатывать 100k+ записей с геотегами.
- **Технологии:** Использовать PostgreSQL с расширением PostGIS или реализовать эффективную формулу Хаверсин в SQL.
- **Удобство использования:** Радиус поиска должен быть регулируемым через UI (ползунок/ввод) с четкими единицами расстояния (км).
- **Данные:** Данные о местоположении должны храниться как координаты широты/долготы (WGS84).

## Предыдущие решения
- Хранить местоположение как отдельные столбцы с плавающей точкой широты и долготы в таблицах Объявления и опционально Животных.
- **MVP (зафиксировано):** PostgreSQL `lat`/`lng` + Haversine + предфильтр bounding box, B-tree индексы по lat/lng. PostGIS/earthdistance — Фаза 2+ (ADR-0009).
- Для MVP реализовать гео-поиск с использованием формулы Хаверсин, оптимизированной с предварительной фильтрацией ограничивающего прямоугольника для снижения вычислительной нагрузки.
- Внешний геокодинг (адрес в координаты) будет обрабатываться через API Яндекс.Карт посредством абстракции frontend/backend.
- Максимальный радиус поиска применяется на уровне 100км для предотвращения злоупотребляющих запросов.
- Минимальный радиус поиска составляет 1км для обеспечения значимых результатов.
- Данные о местоположении обязательны для всех объявлений; пользователи должны предоставлять местоположение через взаимодействие с картой или ввод адреса.

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Запросы гео-поиска должны выполняться за <1s для 95% запросов при ожидаемой нагрузке (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Использует API Яндекс.Карт для геокодинга, как указано в решении о стеке технологий (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Радиус поиска должен быть регулируемым через UI (ползунок/ввод) с четкими единицами расстояния (км) (см. docs/02-requirements/nfr/accessibility.md)

## Пользовательские истории

### Функциональность гео-поиска
**UC-GS-01:** Как пользователь, ищущий животных или объявления рядом со мной, я хочу искать в пределах заданного радиуса, чтобы эффективно находить локальные предложения.
- Критерии приёмки:
  - Радиус поиска регулируется от 1 км до 100 км через ползунок или поле ввода
  - Определение текущего местоположения с разрешения пользователя
  - Ручной ввод местоположения через адрес или взаимодействие с картой
  - Результаты поиска показывают расстояние от местоположения пользователя
  - Гео-поиск выполняется за <1s для 95% запросов
  - Возможность сочетать гео-поиск с другими фильтрами (вид, порода, цена и т.д.)
  - Чёткое указание, когда в радиусе не найдено результатов

**UC-GS-02:** Как пользователь, заботящийся о приватности, я хочу управлять передачей своего местоположения, чтобы комфортно пользоваться платформой, защищая личную информацию.
- Критерии приёмки:
  - Явный запрос разрешения на доступ к местоположению
  - Возможность отключить службы геолокации и использовать только ручной ввод
  - Данные о местоположении хранятся минимально (нужны только широта/долгота для поиска)
  - Точный адрес не передаётся другим пользователям
  - Опция использовать приблизительное местоположение (уровень города/региона) для просмотра
  - Понятное объяснение, как используются и хранятся данные о местоположении

**UC-GS-03:** Как продвинутый пользователь, я хочу сохранять и повторно использовать избранные местоположения и настройки поиска, чтобы быстро обращаться к часто искомым областям.
- Критерии приёмки:
  - Сохранение текущего местоположения как избранного с собственным названием
  - Быстрый доступ к сохранённым местоположениям из интерфейса поиска
  - Сохранение фильтров поиска в сочетании с местоположением для поиска в один клик
  - Синхронизация сохранённых местоположений между устройствами (будущее улучшение)
  - Импорт/экспорт сохранённых местоположений (будущее улучшение)
  - **Скоуп:** сохранение/повторное использование поисков и локаций — **MVP** (хранится в таблице `saved_searches`). Проактивные **алерты** о новых подходящих объявлениях — **Phase 2** (см. `01-discovery/future-features.md`).

## Алгоритм, контракт результата и edge-cases (раунд 4, нормативно)

**Haversine + bounding-box (MVP):** радиус Земли `R = 6_371_000 м`.
- **Bbox-предфильтр** (использует B-tree по lat/lng): `Δlat = radius_m / 111_320`;
  `Δlng = radius_m / (111_320 * cos(radians(lat)))`; фильтр `lat BETWEEN lat0-Δlat AND lat0+Δlat`, аналогично lng.
- **Точное расстояние:** `d = 2*R*asin(sqrt( sin²((lat-lat0)/2) + cos(lat0)cos(lat) sin²((lng-lng0)/2) ))`, `d ≤ radius_m`.
- **Граница:** сравнение `≤` с допуском `±100 м` (NFR), поэтому «ровно на радиусе» — ВКЛЮЧАЕТСЯ, несмотря на float-погрешность.

**Контракт результата:** `ORDER BY distance_m ASC, created_at DESC, id ASC`; `distance_m` возвращается (округлённо);
пагинация `page`/`limit` (default 20, max 100); `total` = COUNT в радиусе. Всегда с `status='ACTIVE'` и фильтром по `market` животного.

**Edge-cases (нормативно):**
- **Антимеридиан (±180° lng):** при пересечении bbox ±180 (актуально для РФ: Чукотка/Камчатка) — разбить на два диапазона lng (`lng ≥ min` OR `lng ≤ max`).
- **Приполюсные:** клампить `Δlng` (cos→0) до 180°.
- **Нет координат:** объявления без `lat/lng` исключаются из гео-результатов (без fallback на центр города в MVP).
- **Радиус:** `1_000 ≤ radius_m ≤ 100_000`; вне — отклоняется.
- `listings.search_radius_m` — **не** фильтр гео-поиска (зарезервирован под «ищу в радиусе X» для matching); авторитетен радиус запроса.

**Combined search и saved searches:** гео + русский FTS (`to_tsvector('russian', …)`) + триграммы + атрибуты
(species/breed/price/type) — один запрос (bbox + bitmap-AND GIN-индексов). Схема `saved_searches.filters` =
параметры гео-запроса: `{ q?, species_id?, breed_id?, listing_type?, price_min?, price_max? }` + хранимые `lat/lng/radius_m`;
переисполнение маппит их в параметры `/geo-search`.
