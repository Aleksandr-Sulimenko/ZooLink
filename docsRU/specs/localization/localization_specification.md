# Спецификация локализации - ZooLink

## Обзор
Этот документ определяет требования к локализации (i18n/l10n), подход реализации и стандарты качества для системы ZooLink. Он определяет подход к поддержке множества языков, управлению переводами и обеспечению согласованного пользовательского опыта для разных локалей.

## Цели и требования локализации

### Поддерживаемые языки
| Язык | Код | Статус | Условие |
|------|-----|--------|---------|
| **Русский** | ru | Основной язык | Официальный язык Российской Федерации |
| **Английский** | en | Вторичный язык | Международный язык для большей доступности |
| **Будущие языки** | fr, es, zh и др. | Планируется для Фаза 2+ | На основе спроса пользователей и расширения рынка |

### Основные принципы локализации
1. **Единственный источник истины**: Все локализуемые данные хранятся в централизованных полях JSONB
2. **Механизм fallback**: Иерархический fallback языков (запрошенный → запасной → по умолчанию → пустая строка)
3. **Оптимизация производительности**: Эффективный запросов и индексация локализованных данных
4. **Обеспечение качества**: Обязательные переводы для основных языков в выпущенных функциях
5. **Расширяемость**: Простое добавление новых языков без изменений схемы БД

### Требования к модели данных
- **Структура JSONB**: Стандартизированный формат для всех локализуемых полей
  ```json
  {
    "ru": "значение на русском",
    "en": "значение на английском",
    "fr": "",
    "es": "",
    "zh": ""
  }
  ```
- **Пустые строки vs отсутствующие ключи**: Различие между намеренно пустыми переводами и отсутствующими ключами
- **Конфигурация языка**: Поддерживаемые языки управляются через таблицу базы данных

### Функциональные требования
1. **Локализация контента**: Все тексты, ориентированные на пользователя, должны быть локализуемыми
2. **Динамическое переключение языка**: Пользователи могут менять язык интерфейса без перезагрузки страницы
3. **Сохранение предпочтения языка**: Предпочтение языка пользователя хранится в профиле/сеансе
4. **Функции локализации на уровне базы данных**: SQL-функции для получения и проверки переводов
5. **Поиск и фильтрация**: Возможность поиска и фильтрации по локализованному контенту
6. **Административное управление переводами**: Интерфейс для управления переводами

### Нефункциональные требования
- **Производительность**: Запросы с локализацией не должны значительно ухудшать производительность
- **Масштабируемость**: Система должна поддерживать добавление новых языков без простоев
- **Сопровождаемость**: Четкое разделение между локализуемым контентом и кодом
- **Согласованность**: Единообразный подход к локализации во всех модулях

## Подход к реализации

### Улучшенный подход на основе JSONB
Мы выбрали улучшенный подход на основе JSONB для локализации, развивая текущую реализацию проекта.

#### 1. Стандартизированная структура JSONB
Все локализуемые поля будут следовать единой структуре JSONB с кодами языков в качестве ключей.

#### 2. Функции-помощники базы данных
Мы реализуем SQL-функции для упрощения работы с локализованными данными:

```sql
-- Функция для получения перевода с fallback
CREATE OR REPLACE FUNCTION get_localized(
    data JSONB,
    lang TEXT DEFAULT current_setting('app.current_language', true),
    fallback_lang TEXT DEFAULT 'en'
) RETURNS TEXT AS $$
BEGIN
    RETURN COALESCE(
        data->>lang,
        data->>fallback_lang,
        ''  -- окончательный fallback
    );
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- Функция для проверки наличия перевода
CREATE OR REPLACE FUNCTION has_translation(
    data JSONB,
    lang TEXT
) RETURNS BOOLEAN AS $$
BEGIN
    RETURN (data->>lang) IS NOT NULL AND (data->>lang) <> '';
END;
$$ LANGUAGE plpgsql IMMUTABLE;
```

#### 3. Оптимизация производительности
Мы добавим целевые GIN индексы для эффективного запросов:

```sql
-- Пример индекса для названий организаций
CREATE INDEX idx_organizations_name_localized_en 
ON organizations USING GIN ((name_localized -> 'en'));

CREATE INDEX idx_organizations_name_localized_ru 
ON organizations USING GIN ((name_localized -> 'ru'));
```

#### 4. Управление языками
Мы создадим таблицу для управления поддерживаемыми языками:

```sql
CREATE TABLE supported_languages (
    code CHAR(2) PRIMARY KEY,  -- ISO 639-1 код языка
    name_localized JSONB NOT NULL,  -- локализованное название языка
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    display_order INTEGER NOT NULL DEFAULT 0,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);
```

#### 5. Fallback на уровне приложения
Разрешение языка будет следовать этой иерархии:
1. Запрошенный язык (из предпочтений пользователя/сеанса)
2. Запасной язык (настроенный, например, 'en')
3. Язык по умолчанию (системный язык по умолчанию)
4. Пустая строка (если перевод недоступен)

## Стандарты качества локализации

### Полнота переводов
- **Обязательные языки**: Русский и английский должны иметь 100% охват переводами для всех выпущенных функций
- **Дополнительные языки**: Другие языки могут иметь частичный охват с четким указанием недостающих переводов
- **Поведение fallback**: Отсутствующие переводы возвращаются к следующему языку в цепочке

### Точность переводов
- **Профессиональный перевод**: Все переводы должны выполняться квалифицированными лингвистами
- **Техническая терминология**: Отраслевые стандартные термины должны оставаться непереведенными (например, UUID, JSONB, API)
- **Сохранение контекста**: Переводчики должны получать контекст о элементах интерфейса и ограничениях по символам
- **Процесс проверки**: Все переводы должны проходить вычиттку и обеспечение качества

### Согласованность форматов
- **Валидность JSONB**: Все локализуемые поля должны содержать валидный JSONB
- **Согласованность ключей**: Все локализуемые поля данной сущности должны поддерживать один и тот же набор языков
- **Кодировка**: UTF-8 кодировка для всего текстового контента
- **Экранирование**: Правильное экранирование специальных символов в строках JSON

## Сценарное тестирование локализации

### Функциональное тестирование
- **Переключение языка**: Проверить, что пользователи могут менять языки и видеть немедленные обновления
- **Поведение fallback**: Протестировать цепочки fallback при отсутствии переводов
- **Динамический контент**: Проверить локализацию динамически загружаемого контента
- **Метки форм и подсказки**: Убедиться, что все элементы форм правильно локализованы
- **Сообщения об ошибках**: Проверить, что сообщения об ошибках появляются на выбранном языке
- **Форматирование дат и чисел**: Подтвердить locale-специфичное форматирование дат, чисел и валют

### Тестирование производительности
- **Производительность запросов**: Измерить время ответа для запросов, связанных с локализованными полями
- **Использование индексов**: Проверить, что GIN индексы используются для запросов поиска по локализованным данным
- **Эффективность кеширования**: Протестировать эффективность любого механизма кеширования переводов
- **Одновременные пользователи**: Проверить производительность локализации при ожидаемой нагрузке пользователей

### Тестирование обеспечения качества
- **Полнота переводов**: Автоматизированные проверки на отсутствие переводов в обязательных языках
- **Проверка формата**: Убедиться, что все поля JSONB валидны и правильно структурированы
- **Проверки согласованности**: Обеспечить согласованную терминологию среди связанных элементов интерфейса
- **Проверка ограничений по символам**: Подтвердить, что переводы укладываются в отведенное место в интерфейсе
- **Обработка специальных символов**: Проверить корректное отображение акцентов, символов и специальных знаков

## Дорожная карта оптимизации локализации

### MVP (Фаза 1)
- Стандартизированная структура JSONB для всех локализуемых полей
- Функции-помощники базы данных (`get_localized`, `has_translation`)
- Базовая таблица управления языками (`supported_languages`)
- Механизм fallback языка на уровне приложения
- Исходный набор поддерживаемых языков: русский (ru) и английский (en)
- Базовое индексирование для оптимизации производительности

### Фаза 2 (Рост)
- Расширенная стратегия кеширования для часто используемых переводов
- Административный интерфейс для управления переводами
- Проверки полноты переводов в пайплайне CI/CD
- Дополнительная поддержка языков на основе спроса пользователей (французский, испанский и т.д.)
- Улучшенные возможности поиска по множеству языков
- Интеграция с системами машинного перевода для предварительных переводов

### Фаза 3 (Зрелость)
- Управление переводами с учетом контекста (разные переводы для разных контекстов)
- Системы памяти переводов и повторного использования
- Расширенные лингвистические особенности (множественное число, переводы с учетом рода)
- Инструменты совместной работы в реальном времени для переводчиков
- Непрерывный поток локализации, интегрированный с процессом разработки
- Аналитика использования языков и эффективности переводов
