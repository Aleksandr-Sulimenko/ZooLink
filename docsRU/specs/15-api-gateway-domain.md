# Спецификация: Домен шлюза API

## Результат
Предоставить безопасный, высокопроизводительный точку входа для всех клиентских запросов к платформе ZooLink. Обрабатывать сквозные проблемы, такие как аутентификация, ограничение скорости, преобразование запросов/ответов, маршрутизация и балансировка нагрузки, обеспечивая при этом наблюдаемость, безопасность и устойчивость.

## Область и границы
**Включено:**
- Маршрутизация запросов к соответствующим backend-сервисам (модулям) на основе пути и метода
- Проверка аутентификации (валидация JWT, валидация сессий)
- Проверка авторизации (проверка контроля доступа на основе ролей) для защищенных конечных точек
- Ограничение скорости (на IP, на пользователя, на конечную точку) для предотвращения злоупотреблений
- Преобразование запросов/ответов (манипуляция заголовками, обогащение тела)
- Терминация SSL/TLS и управление сертификатами
- Журналирование и мониторинг запросов/ответов (журналы доступа, метрики)
- Интеграция с доменом идентификации для контекста пользователя и валидации токенов
- Поддержка версионирования (версионирование по URL, версии по заголовкам)
- Обработка CORS и управление preflight-запросами
- Ограничение размера запросов и валидация полезной нагрузки
- Graceful degradation и паттерны выключателя для сбоев downstream-сервисов
- Endpoints проверки здоровья для мониторинга платформы
- Поддержка как RESTful API, так и WebSocket-соединений (в будущем)

**Исключено:**
- Реализация бизнес-логики (обрабатывается отдельными доменными модулями)
- Хранение или извлечение данных (обрабатывается соответствующими доменными модулями)
- Сложная оркестрация рабочих процессов (обрабатывается доменными сервисами или шаблонами saga)
- Потребление из очередей сообщений (обрабатывается воркерами)
- Обработка долговыполняющихся фоновых заданий
- Прямой доступ к базе данных (шлюз не должен запрашивать базы данных напрямую)
- Обнаружение сервисов микросервисов (если используется сервис-меш, обрабатывается мешем)
- Балансировка нагрузки между несколькими экземплярами шлюза (обрабатывается внешним балансировщиком нагрузки)

## Ограничения
- **Производительность:** Шлюз должен добавлять минимальную латентность (<50мс) к обработке запросов; обрабатывать 10k+ RPS с горизонтальным масштабированием.
- **Безопасность:** Д необходимо защищаться от уязвимостей OWASP Top 10 (инъекции, нарушенная аутентификация, раскрытие конфиденциальных данных и т.д.).
- **Надежность:** Д необходимо быть высокодоступным; отказ шлюза не должен приводить к падению всей платформы (проектирование для отказоустойчивости).
- **Масштабируемость:** Д необходимо поддерживать горизонтальное масштабирование; предпочтителен беcstatus-ный дизайн для легкой репликации.
- **Наблюдаемость:** Д необходимо предоставлять всестороннее журналирование, метрики и трассировку для отладки и мониторинга.
- **Технология:** Д необходимо соответствовать выбранному стеку (NestJS, TypeScript), но может использовать specialized библиотеки gateway или паттерны.
- **Совместимость:** Д необходимо быть совместимым с существующими модулями NestJS и их паттернами взаимодействия.
- **Развертываемость:** Д необходимо быть развертываемым в контейнеризованных средах (Kubernetes) и поддерживать rolling обновления.

## Предыдущие решения
- Шлюз API реализован как модуль NestJS с использованием `@nestjs/core` framework с пользовательским middleware и guards.
- Используются NestJS interceptors для преобразования запросов/ответов и журналирования.
- Используются NestJS guards для проверки аутентификации и авторизации.
- Используется NestJS middleware для ограничения скорости, журналирования и обработки CORS.
- Маршруты определяются динамически на основе зарегистрированных модулей (каждый модуль определяет свои собственные маршруты).
- Стратегия аутентификации: валидация JWT (access tokens) с опциональной обработкой refresh token.
- Авторизация: контроль доступа на основе ролей (RBAC) с использованием ролей из домена идентификации.
- Ограничение скорости: используется алгоритм token bucket или fixed window; настраивается на маршрут и тип пользователя.
- Журналирование: структурированное JSON-журналирование с correlation IDs для трассировки запросов.
- Мониторинг: предоставляются метрики Prometheus для количества запросов, латентности, частоты ошибок.
- Обработка ошибок: централизованный фильтр исключений, форматирующий ошибки согласно стандартному формату ошибок.
- SSL/TLS: терминируется на внешнем балансировщике нагрузки (ALB/Nginx); шлюз работает с HTTP внутренне.
- Версионирование: версионирование API через префикс URL (например, `/api/v1/`).
- Проверки здоровья: liveness and readiness probes для Kubernetes.

## Трассируемость NFR
Эта спецификация отвечает следующим нефункциональным требованиям:
- **Производительность (NFR-PERF)**: Шлюз добавляет <50мс латентности; 95% запросов <200мс общей латентности при нагрузочном тестировании (10k RPS) (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (NFR-SEC)**: Шлюз защищает от уязвимостей OWASP Top 10; реализует secure headers, ограничение скорости и валидацию auth (см. docs/02-requirements/nfr/security.md)
- **Доступность (NFR-AVAIL)**: Шлюз спроектирован для высокой доступности с проверками здоровья и graceful degradation (см. docs/02-requirements/nfr/availability.md)
- **Наблюдаемость (NFR-OBS)**: Шлюз предоставляет структурированное журналирование, метрики и трассировку (см. docs/02-requirements/nfr/observability.md)

## Разбивка на задачи
1. **Бэкенд (NestJS)**
   - [ ] Создать модуль `api-gateway` с помощью CLI NestJS
   - [ ] Определить механизм маршрутизации: динамическая регистрация маршрутов из модулей
   - [ ] Реализовать guard аутентификации (JwtAuthGuard) для валидации access tokens
   - [ ] Реализовать guard авторизации (RolesGuard) для проверки ролей и разрешений пользователя
   - [ ] Реализовать middleware ограничения скорости (используя redis or in-memory хранилище)
   - [ ] Реализовать middleware журналирования (journaling запросов/ответов с correlation IDs)
   - [ ] Реализовать transformation interceptors (обогащение заголовков, форматирование ответов)
   - [ ] Реализовать CORS middleware (настраиваемые origins, методы, заголовки)
   - [ ] Реализовать middleware ограничения размера запросов
   - [ ] Реализовать централизованный фильтр исключений для единообразных ответов об ошибках
   - [ ] Реализовать endpoints проверки здоровья (liveness, readiness)
   - [ ] Реализовать endpoint метрик для Prometheus (количество запросов, латентность, частоты ошибок)
   - [ ] Написать unit и интеграционные тесты для функциональности шлюза
   - [ ] Создать документацию OpenAPI (Swagger), которая агрегирует спецификации всех модулей

2. **Инфраструктура**
   - [ ] Настроить внешний балансировщик нагрузки (ALB/Nginx) для терминации SSL/TLS и переадресации на шлюз
   - [ ] Настроить Redis для хранения хранилища ограничения скорости (если используется распределенное ограничение скорости)
   - [ ] Настроить агрегацию журналирования (ELK стек или аналогичный)
   - [ ] Настроить мониторинг (Prometheus + Grafana) для метрик шлюза
   - [ ] Настроить трассировку (Jaeger или аналогичный) для трассировки запросов
   - [ ] Настроить security headers (Helmet) и CSP (Content Security Policy)
   - [ ] Реализовать стратегию rolling обновлений для развертываний шлюза
   - [ ] Спланировать горизонтальное автосмасштабирование (HPA) на основе CPU/memory/custom метрик

3. **Критерии верификации**
   - [ ] Unit-тесты обеспечивают покрытие >90% для модуля api-gateway (бэкенд)
   - [ ] Интеграционные тесты покрывают: маршрутизация, аутентификация, авторизация, ограничение скорости, журналирование, преобразование, обработка ошибок
   - [ ] Ручное тестирование: проверка защиты endpoint, поведения ограничения скорости, заголовков CORS, преобразований запросов/ответов
   - [ ] Производительность: шлюз добавляет <50мс латентности; 95% запросов <200мс общей латентности при нагрузочном тестировании (10k RPS)
   - [ ] Безопасность: проверка защиты от уязвимостей OWASP Top 10 (например, попытки SQLi блокируются, требуется аутентификация для защищенных endpoint)
   - [ ] Надежность: проверка грациозного обрабатывания сбоев downstream-сервисов (выключатель, резервные ответы)
   - [ ] Наблюдаемость: проверка, что журналы содержат correlation IDs; метрики предоставляются и собираются; трассировка работает
   - [ ] Документация: спецификация OpenAPI сгенерирована и доступна по адресу /api/docs
   - [ ] Трассируемость NFR: проверка, что требования производительности, безопасности, доступности и наблюдаемости правильно учтены и documented
