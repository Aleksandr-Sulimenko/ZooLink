# Спецификация: Домен организаций

## 1. Бизнес-цель и контекст
[БИЗНЕС-КОНТЕКСТ] Позволить организациям (ветеринарным клиникам, питомникам, приютам, фермам по разведению) эффективно управлять своим присутствием на платформе ZooLink, поддерживая многолокационные операции и сотрудничество персонала. Это улучшает бизнес-показатели за счет:
- Увеличения принятия платформы среди бизнес-пользователей (цель: 10k организаций в первый год)
- Обеспечения более высокозначимых транзакций через объявления, связанные с организациями
- Повышения точности данных о владении животными и атрибуции объявлений
- Поддержки соответствия российскому законодательству относительно идентификации юридических лиц

## 2. Глоссарий
- **Organization**: Юридическое лицо, которое может владеть животными, создавать объявления и иметь affiliated пользователей
- **Branch**: Физическое местоположение, принадлежащее организации
- **Organization User**: Платформенный пользователь, affiliated с одной или несколькими организациями
- **Headquarters**: Основная ветвь организации
- **INN**: Идентификационный номер налогоплательщика (Россия)
- **KPP**: Код причины постановки на учет (Россия)
- **Role in Org**: ENUM из {OWNER, ADMIN, STAFF, VET, MODERATOR}, определяющий разрешения пользователя внутри организации

### Контракты API Endpoints

#### Управление организацией
**POST /organizations**
- Запрос: OrganizationCreateDTO
- Ответ: OrganizationResponseDTO

**GET /organizations/{id}**
- Запрос: Path parameter: id (UUID)
- Ответ: OrganizationResponseDTO

**PATCH /organizations/{id}**
- Запрос: Path parameter: id (UUID), OrganizationUpdateDTO
- Ответ: OrganizationResponseDTO

**GET /organizations**
- Запрос: Query parameters: name (string), is_active (boolean), page (int), limit (int)
- Ответ: PaginatedOrganizationResponseDTO

#### Управление ветвями
**POST /branches**
- Запрос: BranchCreateDTO
- Ответ: BranchResponseDTO

**GET /branches/{id}**
- Запрос: Path parameter: id (UUID)
- Ответ: BranchResponseDTO

**PATCH /branches/{id}**
- Запрос: Path parameter: id (UUID), BranchUpdateDTO
- Ответ: BranchResponseDTO

**GET /branches**
- Запрос: Query parameters: organization_id (UUID), city_id (UUID), is_headquarters (boolean), page (int), limit (int)
- Ответ: PaginatedBranchResponseDTO

#### Управление связью организация-пользователь
**POST /organization-users**
- Запрос: OrganizationUserCreateDTO
- Ответ: OrganizationUserResponseDTO

**GET /organization-users**
- Запрос: Query parameters: organization_id (UUID), user_id (UUID), role_in_org (string), page (int), limit (int)
- Ответ: PaginatedOrganizationUserResponseDTO

**PATCH /organization-users/{id}**
- Запрос: Path parameter: id (UUID), OrganizationUserUpdateDTO
- Ответ: OrganizationUserResponseDTO

**DELETE /organization-users/{id}**
- Запрос: Path parameter: id (UUID)
- Ответ: SuccessResponseDTO

## 4. State Machine / Process Flow

### State Machine жизненного цикла организации
| Состояние | Описание | Действия при входе | Действия при выходе |
|-----------|----------|-------------------|---------------------|
| **PENDING_VERIFICATION** | Начальное состояние после создания; ограниченная функциональность | - Записать метку времени создания<br>- Отправить email верификации владельцу<br>- Установить токен верификации | Нет |
| **ACTIVE** | Полная функциональность доступна | - Очистить токен верификации<br>- Включить все функции организации<br>- Уведомить владельца об активации | Нет |
| **SUSPENDED** | Временно ограничена из-за нарушений правил | - Записать причину приостановки<br>- Отключить создание объявлений<br>- Ограничить административные действия<br>- Уведомить владельца | Нет |
| **ARCHIVED** | Навсегда деактивирован; только доступ для чтения | - Установить метку времени деактивации<br>- Анонимизировать чувствительные данные согласно 152-ФЗ<br>- Отключить все возможности модификации | Нет |

#### Переходы состояний организации
|_FROM_ | TO_ | Триггер | Условие срабатывания | Действие |
|--------|-----|---------|----------------------|----------|
| PENDING_VERIFICATION | ACTIVE | Верификация владельца завершена | Токен верификации валиден && данные организации заполнены | Включить полные функции; уведомить владельца |
| PENDING_VERIFICATION | ARCHIVED | Верификация не удалась | Превышено максимальное количество попыток верификации || обнаружены неверные данные | Записать причину неудачи; уведомить владельца |
| ACTIVE | SUSPENDED | Обнаружено нарушение правил | Действие модерации = SUSPEND || автоматическое обнаружение мошенничества | Записать нарушение; уведомить владельца |
| ACTIVE | ARCHIVED | Запрос владельца | Владелец явно запрашивает удаление && нет активных объявлений | Анонимизировать данные; записать архивацию |
| SUSPENDED | ACTIVE | Приостановка снята | Проверка завершена && выполнены корректирующие действия | Восстановить функции; уведомить владельца |
| SUSPENDED | ARCHIVED | Постоянные нарушения | Множественные приостановки в течение 30 дней || серьёзное нарушение правил | Анонимизировать данные; записать архивацию |
| * | ARCHIVED | Требование законодательства | Приказ суда или регулятивный мандат | Анонимизировать ПД; записать действие по соблюдению требований |

### State Machine жизненного цикла ветви
| Состояние | Описание | Действия при входе | Действия при выходе |
|-----------|----------|-------------------|---------------------|
| **ACTIVE** | Ветвь операционна и может использоваться для объявлений | - Проверить геокодирование адреса<br>- Включить ветвь для создания объявлений<br>- Уведомить администраторов организации | Нет |
| **INACTIVE** | Ветвь временно недоступна | - Установить метку времени инактивации<br>- Отключить ветвь для новых объявлений<br>- Уведомить администраторов организации | Нет |
| **CLOSED** | Ветвь постоянно закрыта | - Установить метку времени закрытия<br>- Предотвратить новые объявления<br>- Пометить существующие объявления как неактивные после периода льготы | Нет |

#### Переходы состояний ветви
|_FROM_ | TO_ | Триггер | Условие срабатывания | Действие |
|--------|-----|---------|----------------------|----------|
| ACTIVE | INACTIVE | Ручная инактивация | Пользователь является владельцем/администратором организации && ветвь не имеет активных объявлений | Установить метку времени инактивации; уведомить персонал |
| INACTIVE | ACTIVE | Повторная активация | Пользователь является владельцем/администратором организации || адрес повторно проверен | Очистить метку времени инактивации; уведомить персонал |
| ACTIVE | CLOSED | Постоянное закрытие | Пользователь является владельцем/администратором организации || 6+ месяцев инактивации | Установить метку времени закрытия; начать период льготы |
| INACTIVE | CLOSED | Постоянное закрытие из неактивного состояния | Пользователь является владельцем/администратором организации || 6+ месяцев инактивации | Установить метку времени закрытия |
| CLOSED | * | Переходы не допускаются | Запрещено | Ошибка: Закрытая ветвь не может быть модифицирована |

### State Machine аффилиации организация-пользователь
| Состояние | Описание | Действия при входе | Действия при выходе |
|-----------|----------|-------------------|---------------------|
| **PENDING_INVITE** | Пользователь приглашен, но еще не принял | - Отправить email-приглашение<br>- Сгенерировать токен приглашения<br>- Установить срок действия (7 дней) | Нет |
| **ACTIVE** | Пользователь affiliated с организацией | - Очистить токен приглашения<br>- Назначить разрешения роли<br>- Уведомить администраторов организации | Нет |
| **REVOKED** | Аффилиация прекращена | - Записать метку времени отзыва<br>- Удалить разрешения роли<br>- Уведомить пользователя и организацию | Нет |
| **EXPIRED** | Приглашение не принято | - Очистить токен приглашения<br>- Записать событие истечения срока | Нет |

#### Переходы состояний аффилиации организация-пользователь
|_FROM_ | TO_ | Триггер | Условие срабатывания | Действие |
|--------|-----|---------|----------------------|----------|
| PENDING_INVITE | ACTIVE | Приглашение принято | Токен приглашения валиден && не истек срок действия | Активировать аффилиацию; уведомить администраторов организации |
| PENDING_INVITE | EXPIRED | Приглашение не принято | Токен приглашения истек срок действия | Записать событие истечения срока; уведомить отправителя |
| PENDING_INVITE | REVOKED | Приглашение отозвано | Отправитель является владельцем/администратором организации || администратор системы | Записать отзыв; уведомить приглашенного |
| ACTIVE | REVOKED | Аффилиация прекращена | Запрашивающая сторона является владельцем/администратором организации || пользователь запрашивает удаление | Записать прекращение; уведомить обе стороны |
| * | REVOKED | Мандат системы | Требование законодательства || нарушение безопасности | Записать причину; уведомить затронутые стороны |

### Бизнес-логика - Сценарии Gherkin

#### Создание организации
```gherkin

  Предусловие:
    Дано система операционна
    И доступны валидные учетные данные администратора

  Сценарий: Создание организации с российскими налоговыми идентификаторами
    Дано я предоставляю данные организации с:
      | name_localized | {"en": "Vet Clinic", "ru": "Ветеринарная клиника"} |
      | description_localized | {"en": "Animal care", "ru": "Уход за животными"} |
      | inn | "7707083893" |
      | kpp | "773601001" |
      | address | "123 Veterinary St, Moscow" |
      | phone | "+74951234567" |
      | email | "info@vetclinic.ru" |
    Когда я отправляю запрос на создание организации
    То организация должна быть создана со статусом PENDING_VERIFICATION
    И должно быть отправлено email-уведомление владельцу
    И ИНН и КПП должны быть проверены на уникальность

  Сценарий: Создание организации без налоговых идентификаторов (индивидуальный предприниматель)

    Когда я отправляю запрос на создание организации

    И поля ИНН и КПП должны оставаться null
```

## 7. Требования нефункционального характера

### Требования к производительности
- Задержка API создания организации < 800ms для 95% запросов при нормальной нагрузке
- Задержка API получения организации < 300ms для 95% запросов при нормальной нагрузке
- Задержка API создания ветви < 500ms для 95% запросов при нормальной нагрузке
- Задержка API аффилиации организация-пользователь < 400ms для 95% запросов при нормальной нагрузке
- Поиск организаций по названию: < 1s для 95% запросов при 100k+ организациях
- Список ветвей для организации: < 500ms для 95% запросов

### Требования к безопасности
- Все операции с чувствительными данными организации требуют аутентификации
- Создание организации требует верификации email
- Только пользователи с ролями OWNER или ADMIN могут модифицировать детали организации/ветви
- Пользователи могут создавать объявления только для организаций, с которыми у них есть активная аффилиация (проверено через organization_users)
- Контактные данные организации (телефон, email) отображаются только если организация решила их делиться
- Личные данные пользователя защищены; объявления организации не раскрывают контактную информацию индивидуального создателя, если организация не решила её раскрыть
- Все модификации организации регистрируются для аудиторского следа
- Ограничение скорости: максимум 10 запросов создания организации в час на IP-адрес
- Ограничение скорости: максимум 100 запросов аффилиации в час на IP-адрес

### Требования к надежности
- Сохранность данных организации: 99.9% ежемесячного uptime
- Ежедневные автоматические резервные копии данных организации
- Удаление организации следует шаблону архивации (мягкое удаление) для соблюдения требований
- Хранение аудиторского следа: минимум 5 лет для соблюдения законодательства

### Требования к масштабируемости
- Поддержка до 100,000 организаций без ухудшения производительности
- Поддержка до 1,000,000 ветвей без ухудшения производительности
- Поддержка до 10,000,000 аффилиаций организация-пользователь без ухудшения производительности
- Стратегия индексирования базы данных оптимизирована для:
  - Поиска организации по ИНН
  - Поиска ветви по organization_id + city_id
  - Поиска аффилиации по organization_id + user_id
  - Поиска организации по name_localized

### Требования к наблюдаемости
- Все переходы состояний организации журналируются с: меткой времени, organization_id, user_id (если применимо), previous_state, new_state
- События создания/обновления/удаления организации отправляются в сервис аудиторского лога
- Неудачные попытки верификации журналируются для мониторинга безопасности
- Метрики организации: активные организации, новые регистрации ежемесячно, уровень приостановок
- Пороги тревог: уровень приостановок > 5% ежечасно, неудачные попытки верификации > 100/час

## 8. Открытые вопросы / Предположения

### Предположения
1. Система использует UUID v4 для всех идентификаторов сущностей
2. Справочные данные по городам поступают из домена Администрирования (таблица городов)
3. Верификация email для создания организации использует тот же механизм, что и в домене Идентификации
4. Система следует тем же шаблонам аутентификации и авторизации, что и другие домены (на основе JWT)
5. Структура локализации следует паре ключ-значение в JSONB с кодами языков в качестве ключей (ISO 639-1)
6. Процесс верификации организации следует аналогичным шаблонам верификации пользователя в домене Идентификации
7. Валидация геокодирования для адресов использует внешний сервис (Yandex.Maps API, как указано в техническом стеке)
8. Поле metadata JSONB следует предопределенной схеме для известных расширений (уровни подписки и т.д.)
9. Процесс архивации организации анонимизирует персональные данные в соответствии с 152-ФЗ, но сохраняет аудиторский след
10. Система будет использовать тот же формат обработки ошибок, что определен в error_handling/standard_error_format.md

### Открытые вопросы
1. Какой exact формат и правила валидности для поля метаданных JSONB? Следует ли определить JSON-схему для расширяемости?
2. Следует ли верификация организации включать загрузку документов (ОГРН, свидетельство о регистрации) или полагаться исключительно on ИНН/КПП валидацию?
3. Каковы особые бизнес-правила для классификации типа организации (ветеринарная клиника, питомник, приют и т.д.)? Следует ли сделать это отдельной справочной таблицей?
4. Следует ли внедрять значки верификации организации (верифицированный/галочка), как упоминалось в реестре GAP?
5. Каков период льготы для закрытия ветви перед автоматическим деактивированием существующих объявлений?
6. Следует ли внедрять политики модерации на уровне организации (например, запрещенные породы) как флаги функций?
7. Каковы конкретные ограничения скорости для различных операций организации (создание, модификация, аффилиация)?
8. Следует ли поддерживать иерархические структуры организаций (родитель/дочерняя организация) для франшиз или крупных сетей?
9. Какие поля должны быть включены в поисковый индекс для поиска названия организации (полнотекстовый поиск vs точное совпадение)?
10. Следует ли внедрить API endpoint аналитики организации, как упоминалось в документе бизнес-требований?
