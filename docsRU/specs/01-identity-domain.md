# Спецификация: Домен идентификации

## Результат
Обеспечить безопасную аутентификацию и авторизацию пользователей ZooLink, включая владельцев домашних животных, заводчиков, фермеров, модераторов и администраторов. Включить регистрацию пользователей, вход в систему, управление профилями и контроль доступа на основе ролей.

## Область и границы
**Включает:**
- Регистрация пользователей через телефон (SMS) и OAuth (Google, Apple, Telegram, VK)
- Проверка электронной почты (опционально)
- Проверка телефона через код SMS
- Аутентификация на основе JWT с токенами обновления
- Управление профилем пользователя (просмотр, обновление)
- Управление ролями и разрешениями (владелец домашнего животного, заводчик, фермер, модератор, администратор)
- Вход без пароля (код SMS)
- Управление сессиями

**Исключает:**
- Социальные функции (чат, форумы и т.д.) - отложено на этап 2
- Расширенные функции безопасности (биометрия, аппаратные ключи) - отложено
- Удаление.account/pраво на забвение GDPR - отложено на этап 2
- Инструменты олицетворения администратора/поддержки - отложено

## Ограничения
- **Юридическое:** Должно соответствовать Федеральному закону РФ 152-ФЗ (Персональные данные) для хранения и обработки данных пользователей.
- **Безопасность:** Пароли не используются; аутентификация только через телефон/OAuth. Защита от перебора, подмены SIM-карт (ограничение частоты запросов на SMS).
- **Удобство использования:** Процесс регистрации должен быть простым для непрофессиональных пользователей (массовые владельцы домашних животных).
- **Производительность:** Задержка аутентификации < 1с при нормальной нагрузке.
- **Масштабируемость:** Система должна поддерживать 100k+ активных пользователей.
- **Технологии:** Должно соответствовать выбранному стеку (NestJS, TypeScript, PostgreSQL, Redis).
- **Данные:** Личные данные (номер телефона, имя) должны храниться безопасно и минимально.

## Трассируемость НФТ
Эта спецификация охватывает следующие нефункциональные требования:
- **Производительность (НФТ-ПРОИЗВ)**: Задержка аутентификации < 1с при нормальной нагрузке; задержка API аутентификации < 800ms для 95% запросов при нагрузочном тесте (100 RPS) (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (НФТ-БЕЗОП)**: Пароли не используются; аутентификация только через телефон/OAuth; защита от перебора, подмены SIM-карт; хранение данных соответствует 152-ФЗ (см. docs/02-requirements/nfr/security.md)
- **Доступность (НФТ-ДОСТУП)**: Процесс регистрации должен быть простым для непрофессиональных пользователей (массовые владельцы домашних животных); следует рекомендациям WCAG 2.1 AA (см. docs/02-requirements/nfr/accessibility.md)

## Предыдущие решения
- Технологический стек (ADR 0001): backend NestJS, frontend React, PostgreSQL, Redis.
- Метод аутентификации: Телефон (SMS) + провайдеры OAuth; электронная почта опциональна.
- Нет аутентификации по имени пользователя/паролю.
- JWT хранится во frontend (cookie с httpOnly или локальное хранилище с мерами защиты).
- Управление доступом на основе ролей (RBAC) с ролями: пользователь, заводчик, фермер, модератор, администратор.
- Сущность пользователя включает: id, phoneNumber (подтвержден), firstName, lastName, role, createdAt, updatedAt, isActive.

## Разбивка задач
1. **Бэкенд (NestJS)**
   - [ ] Создать модуль `identity` с помощью NestJS CLI
   - [ ] Определить сущность User (TypeORM/Prasmic) с полями: id, phoneNumber, firstName, lastName, role, isActive, createdAt, updatedAt
   - [ ] Реализовать сервис проверки телефона (отправка SMS через abstraction Twilio)
   - [ ] Реализовать стратегии проверки OAuth (Google, Apple, Telegram, VK)
   - [ ] Создать AuthController (регистрация, вход, обновление токена, профиль)
   - [ ] Создать AuthService (валидация учетных данных, генерация JWT/токенов обновления)
   - [ ] Создать JwtStrategy и RefreshTokenGuard
   - [ ] Реализовать защитники на основе ролей (RolesGuard)
   - [ ] Настроить ограничение частоты запросов для конечных точек SMS
   - [ ] Написать модульные и интеграционные тесты для потоков аутентификации
   - [ ] Создать документацию OpenAPI (Swagger) для конечных точек аутентификации

2. **Фронтенд (React)**
   - [ ] Создать страницы аутентификации: Регистрация, Вход, Профиль
   - [ ] Реализовать ввод телефона с выбором кода страны
   - [ ] Реализовать форму подтверждения кода SMS
   - [ ] Реализовать кнопки входа через OAuth
   - [ ] Создать контекст/хуки аутентификации (useAuth, useUser) для управления состоянием
   - [ ] Защищать маршруты на основе аутентификации и роли
   - [ ] Реализовать механизм обновления токена
   - [ ] Создать страницу просмотра/редактирования профиля пользователя
   - [ ] Написать модульные и end-to-end тесты для потоков аутентификации

3. **Инфраструктура**
   - [ ] Настроить Redis для хранения токенов обновления (или использовать JWT с токенами обновления в базе данных)
   - [ ] Настроить учетные данные поставщика SMS (Twilio) в окружении
   - [ ] Настроить учетные данные провайдеров OAuth (Google, Apple, и т.д.)
   - [ ] Добавить заголовки безопасности (helmet) и конфигурацию CORS
   - [ ] Внедрить логирование событий аутентификации (успех/неудача)

## Критерии верификации
- [ ] Модульные тесты достигают >90% покрытия для модуля идентификации (backend)
- [ ] Интеграционные тесты покрывают: регистрацию через телефон, регистрацию через OAuth, вход, обновление токена, обновление профиля, контроль доступа на основе ролей
- [ ] End-to-end тесты (Cypress/Playwright) покрывают полный поток пользователя: регистрация -> вход -> доступ к защищенному маршруту -> выход
- [ ] Ручное тестирование: проверка времени доставки SMS (<10s), работоспособность потоков переадресации OAuth
- [ ] Обзор безопасности: отсутствие утечки конфиденциальных данных в ответах, токены являются HTTP-only или должным образом защищены
- [ ] Производительность: задержка API аутентификации < 800ms для 95% запросов при нагрузочном тесте (100 RPS)
- [ ] Соответствие: хранение данных соответствует 152-ФЗ (минимальные личные данные, согласие на обработку)
- [ ] Документация: спецификация OpenAPI сгенерирована и доступна по адресу /api/docs
- [ ] Трассируемость НФТ: проверить, что требования производительности, безопасности и доступности корректно учтены и документированы
