# Спецификация: Домен уведомлений

## Результат
Предоставить надежный сервис уведомлений для доставки своевременных оповещений пользователям через email и SMS. Позволить платформе отправлять транзакционные уведомления (действия с учетными записями, решения модерации, предложения по подбору) и рекламные уведомления (обновления сервиса, объявления о функциях) при обеспечении надежности доставки, отслеживания и соблюдения нормативов связи.

## Область и границы
**Включено:**
- Сервис уведомлений email (через **Unisender** по умолчанию; за портом `EmailProvider` — см. [ADR-0008](../04-decisions/0008-rf-provider-matrix.md))
- Сервис уведомлений SMS (через **SMS.RU** по умолчанию; за портом `SmsProvider` — см. [ADR-0008](../04-decisions/0008-rf-provider-matrix.md)). SendGrid/Twilio **не работают в РФ**.
- Шаблонное содержимое уведомлений с поддержкой локализации
- Отслеживание доставки (отправлено, доставлено, не удалось, отскок)
- Ограничение скорости и throttling для соблюдения лимитов провайдеров
- Предпочтения пользователей по уведомлениям (включение/отключение для разных типов уведомлений)
- Механизм очереди для обработки всплесков уведомлений
- Логика повторных попыток для неудачных доставок (экспоненциальная отсрочка)
- Интеграция с другими доменами (Идентификация, Модерация, Подбор и т.д.) через события или прямые вызовы сервиса
- Поддержка как транзакционных, так и рекламных уведомлений
- Механизм отписки для рекламных коммуникаций
- Локализация содержимого уведомлений (английский/русский)
- Поддержка богатого содержимого (HTML-письма, Unicode SMS)

**Исключено:**
- Push-уведомления (мобильные/приложения) - откладывается на этап 2
- Внутренние уведомления (в веб-интерфейсе) - откладывается на этап 2
- Голосовые уведомления - откладывается на этап 2
- Уведомления чат-ботов (Telegram, WhatsApp) - откладывается на этап 2
- Расширенная аналитика (A/B тестирование, отслеживание вовлеченности) - откладывается на этап 2
- Планирование уведомлений (отправка в определенное время) - откладывается на этап 2

## Ограничения
- **Юридическое:** Соблюдать Федеральный закон 152-ФЗ (Персональные данные) при обработке контактной информации пользователей и закон о рекламе/антиспам (38-ФЗ «О рекламе» — согласие на рекламные сообщения). ПДн остаются внутри инфраструктуры РФ.
- **Производительность:** Задержка вызова API уведомлений < 500ms; фактическое время доставки зависит от провайдера, но должно быть < 10с для SMS, < 30с для email при нормальных условиях.
- **Надежность:** Система должна достигать >99% успешности доставки для валидных уведомлений; неудачные уведомления должны быть повторимыми.
- **Удобство использования:** Содержание уведомлений должно быть четким, лаконичным и выполнимым; пользователи должны легко понимать, как отписаться или управлять предпочтениями.
- **Масштабируемость:** Система должна поддерживать 100k+ уведомлений в день.
- **Технология:** Должна соответствовать выбранному стеку (NestJS, TypeScript, PostgreSQL, Redis).
- **Данные:** Журналы уведомлений должны храниться для аудита и устранения неполадок; личные данные (email, телефон) должны обрабатываться в соответствии с 152-ФЗ.
- **Стоимость:** Затраты на уведомления должны контролироваться и оптимизироваться; следует максимизировать использование бесплатных уровней перед возникновением затрат.

## Предыдущие решения
- Сервис уведомлений реализован как отдельный модуль NestJS с провайдерами для email и SMS.
- Используются РФ-провайдеры (**Unisender** для email, **SMS.RU** для SMS) через их API за портами `EmailProvider`/`SmsProvider` ([ADR-0008](../04-decisions/0008-rf-provider-matrix.md)).
- Шаблоны уведомлений хранятся как handlebars-шаблоны с поддержкой локализации.
- Запросы на уведомления помещаются в очередь в Redis (или базе данных) для асинхронной обработки, чтобы не блокировать вызовы API.
- Каждая попытка уведомления логируется со статусом, ответом провайдера и временными метками.
- Предпочтения пользователей по уведомлениям хранятся в домене Идентификации (сущность пользователя или отдельная таблица предпочтений).
- Ограничение скорости реализовано на уровне сервиса для соблюдения лимитов провайдеров.
- Неудачные уведомления повторяются с экспоненциальной отсрочкой (максимум 3 попытки).
- Локализация содержимого уведомлений использует тот же механизм, что и фронтенд (библиотеки i18n).
- Ссылки для отписки включаются в рекламные уведомления и управляются через домен Идентификации.

## Трассируемость NFR
Эта спецификация отвечает следующим нефункциональным требованиям:
- **Производительность (NFR-PERF)**: Задержка API уведомлений < 500ms для 95% запросов при нагрузочном тестировании (100 RPS) (см. docs/02-requirements/nfr/performance.md)
- **Безопасность (NFR-SEC)**: Сервис уведомлений защищает контактную информацию пользователей; API-ключи хранятся безопасно (см. docs/02-requirements/nfr/security.md)
- **Доступность (NFR-ACC)**: Содержание уведомлений следует рекомендациям доступности для читаемости (см. docs/02-requirements/nfr/accessibility.md)

## Разбивка на задачи
1. **Бэкенд (NestJS)**
   - [ ] Создать модуль `notification` с помощью CLI NestJS
   - [ ] Определить модель NotificationLog (Prisma) с полями: id, userId, тип (EMAIL/SMS), templateId, получатель, тема/содержание, статус (SENT/DELIVERED/FAILED/BOUNCED), ответ провайдера, попытки, createdAt, updatedAt
   - [ ] Определить модель NotificationTemplate (Prisma) для управления шаблонами (id, имя, тип, subjectTemplate, bodyTemplate, язык, isActive)
   - [ ] Реализовать NotificationController (отправка уведомления, получение журналов, управление предпочтениями)
   - [ ] Реализовать NotificationService (бизнес-логика очереди, рендеринга шаблонов, интеграции с провайдерами)
   - [ ] Реализовать адаптеры `EmailProvider` (Unisender) и `SmsProvider` (SMS.RU) за портами
   - [ ] Реализовать очередь уведомлений (на основе Redis или базы данных)
   - [ ] Реализовать механизм повторных попыток с экспоненциальной отсрочкой
   - [ ] Реализовать ограничение скорости за провайдером и за пользователем
   - [ ] Создать сервис рендеринга шаблонов (handlebars с локализацией)
   - [ ] Написатьunit и интеграционные тесты для потоков уведомлений

2. **Инфраструктура**
   - [ ] Настроить Redis для очереди уведомлений (или использовать таблицы базы данных)
   - [ ] Настроить учетные данные РФ-провайдеров (ключ API Unisender, api_id SMS.RU) в окружении
   - [ ] Настроить журналирование событий уведомлений (отправлено, не удалось, повторно)
   - [ ] Добавить заголовки безопасности и конфигурацию CORS
   - [ ] Реализовать мониторинг показателей доставки уведомлений и затрат провайдеров

3. **Критерии верификации**
   - [ ] Unit-тесты обеспечивают покрытие >90% для модуля уведомлений (бэкенд)
   - [ ] Интеграционные тесты покрывают: очередь уведомлений, рендеринг шаблонов, интеграция с провайдерами (с mock), логика повторных попыток, ограничение скорости
   - [ ] Ручное тестирование: проверка доставки email/SMS с реальными провайдерами (используя тестовые учетные данные), проверка журналов, проверка обработки предпочтений
   - [ ] Производительность: задержка API уведомлений < 500ms для 95% запросов при нагрузочном тестировании (100 RPS)
   - [ ] Надежность: достижение >99% успешности доставки в тестовых сценариях с имитацией сбоев провайдеров
   - [ ] Безопасность: проверка, что API-ключи не раскрываются в журналах или ответах
   - [ ] Документация: спецификация OpenAPI сгенерирована и доступна по адресу /api/docs
   - [ ] Трассируемость NFR: проверка, что требования производительности, безопасности и доступности правильно учтены и documented
