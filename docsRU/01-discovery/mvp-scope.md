# MVP Scope: ZooLink

## In Scope (Что делаем в Фазе 1 – MVP)

### Core User Journeys
1. **Регистрация и аутентификация**
 - По номеру телефона (SMS-код)
 - Через провайдеров OAuth: Google, Apple, Telegram, VK
 - Верификация email опциональна (не обязательна для MVP)
 - Минимальный профиль: имя, фото аватара, город (из справочника)

2. **Сущность "Животное" (Animal Domain)**
 - Создание профиля животного как отдельной Aggregate Root
 - Атрибуты: вид (кошка/собака/КРС/МРС и т.д.), порода, пол, дата рождения, окрас, метки (чип/тату)
 - Возможность загрузить 1-3 фото (превью в списке, галерея при открытии)
 - Статус: `active`/`archived` (архивация вместо удаления для историчности)

3. **Marketplace объявлений (Listing Domain)**
 - Создание объявления двух типов:
 - **Продажа**: цель – передача ownership за вознаграждение или даром
 - **Вязка**: goal – найти партнера для разведения (указание дат течки, желаемых характеристик партнера)
 - Обязательные поля при создании:
 - Ссылка на существующее профиль животного (животное – сущность, объявление – связь)
 - Тип сделки (продажа/вязка)
 - Цена/условия (для продажи: фикс. сумма/договорная/бесплатно; для вязки: под Pick/нанесение оплаты/бесплатно)
 - Геолокация (точка или радиус поиска; обязательна для модерации)
 - Описание (текст до 2000 символов)
 - Фото (1-5 шт., минимум 1)
 - Фильтры поиска:
 - По типу (продажа/вязка)
 - По виду животного (кошка/собака/птица/КРС/МРС/мелкие сельхоз и т.д.)
 - По породе (из справочника, поддерживает кастомные значения)
 - По гео-радиусу (1-100 км от точки пользователя)
 - По цене (мин/макс)
 - По датам (для вязки: течка ожидаема/прошла)
 - Результаты поиска:
 - Карточка объявления: фото животного, вид/порода, цена/условия, расстояние
 - Сортировка: по дате публикации (новые первые), по расстоянию, по цене

4. **Предмодерация и жизненный цикл объявления**
 - Все новые объявления поступают в статус `PENDING_MODERATION`
 - Модератор (роль в системе) видит список в админке и выполняет проверку:
 - Соответствие правилам публикации (нет запрещенного контента, фото соответствует объявленному виду/породе)
 - Полнота заполнения обязательных полей
 - Отсутствие явного спама/мошенничества (шаблонные тексты, подозрительные цены)
 - Возможные итоги модерации:
 - `APPROVED` → объявление публикуется и становится видимым в поиске
 - `REJECTED` → возвращается пользователю с комментарием (можно исправить и повторно отправить)
 - После модерации статус больше не меняется автоматически (архивация – ручное действие пользователя)

5. **Взаимодействие между пользователями (без чата)**
 - После модерации объявления:
 - Покупатель/ interested party видит кнопку "Показать контакты"
 - При нажатии раскрывается: номер телефона продавца (если предоставлен при регистрации) + ссылки на профили в соцсетях (Telegram/VK, если привязаны)
 - Система фиксирует событие "Контакты показаны" для аналитики (кто показал, когда, какое объявление)
 - Альтернатива: пользователь может оставить премодераированный комментарий к объявлению (текст до 500 символов, видимый всем после модерации комментария)

6. **Административная панель (Admin Domain)**
 - Доступ только для ролей `moderator` и `admin`
 - Функции:
 - Очередь объявлений на модерации (фильтры: по виду животного, типу сделки, дате подачи)
 - Инструменты модерации: одобрить/отклонить с обязательным комментарием при отклонении
 - Управление справочниками: породы, виды животных, метки о здоровья (прививки, тесты)
 - Бан пользователей (временный/постоян) с указанием причины
 - Просмотр базовой аналитики: кол-во объявлений на модерации, среднее время модерации, топ-причин отклонения
 - Упрощенный UI: таблицы + формы, без графиков на MVP

### Технические границы MVP
- **Фронтенд**: SPA на React/Vue (выбор в ADR 0001) + PWA-обертка для офлайн-кеширования статики и базовых данных (справочники, профили)
- **Бэкенд**: NestJS модули по доменам (Identity, Animal, Listing, Admin, Moderation)
- **База данных**: PostgreSQL с схемами, соответствующими bounded contexts
- **Хранение файлов**: AWS S3-совместимое хранилище (MinIO в dev) для фото животных и объявлений
- **Кеширование**: Redis для сессий и справочников (породы, виды) – TTL 24ч
- **Интеграции на MVP**:
 - SMS-шлюз (Twilio или аналог с бесплатным tier) для 2FA
 - OAuth провайдеры (Google, Apple, Telegram, VK) – через библиотеки
 - Геокодер и поиск по радиусу: Яндекс.Карты (бесплатный tier до 10 000 запросов/день) → myöhemmin можно заменить на Mapbox или самописный с PostGIS
 - Email-сервис (SendGrid или аналог) только для системных уведомлений (регистрация, модерация результат)

### Нефункциональные ограничения (которые снимаются позже)
| Ограничение | Планируемое изменение | Пояснение |
|-------------|----------------------|-----------|
| **Веб-only (PWA)** | Добавление нативных приложений (iOS/Android) | Архитектура backend-for-frontend (BFF) позволяет добавить отдельные эндпоинты под мобильные клиенты без изменения core API |
| **Предмодерация всех объявлений** | Постмодерация с тригерами для повторной проверки | При росте объема можно внедрить гибридную модель: автопроверка ML-моделью + постмодерация жалоб |
| **Один язык интерфейса** | Мультиязычность (RU + EN + другие) | Готово через i18n-файлы и локализациюぅ в сущностях |
| **Ограничение на фото в объявлении (5 шт.)** | Альбомы и видео | Хранение в S3 позволяет легко увеличить лимиты или добавить типы контента позже |

## Архитектурные соображения для расширяемости

### Как мы избегаем "маштабного переписывания"
1. **Разделение по bounded contexts (DDD)**
 - Каждый домен (Identity, Animal, Listing, Admin) – отдельный модуль с CLEAN границами
 - Коммуникация через événements (платформа: NestJS CQRS + события) или синхронные вызовы только через well-defined API-контракты
 - Пример: добавление чата потребует нового доменаMessaging, который будет подписываться на событияListingCreated иListingContactShown, не меняя существующих доменов

2. **Контракт-first подход к API**
 - Все API описаны в OpenAPI/YAML (в`03-architecture/api-contracts/`)
 - Будущие изменения делаются через добавление новых полей/эндпоинтов, не удаляя старые (обратная совместимость на уровне версии v1)
 - Пример: добавление поляvideo_url в сущностьListing можно сделать через backward-compatible изменение схемы

3. **Расширяемая схема БД**
 - Используем PostgreSQL с JSONB-хранилищем для редко используемых/экспериментальных атрибутов (например, в сущностиЖивотного полеextra_attributes JSONB)
 - Миграции пишутся только для изменений core-структуры, не для добавления опциональных полей
 - Пример: добавление поляgenetic_test_results вЖивотное – сначала как JSONB, потом при стабилизации – вынос в отдельную таблицу

4. **Feature toggles и абстракции**
 - Критические الحدود (например, монетация, чат) защищены за выключателями в конфигурации
 - В коде проверки типа: `if (featureToggle.isEnabled('CHAT_MODULE')) { ... }`
 - Позволяет включать/выключать функциональность без деплоя нового кода

5. **Разделение ответственности в хранилище файлов**
 - Фото объявлений хранятся в бакете/папке`/listings/{listingId}/photos/`
 - Легко добавить папки для видео (`/videos/`), альбомов (`/albums/`) или документов (`/documents/`) без изменения логики загрузки

## Связь с целями по фазам (из Problem Statement)
- **Фаза 1 цели (500+ объявлений, 70% прошедших модерацию)** достигаются за счет In Scope функционала: простая подача объявлений + эффективная premoderation workflow
- **Фаза 2 цели (MRR 150k₽, время до сделки ≤14д)** готовятся через Out of Scope элементы: Boost-листингов (изменит ранжирование в поиске), чат (ускоряет переговоры), аналитика для продавцов (помогает ценообразовать)
- **Фаза 3 цели (комиссия в эскроу, B2B SaaS)** готовятся через архитектурные заделки: разделение витрин pet/livestock упрощает ввод разных правил комиссии, готовые поля для внешних ID облегчают интеграцию с Меркурием

## Ограничения и допущения (из Assumptions)
- **Считаем, что Яндекс.Карты бесплатны до 10к запросов/день** – если лимит будет превышен, переходим на собственные данные OSM + PostGIS (архитектура допускает замену геопровайдера без изменения контрактов)
- **Считаем, что moderation workload на MVP<50 объявлений/день** – при росте вводим автоматическую пре-модерацию через ML-модели (текст/фото) как отдельный сервис
- **Считаем, что основной ток пользователей – мобильные устройства** – поэтому PWA-оптимизация важнее нативной скорости на этапе валидации гипотезы
