# Стратегии развертывания для ZooLink

**Статус**: Черновик  
**Дата**: 2026-06-13  

## Обзор

В этом документе описываются стратегии развертывания, конфигурации окружения и процедуры выпуска для платформы ZooLink. Подход к развертыванию разработан для поддержки архитектуры модульного монолита, готового к микросервисам, использующего NestJS, PostgreSQL, Redis и хранилище, совместимое с S3.

## Окружения развертывания

ZooLink поддерживает четыре различных окружения для поддержки рабочих процессов разработки, тестирования, предварительного выпуска и production:

### 1. Окружение разработки
- **Назначение**: Индивидуальные рабочие пространства разработчиков и тестирование ветвей функций
- **Инфраструктура**: Локальный Docker Compose или отдельные контейнеры сервисов
- **Конфигурация**: 
  - База данных: PostgreSQL с тестовыми данными
  - Кэш: Redis (локальный)
  - Хранилище: MinIO (S3-совместимый, локальный)
  - Внешние сервисы: Макетные или использующие бесплатные песочницы
- **Развертывание**: Ручное через `docker-compose up` или инструменты IDE, интегрированные с контейнерами
- **Доступ**: localhost:3000 (API), localhost:3001 (frontend)

### 2. Тестовое окружение
- **Назначение**: Автоматическое интеграционное и end-to-end тестирование
- **Инфраструктура**: Изолированное пространство имен Kubernetes или Docker swarm
- **Конфигурация**:
  - База данных: PostgreSQL с тестовым набором данных
  - Кэш: Redis (отдельный экземпляр)
  - Хранилище: Bucket MinIO на каждый запуск теста
  - Внешние сервисы: Тестовые аккаунты/песочницы
- **Развертывание**: Триггерируется pipeline CI при merge-запросах
- **Доступ**: Только внутренний доступ к кластеру

### 3. Окружение предварительного выпуска (Staging)
- **Назначение**: Предварительная валидация перед produkcцией, тестирование производительности и обзор заинтересованных сторон
- **Инфраструктура**: Кластер Kubernetes с конфигурацией parity с production
- **Конфигурация**:
  - База данных: PostgreSQL (readonly реплика схемы prod, анонимизированные данные)
  - Кэш: Redis Cluster
  - Хранилище: S3-совместимое (MinIO или AWS, отдельный bucket)
  - Внешние сервисы: Песочницы/тестовые аккаунты с лимитами, похожими на production
- **Развертывание**: Ручное продвижение из тестирования после валидации
- **Доступ**: staging.zoolink.internal (требуется VPN)

### 4. Окружение production
- **Назначение**: Платформа для живых пользователей
- **Инфраструктура**: Кластер Kubernetes (управляемый сервис или самодостаточный)
- **Конфигурация**:
  - База данных: Настройка Primary-Replica PostgreSQL с автоматическим failover
  - Кэш: Redis Cluster с persistence
  - Хранилище: Yandex Object Storage / VK / Selectel (S3-совместимое) или self-hosted MinIO — ADR-0008 (AWS S3 заблокирован в РФ)
  - Внешние сервисы: Ключи API production с мониторингом
  - CDN: Yandex Cloud CDN / VK / Selectel / Ngenix — ADR-0008 (CloudFront/Cloudflare заблокированы в РФ)
- **Развертывание**: Автоматически через GitOps или ручные шлюзы утверждения
- **Доступ**: Публичный доступ по адресу app.zoolink.com

## Технологии развертывания

### Контейнеризация
- Все сервисы упакованы в Docker-образы
- Многоэтапные сборки для минимизации размера образа
- Базовые образы: Node.js 18-alpine для NestJS сервисов
- Интеграция сканирования безопасности в pipeline сборки

### Оркестровка
- Основная: Kubernetes (EKS, GKE или самодостаточный)
- Альтернатива для dev/testing: Docker Compose или Docker Swarm
- Диаграммы Helm для согласованного развертывания сервисов
- Изоляция пространства имен на окружение

### Infrastructure as Code
- Terraform для подготовки облака (где применимо)
- Манифесты Kubernetes управляются через Helm
- Файлы значений, специфичные для окружения
- Версионно-контролируемые определения инфраструктуры

## Непрерывная интеграция/непрерывное развертывание (CI/CD)

### Этапы пайплайна
1. **Коммит кода**: Триггерируется при push в любую ветку
2. **Сборка**: 
   - Создание Docker-образов
   - Выполнение unit-тестов
   - Сканирование безопасности (SAST)
   - Проверка уязвимостей зависимостей
3. **Тестирование**:
   - Интеграционные тесты против эфемерных сервисов
   - Тестирование контрактов (Pact) для совместимости API
   - Бенчмарки производительности
4. **Развертывание в staging**: Автоматическое развертывание в тестирование/staging при коммите в main ветку
5. **Ручное утверждение**: Требуется для развертывания в production
6. **Развертывание в production**: Стратегия blue/green или rolling update
7. **Post-Deploy**: Smoke-тесты и валидация мониторинга

### Инструменты
- Платформа CI: GitHub Actions, GitLab CI, или Jenkins
- Реестр образов: GitHub Packages, Google Container Registry, или Docker Hub
- Безопасность: Trivy, Snyk, или аналогичные для сканирования уязвимостей
- Тестирование: Jest для unit-тестов, SuperTest для API, Cypress для E2E

## Паттерны развертывания

### Blue/Green Deployment
- Используется для мажорных релизов с потенциально ломкими изменениями
- Два идентичных production окружения (blue и green)
- Трафик переключается через балансировщик нагрузки после валидации
- Быстрый откат путем переключения трафика обратно

### Rolling Update
- Используется для минорных/патч релизов
- Постепенная замена подов по узлам
- Поддержание доступности во время развертывания
- Мониторинг на предмет деградации во время выката

### Canary Release
- Используется для флагов функций или рискованных изменений
- Малый процент трафика направляется на новую версию
- Постепенное увеличение на основе метрик и показателей ошибок
- Полный выкат или откат на основе критериев успеха

### Стратегия миграции базы данных
- Только обратно-совместимые изменения схемы
- Скрипты миграции выполняются как initContainers или отдельные задачи
- Флаги функций для использования таблиц/колонок во время перехода
- Автоматический откат при обнаружении сбоя
- Особенности PostgreSQL:
  - ADD COLUMN со значением DEFAULT NULL (избегает перезаписи таблицы)
  - CREATE INDEX CONCURRENTLY
  - Валидация ограничений NOT NULL в отдельном шаге

## Управление конфигурацией

### Переменные окружения
- Секреты хранятся в Kubernetes Secrets или cloud secret менеджере
- Несекреты в ConfigMaps
- Конфигурация, соответствующая принципу 12-факторного приложения
- Переопределения, специфичные для окружения, через значения Helm

### Общая конфигурация
- Общие настройки в базовых файлах конфигурации
- Наложения, специфичные для окружения
- Feature flags управляются через LaunchDarkly или аналог (будущее)
- Runtime перезагрузка где возможно (без перезапуска)

### Управление секретами
- Kubernetes Secrets (base64 закодированные) или их эквиваленты в облаке
- Автоматическая ротация паролей базы данных и API ключей
- Журнал аудита для доступа к секретам
- Ограниченные разрешения через RBAC

## Мониторинг и проверки работоспособности

### Zонды живучести (Liveness Probes)
- HTTP эндпоинт `/health/live`, возвращающий 200, когда процесс запущен
- Проверка на deadlocks или серьезную деградацию
- Настроен с соответствующими порогами сбоя

### Zонды готовности (Readiness Probes)
- HTTP эндпоинт `/health/ready`, возвращающий 200, когда сервис может принимать трафик
- Проверка зависимостей: подключение к базе данных, доступ к кэшу, доступ к хранилищу
- Предотвращает отправку трафика на нездоровые экземпляры

### Zонды запуска (Startup Probes)
- Для приложений с медленным запуском (миграции, прогрев кэша)
- Дает приложению время на инициализацию перед проверками живучести/готовности

## Процедуры отката

### Автоматический откат
- Триггерируется сбоями проверок работоспособности или превышением пороговых значений метрик
- Возврат к предыдущему известному хорошему развертыванию
- Отправка уведомления команде on-call

### Ручной откат
- Инициируется через панель развертывания или команды kubectl
- Откат базы данных требует отдельной процедуры (восстановление во времени)
- План коммуникации о воздействии на пользователей

### Откат миграции данных
- Изменения схемы спроектированы как forward-compatible
- Миграции данных написаны как обратимы где возможно
- Создание бэкапа перед рисковыми миграциями

## Рассмотрения производительности и масштабирования

### Запросы/лимиты ресурсов
- Запросы CPU и памяти основаны на нагрузочном тестировании
- Горизонтальное автосмасштабирование (HPA) настроено для веб-сервисов
- Вертикальное автосмасштабирование (VPA) рассматривается для batch/worker сервисов
- Квоты ресурсов на пространство имен для предотвращения шумных соседей

### Триггеры масштабирования
- Частота HTTP-запросов (основная)
- Глубина очереди (для рабочих сервисов)
- Уровень использования пула соединений БД
- Пользовательские метрики (специфичные для бизнес-логики)

### Стратегия кеширования
- Redis используется для:
  - Хранения сессий
  - Счетчиков ограничения скорости
  - Вычисленных результатов запросов (краткосрочный TTL)
  - Таблиц лидеров и счетчиков в реальном времени
  - Процедуры предупредительного заполнения кеша для предсказуемых паттернов трафика

## Рассмотрения безопасности при развертывании

### Безопасность образов
- Базовые образы сканируются на уязвимости
- Выполнение от не-root пользователя в контейнерах
- Только для чтения корневая файловая система где возможно
- Отброшенные возможности и ограниченные системные вызовы

### Безопасность сети
- Сетевые политики, ограничивающие межсервисное взаимодействие
- Сервисная сетка (Istio/Linkerd) рассматривается для фазы 2
- Ingress контроллеры с возможностями WAF
- TLS окончание на ingress или балансировщике нагрузки

### Защита секретов
- Секреты монтируются как тома или внедряются через переменные окружения
- Нет секретов в переменных окружения, видимых через ps
- Регулярная ротация долгосрочных учетных данных

## Восстановление после катастроф

### Стратегия резервного копирования
- База данных: Ежедневные логические бэкапы, часовое архивирование WAL
- Хранилище объектов: Включено версионирование, репликация межрегиональная
- Конфигурация: Git-репозиторий как источник истины
- Процедуры тестового восстановления выполняются ежеквартально

### Цель времени восстановления (RTO)
- Критические сервисы: < 30 минут
- Полная платформа: < 2 часов
- Зависит от частоты резервного копирования и процедур восстановления

### Цель точки восстановления (RPO)
- База данных: < 1 час (восстановление во времени возможно)
- Хранилище объектов: < 15 минут (задержка репликации)
- Содержимое, созданное пользователями: Зеркала с отсроченным удалением

## Документация по развертыванию

### Руководства по выполнению (runbooks)
- Общие процедуры развертывания задокументированы
- Процедуры реагирования на инциденты при сбоях развертывания
- Руководства по устранению неполадок, специфичные для окружения

### Управление изменениями
- Все развертывания привязаны к tickets или merge запросам
- Пост-развертывающий обзор для значительных изменений
- Журнал аудита того, кто что развернул и когда

## Связанные решения

- **ADR-0001**: Выбор стека технологий (влияет на выбор контейнеризации и оркестровки)
- **Архитектурные решения**: Архитектура модульного монолита, готового к микросервисам, влияет на границы развертывания
- **Нефункциональные требования**: Производительность и цели доступности информируют стратегии масштабирования
