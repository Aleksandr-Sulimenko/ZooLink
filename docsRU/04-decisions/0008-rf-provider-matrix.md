# ADR-0008: Матрица сторонних провайдеров под РФ

**Status**: Accepted
**Date**: 2026-06-17

## Контекст и постановка проблемы

[ADR-0001](0001-tech-stack.md) и архитектурные диаграммы называют ряд сторонних провайдеров, которые
**недоступны или нестабильны для продукта на рынке РФ** из-за санкций и платёжных ограничений:
Stripe, PayPal, Twilio, SendGrid, Datadog/New Relic и AWS/CloudFront/Cloudflare. Аудит
(`BACKEND_TECH_AUDIT.md`, Суб-агент 3) классифицировал это как **блокер релиза**: код, написанный под эти
дефолты, не заработает в продакшене.

ADR-0001 уже предписывает **слои абстракции** для SMS, OAuth, карт и хранилища, поэтому исправление — это смена
**дефолтов**, а не архитектуры. Этот ADR фиксирует канонический набор провайдеров под РФ.

## Факторы решения

- **Доступность в РФ**: провайдер должен принимать клиентов и платежи из РФ.
- **Комплаенс**: платежи должны поддерживать **54-ФЗ** (фискальные чеки) и PCI DSS на стороне провайдера; ПДн по **ФЗ-152**.
- **Абстракция**: каждый внешний провайдер за интерфейсом; смена вендора не должна затрагивать доменный код.
- **Стоимость / self-hosting**: предпочитать self-hostable там, где это убирает платёжную зависимость (мониторинг, хранилище).

## Рассмотренные варианты

По каждой возможности сравнивались реалистичные варианты под РФ (эквайринг: ЮKassa vs Т-Касса vs CloudPayments;
SMS: SMS.RU vs SMSC vs MTS Exolve; и т.д.). Решение фиксирует **дефолт** плюс принятые **альтернативы**; поскольку
каждая возможность за интерфейсом, альтернативы остаются drop-in.

## Решение

Принять следующую **каноническую матрицу провайдеров**. MVP реализует интерфейс для каждой возможности;
конкретные вендоры с пометкой *Фаза 2+* отложены, но их интерфейс определяется уже сейчас.

| Возможность | ❌ Нельзя использовать (заблокировано в РФ) | ✅ Дефолт | Принятые альтернативы | Фаза |
|---|---|---|---|---|
| **Платежи / эквайринг** | Stripe, PayPal | **ЮKassa + СБП** | Т-Касса (Тинькофф), CloudPayments, Robokassa | Фаза 2+ (гейтится `feature_toggles.payments`) |
| **SMS** | Twilio | **SMS.RU** | SMSC.RU, MTS Exolve, SMS Aero | MVP |
| **Email (транзакционный)** | SendGrid | **Unisender** | Mailopost, SMTP-релей РФ-хостера | MVP |
| **Карты / геокодинг** | — | **Yandex.Maps** | 2GIS | MVP |
| **OAuth** | — | **Google, Apple, Telegram, VK** | — | MVP |
| **Объектное хранилище** | AWS S3 | **Yandex Object Storage** | VK Cloud, Selectel, self-hosted MinIO (он же dev) | MVP |
| **CDN** | CloudFront, Cloudflare | **Yandex Cloud CDN** | VK Cloud CDN, Selectel CDN, Ngenix | MVP |
| **Мониторинг / APM** | Datadog, New Relic | **Prometheus + Grafana** | VictoriaMetrics, Yandex Monitoring | MVP |
| **Трекинг ошибок** | — | **Sentry (self-hosted)** | GlitchTip | MVP |

## Последствия

### Положительные
- Продакшен работает в РФ с первого дня; нет переписывания при выходе из фазы валидации.
- Self-hosted мониторинг/хранилище/трекинг ошибок убирают зависимость от иностранного биллинга.
- Абстракции делают выбор вендора обратимым.

### Отрицательные
- У части РФ-провайдеров более тонкие SDK/доки, чем у глобальных грандов → чуть больше работы на адаптер.
- Платежи добавляют обязательства по **54-ФЗ** (онлайн-чеки) — обрабатывается в Payment-домене.

### Нейтральные
- Yandex.Maps уже был выбранным провайдером карт в ADR-0001 — без изменений.

## Замечания по реализации

- Каждая возможность за port/интерфейсом: `SmsProvider`, `EmailProvider`, `PaymentProvider`, `MapsProvider`,
  `ObjectStorage`, `Metrics`. Адаптеры на каждого конкретного вендора; вендор выбирается через config/env.
- **Платежи** остаются за `feature_toggles.payments` (см. схему) и `specs/14-payment-domain.md`; интегрировать
  выпуск чеков по 54-ФЗ через фискализацию провайдера (ЮKassa это поддерживает).
- **ПДн / ФЗ-152**: держать персональные данные внутри инфраструктуры РФ; не экспортировать ПДн в заблокированные SaaS.
- Обновить `specs/13-notification-domain.md` и `specs/14-payment-domain.md` со ссылкой на эту матрицу.
