# ADR-0004: Животное как агрегатный корень

**Статус**: Принято  
**Дата**: 2026-05-30  

## Контекст и постановка проблемы

В платформе ZooLink животные являются центральными для всех бизнес-операций. Пользователи создают профили животных, затем создают объявления, связанные с этими животными (для продажи, вязки, выставок и т.д.). Одно и то же животное может иметь несколько объявлений на протяжении своей жизни (разные продажи, множественные события вязки, участие в выставках).

Нам нужно было решить, как моделировать взаимосвязь между животными и объявлениями в нашей доменной модели. Конкретно, нам нужно было определить, следует ли:
- Животное быть сущностью внутри агрегата объявления
- Объявление быть сущностью внутри агрегата животного
- Животное и Объявление быть отдельными агрегатами с отношениями
- Животное быть объектом-значением внутри Объявления

Первоначальные соображения включали моделирование объявлений как первичного агрегата с вложенными деталями животного, или traktование животного и объявления как отдельных сущностей с простой связью внешнего ключа. Однако мы признали, что животные имеют жизненный цикл, независимый от любого конкретного объявления, и что объявления являются переходными событиями в жизни животного.

## Драйверы решения

1. **Независимость продолжительности жизни**: Существует до своего первого объявления и может существовать после последнего объявления
2. **Множественные объявления**: Одно животное может иметь много объявлений на протяжении своей жизни (продажи, вязки, выставки и т.д.)
3. **Неизменяемые основные атрибуты**: Вид, порода, пол, дата рождения фиксируются при создании животного
4. **Изменяемые атрибуты**: Кличка, цвет, записи о здоровье, записи о репродукции могут изменяться
5. **Передача собственности**: Животные могут менять владельцев, сохраняя свою идентичность
6. **Целостность данных**: Необходимо предотвратить противоречивые объявления о одном и том же животном
7. **Шаблоны запросов**: Частая необходимость найти все объявления для данного животного
8. **Доменная логика**: Бизнес-правила о пригодности животного для определённых типов объявлений (например, стерилизованные домашние животные для вязки)

## Рассмотренные варианты

### Вариант 1: Объявление как агрегатный корень со встроенными деталями животного
Каждое объявление содержит всю информацию о животном, встроенную внутри него.

Плюсы:
- Простая модель данных для представлений, сфокусированных на объявлении
- Атомарное создание объявления (без транзакций между агрегатами)
- Легко сделать снимок состояния животного на момент объявления

Минусы:
- Дублирование данных, когда животное имеет несколько объявлений
- Риск несогласованности при изменении атрибутов животного между объявлениями
- Трудно поддерживать полную историю животного
- Сложная передача собственности (необходимо обновить все исторические объявления)
- Отрицает независимое существование и идентичность животного

### Вариант 2: Отдельные агрегаты со связью внешнего ключа
Животное и Объявление являются отдельными агрегатами, связанными через внешний ключ animal_id.

Плюсы:
- Чёткое разделение ответственности
- Животное поддерживаетнезависимую идентичность и жизненный цикл
- Нет дублирования основных атрибутов животного
- Легко выполнить запрос всех объявлений для животного
- Прямолинейная передача собственности (обновление owner_id животного)

Минусы:
- Требуются межагрегатные транзакции для создания объявления (валидация существования животного и его принадлежности пользователю)
- Потенциал для сиротских объявлений при удалении животного (смягчается мягким удалением)
- Немного более сложная initial-создание объявления

### Вариант 3: Животное как объект-значение внутри Объявления
Детали животного рассматриваются как неизменяемые значения, копируемые в каждое объявление.

Плюсы:
- Наиболее простая реализация изначально
- Объявление самодостаточно для отображения

Минусы:
- Те же недостатки, что и Вариант 1, но хуже для изменяемости
- Полностью отрицает идентичность животного между объявлениями
- Невозможно отслеживать историю жизни животного
- Не подходит для реальности домена, где животные сохраняются после отдельных транзакций

### Вариант 4: Животное как агрегатный корень с Объявлениями как дочерними сущностями (Выбранный)
Животное является агрегатным корнем, а объявления являются сущностями внутри агрегата животного, ссылаясь обратно к нему.

Плюсы:
- Соответствует реальности домена: животные существуют независимо и имеют много событий объявлений
- Централизует идентичность и управление жизненным циклом животного
- Предотвращает несогласованные данные о животном между объявлениями
- Осуществляет правила собственности на уровне агрегата животного
- Естественное соответствие для запросов истории животного и текущего состояния
- Поддерживает бизнес-правила вроде «стерилизованные животные не могут быть размещены для вязки»
- Чёткая транзакционная граница для операций, связанных с животными

Минусы:
- Немного сложнее реализовать запросы, ориентированные на объявление (необходимо проходить через животное)
- Требует тщательного проектирования, чтобы избежать загрузки всей истории животного при необходимости только данных объявления
- Потенциал для больших агрегатов, если у животного много объявлений (смягчается пагинацией и шаблонами доступа)

## Решение

Мы будем моделировать **Животное как агрегатный корень** с Объявлением как сущностью внутри агрегата животного. Это означает:

1. **Агрегатный корень Животного**:
   - Глобальная идентичность (animal_id), которая сохраняется во всех объявлениях
   - Содержит неизменяемые атрибуты: вид, порода, пол, дата рождения (приблизительная)
   - Содержит изменяемые атрибуты: кличка, цвет/окрас, чип/тату, базовые записи о здоровье/репродукции (хранятся как JSONB)
   - Информация о собственности (owner_id, связанная с пользователем или организацией)
   - Флаг мягкого удаления для деактивированных животных
   - Правила валидации для комбинаций атрибутов (например, видоспецифичные поля)

2. **Сущность Объявления** (внутри агрегата Животного):
   - Локальная идентичность в контексте животного (listing_id)
   - Ссылается на родительское животное (animal_id)
   - Содержит данные, специфичные для объявления: заголовок, описание, цена, расположение, фотографии, статус
   - Содержит поля, специфичные для типа объявления (условия продажи, условия вязки, детали выставки и т.д.)
   - Статус модерации и аудиторский след
   - Метки времени для создания, модерации, публикации

3. **Инварианты и бизнес-правила**:
   - Неизменяемые атрибуты животного (вид, порода, пол, дата рождения) не могут изменяться после создания
   - Изменения собственности должны проходить через агрегат животного (обновление owner_id)
   - Создание объявления валидирует, что животное существует и принадлежит текущему пользователю/организации
   - Некоторые типы объявлений требуют, чтобы животное соответствовало критериям (например,нестерилизованное для вязки, стерилизованное для акций по привлечению внимания к приютам)
   - Мягкое удаление животного скрывает все связанные объявления из общего доступа
   - Жёсткое удаление запрещено; используйте мягкое удаление для целостности данных

4. **Поддерживаемые шаблоны запросов**:
   - Получить профиль животного с последними объявлениями
   - Получить все объявления для животного (отфильтрованные по статусу/типу/дате)
   - Получить объявления по владельцу (через собственность животного)
   - Поиск объявлений между животными ( денормализованный поисковый индекс для эффективности)
   - Получить историю объявлений животного для аналитики

## Последствия

### Положительные
- Точно отражает реальность домена, где животные имеют независимое существование
- Предотвращает несогласованность данных для основных атрибутов животного
- Упрощает передачу собственности и управление жизненным циклом животного
- Осуществляет бизнес-правила на соответствующем уровне агрегата
- Поддерживает богатый запрос history животного и шаблонов объявлений
- Чёткие транзакционные границы для операций, связанных с животными

### Отрицательные
- Запросы, ориентированные на объявление, требуют соединения через агрегат животного
- Необходимо спроектировать модели чтения/поисковые индексы для эффективного поиска объявлений
- Потенциал для больших агрегатов животного (смягчается CQRS и шаблонами доступа)
- Начальная кривая обучения для разработчиков, привыкших к подходам, ориентированным на объявление

### Нейтральные
- Не препятствует eventual consistency схемам для операций чтения, преобладающих над записью
- Основная инфраструктура (база данных, кэш) не изменилась
- API контракты остаются ориентированными на объявление, внутренне уважая границы агрегата

## Примечания по реализации

1. **Модель данных**:
   - Таблица `animals` со столбцами для идентичности, неизменяемых/изменяемых атрибутов, собственности, мягкого удаления
   - Таблица `listings` с внешним ключом на животных, данными, специфичными для объявления, статусом, отметками времени
   - Индексы на `animals.owner_id` для запросов на основе собственности
   - Индексы на `listings.animal_id` для запросов объявлений конкретного животного
   - Поисковые индексы (PostgreSQL GIN/GiST или Elasticsearch) для поиска объявлений, денормализованных из данных животного + объявления

2. **Проектирование API**:
   - Конечные точки, ориентированные на животное: `/animals/{id}`, `/animals/{id}/listings`
   - Конечные точки, ориентированные на объявление: `/listings` (поиск/фильтрация), `/listings/{id}`
   - Внутренние сервисы обеспечивают соблюдение границ агрегата (например, ListingService проверяет принадлежность животного через AnimalService)
   - Публикуются события для событий жизненного цикла животного (создано, изменена принадлежность, деактивировано)

3. **Реализация доменной логики**:
   - Агрегатный корень животного валидирует ограничения неизменяемых полей
   - Агрегат животного валидирует переходы собственности
   - Фабрика/сервис объявления валидирует соответствие животного типу объявления
   - Рабочий процесс модерации работает над объявлениями, но опирается на данные животного для валидации

4. **Соображения по производительности**:
   - Индексы чтения для запросов объявлений ( денормализованные данные животного + объявления)
   - Кэширование часто используемых профилей животных
   - Пагинация для истории объявлений животного
   - Пакетные операции для передач собственности, затрагивающих множество объявлений

## Связанные решения

- **ADR-0002**: Жёсткое разделение между рынком домашних животных и рынком скота (агрегат животного служит обоим доменам)
- **ADR-0003**: Премодерация рабочего процесса (объявления проходят через состояния модерации)
- **ADR-0005**: Нет чата в MVP (раскрытие контактов происходит после модерации, привязанное к статусу объявления)
- **ADR-0001**: Выбор технологического стека (NestJS поддерживает принципы DDD с модулями и сервисами)
