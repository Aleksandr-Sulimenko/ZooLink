# ADR-0003: Премодерация рабочего процесса для объявлений

**Статус**: Принято  
**Дата**: 2026-05-29  

## Контекст и постановка проблемы

Платформа ZooLink требует, чтобы все объявления проходили модерацию перед тем, как стать общедоступными. Такой премодерационный подход необходим для поддержания качества платформы, предотвращения мошенничества, обеспечения соблюдения нормативов и защиты пользователей от неподобающего контента. Однако нам необходимо определить чёткий рабочий процесс, который балансирует тщательный осмотр с приемлемым временем выхода на рынок для легитимных объявлений.

Изначально мы рассматривали постмодерацию (объявления появляются сразу, затем просматриваются) или отсутствие модерации (полагаясь исключительно на пользовательские сообщения). Однако анализ показал, что подобные подходы comprometят целостность платформы и доверие пользователей, особенно с учётом характера сделок с животными, которые связаны с важной финансовой и эмоциональной ценностью.

## Драйверы решения

1. **Целостность платформы**: Предотвращение мошенничества, мошеннических объявлений и неподобающего контента
2. **Безопасность пользователей**: Защита покупателей и продавцов от злонамеренных действующих лиц
3. **Соблюдение нормативов**: Особенно важно для отслеживания перемещения скота и документации
4. **Стандарты качества**: Обеспечение того, чтобы объявления соответствовали минимальным требованиям полноты и точности
5. **Эффективность модерации**: Чёткий рабочий процесс снижает когнитивную нагрузку и повышает согласованность
6. **Ожидания пользователей treaties**: Пользователи должны понимать, почему объявления не сразу видны
7. **Время выхода на рынок**: Легитимные объявления должны публиковаться в разумные сроки

## Рассмотренные варианты

### Вариант 1: Постмодерация (Объявления появляются сразу, затем просматриваются)
Объявления становятся видимыми немедленно после создания, а модерация происходит асинхронно.

Плюсы:
- Мгновенное удовлетворение для продавцов
- Простая техническая реализация
- Отсутствие задержки в видимости объявлений

Минусы:
- Мошеннические/обманные объявления видны пользователям до удаления
- Негативный пользовательский опыт, когда легитимные пользователи сталкиваются с плохим контентом
- Трудно обеспечивать стандарты качества заранее
- Потенциальная юридическая ответственность за временное размещение незаконного контента
- Требуется реактивное удаление вместо профилактических мер

### Вариант 2: Отсутствие формальной модерации (Полагаться на отчёты пользователей)
Объявления появляются немедленно, модерация происходит только при сообщении пользователей о проблемах.

Плюсы:
- Минимальная техническая нагрузка
- Самая быстрая возможная публикация объявлений

Минусы:
- Высокая вероятность того, что вредоносный контент будет оставаться до сообщения
- Положение на пользователей бремя полиции платформы
- Плохой пользовательский опыт для покупателей, сталкивающихся с проблемными объявлениями
- Практически невозможно поддерживать стандарты качества
- Значительный риск использования платформы для незаконной деятельности

### Вариант 3: Премодерация с чётким рабочим процессом (Выбранный)
Все объявления должны быть одобрены модератором до их публичной видимости, с определёнными ролями, обязанностями и временными рамками.

Плюсы:
- Предотвращает появление вредоносного контента в публичном доступе
- Поддерживает высокие стандарты качества во всех объявлениях
- Даёт чёткие ожидания продавцам (почему объявление не сразу видимо)
- Позволяет проактивно соблюдать нормативы (особенно для скота)
- Формирует доверие пользователей к безопасности и целостности платформы
- Позволяет specialised обучение модераторов по домену (домашние животные vs скот)

Минусы:
- Задержка между созданием объявления и его публикацией
- Требуются ресурсы модераторов для поддержания объёма
- Более сложная техническая реализация (очереди, отслеживание статуса)

## Решение

Мы будем реализовывать **премодерационный рабочий процесс** со следующими характеристиками:

1. **Состояния жизненного цикла объявления**:
   - `DRAFT`: Первоначальное создание, редактируемое владельцем
   - `PENDING_MODERATION`: Отправлено на рецензирование, не редактируемое
   - `PUBLISHED`: Одобрено, видно в поиске и списках
   - `REJECTED`: Не одобрено, возвращено в `DRAFT` с обратной связью модератора
   - `ARCHIVED`: Вручную скрыто владельцем или администратором
   - `COMPLETED`: Сделка завершена (опциональное действие пользователя)

2. **Процесс модерации**:
   - Модераторы имеют доступ к специальной очереди модерации
   - Очередь показывает объявления, сгруппированные по домену (домашние животные/скот) с превью-карточками
   - Для каждого объявления модераторы проверяют:
     - Соответствие фото объявлению (животное соответствует заявленной породе/виду)
     - Полнота обязательных полей
     - Соответствие правилам платформы (нет спама, незаконного контента, ложных заявлений)
     - Для скота: регуляторные флаги, указывающие на необходимость транспортной документации
   - Действия модератора:
     - **Одобрить**: Статус объявления → `PUBLISHED`, становится доступным для поиска
     - **Отклонить**: Статус объявления → `DRAFT` с комментариями модератора; владелец может отредактировать и повторно отправить

3. **Целевые временные рамки**:
   - Объявления о домашних животных: Целевое время модерации < 4 часов в рабочее время (9:00–21:00)
   - Объявления о скоте: Целевое время модерации < 6 часов в рабочее время (более сложная валидация)
   - Эти цели балансируют тщательность с разумным временем ожидания

4. **Роли модераторов и экспертиза**:
   - Модераторы домашних животных: Обучены характеристикам компаньонных животных, типичным мошенничествам при продаже животных
   - Модераторы скота: Знакомы с фермерскими животными, продуктивными качествами, правилами транспортировки
   - Кросс-обучение доступно, но специализация поощряется для эффективности

5. **Техническая реализация**:
   - API объявлений обеспечивает, чтобы только статус `PUBLISHED` появлялся в конечных точках поиска/списков
   - API очереди модерации предоставляет постраничный доступ к объявлениям со статусом `PENDING_MODERATION`
   - Переходы статуса атомарны и аудитируются
   - Действия модераторов логируются для подотчётности и обучения
   - Владельцы объявлений получают уведомления при изменении статуса

## Последствия

### Положительные
- Убирает вредоносный контент с публичного доступа
- Чёткий, предсказуемый процесс для продавцов
- Позволяет доменно-специфичную экспертизу модераторов
- Репутация платформы безопасности и качества
- Проактивное соблюдение нормативов (особенно важно для скота)
- Основа для будущей модерации, поддерживаемой ML

### Отрицательные
- Задержка в видимости объявлений (смягчается чётким общением и целями по времени)
- Требуются постоянные инвестиции в команду модераторов
- Техническая сложность управления состоянием и очередями

### Нейтральные
- Не влияет на рабочие процессы создания или редактирования объявлений в состоянии черновика
- Основная модель данных объявлений unchanged (добавляется поле статуса)
- Поиск и API объявлений unchanged за исключением фильтрации по статусу

## Примечания по реализации

1. **API контракты**:
   - API объявлений включает поле `status` с допустимыми значениями
   - Конечные точки модерации для доступа к очереди и обработки решений
   - Механизм WebSocket или опроса для реального времени обновления статуса владельцам

2. **Компоненты UI**:
   - Интерфейс очереди модерации с фильтрацией и групповыми действиями
   - Вид детали объявления для модераторов, показывающий всю соответствующую информацию
   - Система уведомлений владельцев об изменении статуса
   - Чёткое указание причины отклонения с выполнимыми указаниями

3. **Модель данных**:
   - Таблица `listings` содержит поле `status` типа ENUM
   - Таблица `moderation_actions` журнала всех решений модератора с отметками времени и идентификатором модератора
   - Индексы по `status` и `created_at` для эффективной обработки очереди

4. **Бизнес-правила**:
   - Только владельцы объявлений могут переходить из состояния `DRAFT` в `PENDING_MODERATION`
   - Только модераторы могут переходить из состояния `PENDING_MODERATION` в `PUBLISHED` или `REJECTED`
   - Владельцы объявлений могут редактировать и повторно отправлять отклонённые объявления
   - После `PUBLISHED` определённые поля становятся неизменяемыми (см. ADR-0004)

## Связанные решения

- **ADR-0002**: Жёсткое разделение между рынком домашних животных и рынком скота (отдельные очереди модерации)
- **ADR-0004**: Животное как агрегат (объявления привязаны к неизменной сущности животного)
- **ADR-0005**: Нет чата в MVP (раскрытие контактов происходит после модерации)
