# ADR-0007: Стратегия ORM — Prisma как основной + типизированный raw-SQL escape hatch

**Status**: Accepted
**Date**: 2026-06-17

## Контекст и постановка проблемы

[ADR-0001](0001-tech-stack.md) выбрал **Prisma** в качестве ORM. Аудит бэкенд-стека
(`BACKEND_TECH_AUDIT.md`, Суб-агент 2 — Senior DBA) подтвердил, что Prisma отлична для типичного CRUD и
типобезопасности, но отметил две структурные слабости относительно схемы ZooLink:

1. **Геопространственные типы.** Prisma не моделирует нативно `GEOGRAPHY(POINT,4326)` (PostGIS). Даже MVP-поиск
   по `lat`/`lng` (Haversine) выигрывает от написанного вручную SQL для радиусных запросов.
2. **Сложный JSONB.** Схема JSONB-тяжёлая (локализация, `health_records`, `reproductive_data`, `metadata`,
   `filters`). Глубокая фильтрация/агрегация по JSONB неудобна, а иногда невозможна в query-API Prisma.

Нужно решить это **без** переоткрытия Accepted ADR-0001 (без полной замены ORM) и без того, чтобы разработчики
импровизировали по каждой фиче.

## Факторы решения

- **Преемственность**: ADR-0001 в статусе Accepted; минимизировать ломку и переработку.
- **Мощность запросов**: гео + сложный JSONB должны быть выразимы и производительны.
- **Типобезопасность**: сохранить гарантии на этапе компиляции, где возможно.
- **Защита от SQL-инъекций**: любой raw SQL обязан быть параметризован (NFR безопасности).
- **Кадры**: стек должен оставаться изучаемым для кадрового рынка РФ.

## Рассмотренные варианты

### Вариант 1: Prisma + Kysely / параметризованный raw SQL (escape hatch)
Оставить Prisma основным ORM для схемы, миграций и большей части CRUD. Использовать **Kysely** (типизированный
SQL-конструктор) или Prisma `$queryRaw` (параметризованный) для гео и сложного JSONB.

Плюсы:
- Ноль переписывания решения ADR-0001; миграции Prisma остаются источником DDL рядом с `database_schema.sql`.
- Гео/JSONB получают полную мощь SQL; Kysely сохраняет типобезопасность.
- Чёткая документированная граница «когда спускаться в SQL».

Минусы:
- Два механизма запросов в кодовой базе (смягчается явным правилом использования).
- Одна дополнительная зависимость (Kysely).

### Вариант 2: Полный переход на TypeORM
Плюсы:
- Единый инструмент; QueryBuilder хорошо работает с raw/гео; шире знаком в РФ.
Минусы:
- Противоречит Accepted ADR-0001 (требует superseded); переписывание миграций; у TypeORM свои острые углы.

### Вариант 3: Только чистый Prisma
Плюсы:
- Простейший набор зависимостей.
Минусы:
- Гео и сложный JSONB продавливаются через ad-hoc `$queryRaw` без общего типизированного паттерна → несогласованность
  и латентный риск SQL-инъекций при неаккуратной реализации.

## Решение

Принять **Вариант 1**. **Prisma остаётся основным ORM** (схема, миграции, CRUD). Для гео-запросов
(MVP Haversine по `lat`/`lng`; в будущем PostGIS) и сложного JSONB использовать **Kysely** (предпочтительно,
типизированно) или **параметризованный** Prisma `$queryRaw`. Интерполяция пользовательского ввода в строку SQL
запрещена.

## Последствия

### Положительные
- ADR-0001 остаётся в силе; нет дорогой миграции ORM.
- Гео и JSONB — первоклассные через SQL, с сохранением типобезопасности через Kysely.
- Единое документированное правило исключает импровизацию каждого разработчика.

### Отрицательные
- Разработчики должны понимать, когда Prisma, а когда Kysely (документировано ниже).
- Две библиотеки касаются БД; риск дрейфа схемы между ними (смягчение: `database_schema.sql` — источник истины;
  Prisma-схема и типы Kysely генерируются/проверяются относительно него).

### Нейтральные
- `database_schema.sql` остаётся канонической схемой; `schema.prisma` её зеркалит.

## Замечания по реализации

- **Правило использования:** Prisma для CRUD сущностей и простых связей. Kysely/`$queryRaw` для: радиусного/гео
  поиска, рекурсивной родословной (`WITH RECURSIVE` по `mother_id`/`father_id`), containment/агрегации по JSONB и
  любого запроса, который Prisma не выражает эффективно.
- **Гео:** моделировать `location_point` как `Unsupported("geography")` в Prisma; никогда не запрашивать через Prisma напрямую.
- **Безопасность:** весь raw SQL параметризован; добавить ESLint/CI-проверку против конкатенации строк в запросах.
- **Харднинг инъекций Kysely (round-6, нормативно).** Kysely `<= 0.28.16` содержал HIGH-уязвимости SQL-инъекций
  (traversal по JSON-path ключам/`.at()`; инъекция литерала через `sql.lit`; усиление инъекции при подавлении ошибок
  компиляции или использовании `Kysely<any>` — GHSA-pv5w-4p9q-p3v2, GHSA-wmrf-hv6w-mr66, GHSA-8cpq-38p9-67gx).
  Поэтому: (a) **пин Kysely `>= 0.29`** (пропатченная линия); (b) экземпляр Kysely ДОЛЖЕН быть типизирован конкретной
  схемой `DB` — **никогда `Kysely<any>`** (заменить заглушку-интерфейс `DB` на codegen-типы до выхода гео/JSONB
  запросов); (c) **никогда не передавать недоверенный ввод** в `sql.lit()` или в JSON-path-леги (`.key()`/`.at()`) —
  биндить как параметр; (d) не подавлять ошибки компиляции Kysely. Это штатная митигация, удерживающая escape-hatch
  ADR-0007 безопасным по мере появления гео/JSONB запросов.
- **Пул соединений:** встроенный пул Prisma в MVP; PgBouncer (transaction-режим) перед PostgreSQL при росте числа
  соединений (см. `specs/performance_specification.md`).
