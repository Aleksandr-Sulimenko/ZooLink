# ADR-0009: Архитектура MVP — модульный монолит; микросервисы/K8s — Фаза 2+

**Status**: Accepted
**Date**: 2026-06-17

## Контекст и постановка проблемы

Аудит (`BACKEND_TECH_AUDIT.md`, главная находка) обнаружил, что документация описывает **два противоречащих
друг другу бэкенда**:

- [ADR-0001](0001-tech-stack.md) (канон): **модульный монолит** (NestJS), Docker Compose для MVP, Kubernetes
  «рассматривается для Фазы 2+», без брокера сообщений.
- `component-diagram.md` и `deployment-diagram.md`: **микросервисы**, gRPC, отдельная шина событий, SSR Web Gateway,
  read-реплики, service mesh и полноценный мульти-зональный кластер **Kubernetes**.

Для задокументированной нагрузки MVP (`specs/performance_specification.md`: 500 одновременных пользователей,
50 RPS среднее, 200 RPS пик) тяжеловесная архитектура — это **overengineering**, который будет стоить месяцев и
постоянной эксплуатационной нагрузки. Без явной границы команда может построить диаграммы вместо ADR.

## Факторы решения

- **Реальность нагрузки**: 50–200 RPS комфортно держит монолит на 2–3 репликах за прокси.
- **Время до валидации**: MVP нужен для проверки гипотез, а не для преждевременного масштабирования.
- **Эксплуатируемость**: Docker Compose обслуживается малой командой; K8s + service mesh — нет, для MVP.
- **Без преждевременной распределённости**: распределённые системы добавляют режимы отказа (gRPC, eventual
  consistency) без текущей выгоды.

## Рассмотренные варианты

### Вариант 1: Модульный монолит для MVP; микросервисы/K8s как явное целевое состояние (Фаза 2+)
Плюсы:
- Соответствует ADR-0001 и реальной нагрузке; быстрее всего выпустить; дешевле всего эксплуатировать.
- Границы модулей (bounded contexts DDD) сохраняют чистый путь извлечения позже.
Минусы:
- Диаграммы нужно переразметить как Target State (сделано в этой ремедиации).

### Вариант 2: Строить микросервисы/K8s сейчас (как подразумевают диаграммы)
Плюсы:
- «Future-proof» топология с первого дня.
Минусы:
- Огромные накладные для 50 RPS; замедляет валидацию; высокая стоимость эксплуатации; противоречит ADR-0001.

## Решение

**MVP (Фаза 1) — это модульный монолит.** Принять следующий обязательный объём:

**MVP (Фаза 1) — ВКЛЮЧЕНО:**
- Единый деплой NestJS с внутренними модулями = bounded contexts (`identity`, `animal`, `pet-marketplace`,
  `livestock-marketplace`, `admin`, `moderation`, …). Межмодульные вызовы = in-process DI.
- **Docker Compose** на 1–2 VM: `api` (1–N реплик), `postgres`, `redis`, `minio`, фоновый `worker`.
- **Reverse proxy** (Nginx/Caddy/Traefik) для TLS + статики SPA + origin CDN.
- **События через таблицу `outbox_events`**, вычитываемую фоновым воркером (polling / `pg_notify`) — **без
  отдельного брокера**.
- Stateless API (JWT), чтобы реплики масштабировались горизонтально; Redis для общего состояния и rate limiting.
- **Изоляция через Docker-сети**: публичен только reverse proxy; БД/Redis/MinIO не выставлены наружу.

**Фаза 2+ — ОТЛОЖЕНО (НЕ строить в MVP):**
- Декомпозиция на микросервисы, **gRPC** между сервисами, **service mesh** (Istio/Linkerd).
- Отдельная **шина событий** / брокер (RabbitMQ/Kafka) — внедрять, только когда вычитки outbox станет недостаточно.
- **Kubernetes**, HPA/VPA/Cluster Autoscaler, мульти-зональный DR.
- Read-реплики, SSR Web Gateway, Elasticsearch, PostGIS.

## Последствия

### Положительные
- Единая согласованная цель для команды; нет двусмысленности «какая диаграмма настоящая?».
- Самый дешёвый/быстрый путь к продакшен-MVP, удовлетворяющему perf-NFR.
- Чистые швы извлечения сохранены через DDD-модули.

### Отрицательные
- Будущий шаг миграции на микросервисы/K8s, когда этого потребует масштаб (запланированный, а не случайный).

### Нейтральные
- Целевые диаграммы остаются ценными как чертёж Фазы 2+; теперь они размечены соответствующе.

## Замечания по реализации

- `component-diagram.md` и `deployment-diagram.md` несут баннер **MVP vs Target State** со ссылкой сюда.
- Фоновый `worker` владеет: вычиткой outbox, плановыми задачами (cron), асинхронными уведомлениями, очисткой
  незавершённых загрузок.
- Определить контракт **outbox relay** в data-model (таблица существует: `outbox_events`).
- Пересмотреть этот ADR, когда устойчивый RPS, размер команды или потребность в изоляции доменов перейдут пороги из
  `specs/performance_specification.md` (Фаза 2 «Рост»).
