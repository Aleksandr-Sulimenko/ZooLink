# ADR 0001: Выбор технологического стека

## Статус
Принято

## Контекст
Необходимо выбрать технологический стек для MVP ZooLink, который поддерживает:
- Быструю разработку и итерации для проверки бизнес-гипотез
- Четкое разделение отвественности, согласованное с ограниченными контекстами предметно-ориентированного проектирования (DDD)
- Масштабируемость для будущего роста при сохранении простоты для MVP
- Знакомство команды и доступность талантов
- Соответствие нефункциональным требованиям (безопасность, производительность, поддерживаемость)

## Решение
Мы будем использовать следующий технологический стек для MVP:

### Бэкенд
- **Язык**: Node.js (TypeScript)
- **Фреймворк**: NestJS
- **ORM/ODM**: Prisma ORM
- **Стиль API**: RESTful с спецификацией OpenAPI 3.0
- **Аутентификация**: JWT (JSON Web Tokens) с токенами обновления
- **Возможности реального времени**: Подготовка к будущей интеграции WebSocket (не используется в MVP)

### Фронтенд
- **Тип**: Single Page Application (SPA) с возможностями PWA
- **Фреймворк**: React с TypeScript
- **Управление состоянием**: React Context API (для простоты в MVP) или Zustand
- **Библиотека UI**: Безголовные UI-компоненты с Tailwind CSS для стилизации
- **Инструмент сборки**: Vite

### База данных
- **Основная**: PostgreSQL 14+
- **Кеш**: Redis 7+ (для сессий и кеширования справочных данных)
- **Хранилище файлов**: S3-совместимое объектное хранилище (MinIO для разработки, AWS S3 или аналогичное для продакшена)
- **Поиск**: Встроенный полнотекстовый поиск PostgreSQL и геоиндексация (PostGIS рассматривается для Фазы 2+)

### DevOps & Инфраструктура
- **Контейнеризация**: Docker
- **Оркестрация**: Docker Compose для разработки, Kubernetes рассматривается для Фазы 2+
- **CI/CD**: GitHub Actions
- **Мониторинг**: Подход на основе обещаний с простым логированием и метриками (будет улучшен в Фазе 2+)
- **Логирование**: Структурированное JSON-логирование (Winston или Pino)

### Интеграции с третьими сторонами
- **SMS**: Twilio (или аналогичный с бесплатным тарифом для MVP)
- **Провайдеры OAuth**: Google, Apple, Telegram, VK (через стандартные библиотеки OAuth 2.0)
- **Геокодирование и карты**: Yandex Maps (бесплатный тариф) с абстракционным слоем для легкой замены
- **Email**: SendGrid (или аналогичный) только для транзакционных писем
- **Загрузка файлов**: Предварительно подписанные URL для S3-совместимого хранилища

## Последствия

### Положительные
- **NestJS обеспечивает отличную поддержку DDD**: Модули, внедрение зависимостей, четкое разделение между Контроллерами, Сервисами и Репозиториями согласуется с ограниченными контекстами
- **TypeScript всюду**: end-to-end типовая безопасность снижает количество ошибок и улучшает опыт разработчика
- **Prisma ORM**: типобезопасный доступ к базе данных с отличным DX, простыми миграциями и хорошей поддержкой PostgreSQL
- **React + Vite**: быстрые сборки для разработки, отличная производительность и поддержка PWA из коробки
- **PostgreSQL**: надежный, функционально богатый, поддерживает JSONB для расширяемости, хорошие расширения GeoIP доступны
- **Знакомство команды**: Стек соответствует общим навыкам веб-разработки, снижая время адаптации
- **Потенциал масштабируемости**: Каждый слой может масштабироваться независимо (воркеры API, реплики БД для чтения, слой кеша)

### Отрицательные
- **Кривая обучения**: Команда может потребоваться изучить соглашения NestJS, если не имеет опыта
- **Накладные расходы для MVP**: NestJS добавляет некоторый boilerplate по сравнению с более простыми серверами Express.js
- **Размер пакета**: Приложения React могут иметь больший размер пакета (смягчается код-сплитингом и ленивой загрузкой)
- **Операционные расходы PostgreSQL**: Более сложная настройка, чем у SQLite (но необходима для продакшн-качеств)

### Нейтральные
- **Смягчение привязки к поставщику**: Абстракционные слои для SMS, OAuth, маппинга и хранилища позволяют переключать поставщиков
- **Четкий путь миграции**: Хорошо определенные интерфейсы между слоями позволяют постепенному рефакторингу при необходимости

## Подход к реализации
1. **Структура бэкенда**:
 - `src/modules/` содержащий доменные модули: `identity`, `animal`, `pet-marketplace`, `livestock-marketplace`, `admin`, `moderation`
 - Общие библиотеки: `src/lib/` (база данных, защитники аутентификации, DTOs, валидация)
 - Основное приложение: `src/app.module.ts`

2. **Структура фронтенда**:
 - `src/components/` для переиспользуемых UI-компонентов
 - `src/pages/` для маршрутных страниц
 - `src/hooks/` для пользовательских React-хуков
 - `src/lib/` для клиентов API, утилит, констант
 - `src/styles/` для конфигурации Tailwind и глобальных стилей

3. **База данных**:
 - Схема Prisma в `prisma/schema.sql` (фактически `prisma/schema.prisma`)
 - Директория миграций: `prisma/migrations/`
 - Seed-данные для справочных таблиц (породы, виды, города)

4. **DevOps**:
 - Docker compose файл для локальной разработки (бэкенд, фронтенд, postgres, redis, minio)
 - Отдельный продакшен compose или Helm charts для Фазы 2+

## Соответствие требованиям
- **Безопасность**: NestJS имеет встроенные защитники валидации, TypeScript снижает риски XSS, Prisma предотвращает SQL-инъекции
- **Производительность**: NestJS построен на Express (быстрый), сборки React/Vite оптимизированы, PostgreSQL с правильной индексацией
- **Поддерживаемость**: Четкие границы модулей, самодокументирующийся код TypeScript, хорошо протестированные зависимости
- **Расширяемость**: Модульная архитектура позволяет добавлять новые домены без нарушения существующих
- **Скорость команды**: Популярный стек с отличной документацией и поддержкой сообщества

## Рассмотренные альтернативы
1. **Ruby on Rails**
 - Плюсы: Конвенция вместо конфигурации, отлично подходит для MVP, ORM ActiveRecord
 - Минусы: Менее распространенный набор навыков, вопросы производительности при масштабировании, слабее поддержка TypeScript
 - Отклонено из-за знакомства команды и долгосрочных опасений по масштабированию

2. **Django (Python)**
 - Плюсы: Отличный ORM, встроенная админ-панель, надежные стандарты безопасности по умолчанию
 - Минусы: Ограничения асинхронности Python, менее идеально для эволюции в микросервисы, ORM менее гибок, чем Prisma для сложных запросов
 - Отклонено из-за предпочтения full-stack JavaScript/TypeScript

3. **Go (Gin/Echo) + React**
 - Плюсы: Отличная производительность, простое развертывание, сильная конкурентность
 - Минусы: Отсутствие установленных фреймворков DDD, слабее варианты ORM, меньше инструментов интеграции с фронтендом
 - Отклонено из-за меньшей начальной скорости разработки для сложного моделирования доменов

4. **Serverless (AWS Lambda)**
 - Плюсы: Отличное масштабирование, оплата по факту использования
 - Минусы: Проблемы с холодным стартом, сложное распределенное трассирование, сложнее реализовать долговыполняющиеся процессы, избыточно для масштаба MVP
 - Отклонено из-за увеличенной сложности и ненужного масштабирования для фазы валидации

5. **Laravel (PHP)**
 - Плюсы: Зрелая экосистема, ORM Eloquent, хорош для MVC
 - Минусы: Восприятие как legacy стек, менее идеально для эволюции реального времени, ограничения асинхронности PHP
 - Отклонено из-за предпочтений команды и долгосрочной технологической траектории

## Примечания
Это решение будет пересмотрено, если:
- Состав команды значительно изменится
- Профилирование производительности покажет узкие места в выбранном стеке
- Появятся новые требования, плохо поддерживаемые этим стек (например, необходимость в тяжелом реальном времени совместной работе)
