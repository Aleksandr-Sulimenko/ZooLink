# ADR-0002: Жёсткое разделение рынков домашних животных и скота

**Статус**: Принято  
**Дата**: 2026-05-28  

## Контекст и постановка проблемы

Платформа ZooLink должна обслуживать два различных рыночных сегмента с фундаментально разными требованиями:
- Рынок домашних животных: компаньонные животные (собаки, кошки, птицы, кролики, рептилии и т.д.) с акцентом на характер, здоровье для домашней среды
- Рынок скота: фермерские животные (крупный рогатый скот, лошади, овцы, козы, свины, птица и т.д.) с акцентом на продуктивные качества, генетику и коммерческую ценность

Изначально рассматривался подход с единой рыночной площадкой, где все животные могли быть размещены в едином интерфейсе с условными полями в зависимости от типа животного. Однако анализ показал значительные различия, которые нарушат пользовательский опыт и целостность платформы при обработке только через условный интерфейс.

## Драйверы решения

1. **Опыт пользователя**: Владельцы домашних животных и фермеры имеют разные психологические модели, терминологию и приоритеты
2. **Правила валидации**: Разные требования к атрибутам, логика валидации и бизнес-правила
3. **Поиск и обнаружение**: Разные аспекты поиска и потребности в фильтрации
4. **Правовые/нормативные**: Разные требования к соблюдению норм (особенно для отслеживания перемещения скота)
5. **Модерация**: Разные области фокуса для модерации и необходимая экспертиза
6. **Будущая расширяемость**: Разные пути эволюции для каждого рыночного сегмента

## Рассмотренные варианты

### Вариант 1: Единая рыночная площадка с условными полями
Единый интерфейс создания объявлений, который показывает/скрывает поля в зависимости от типа животного (домашнее животное vs скот).

Плюсы:
- Простая техническая реализация initially
- Единая кодовая база для создания/редактирования объявлений
- Общий интерфейс модерации

Минусы:
- Запутанный пользовательский опыт с нерелевантными полями, которые показываются/скрываются
- Сложная условная логика валидации
- Сложно поддерживать по мере расхождения требований
- Плохой опыт поиска/факетов (смешивание несовместимых фильтров)
- Модераторам потребуется экспертиза в обеих областях
- Сложнее развивать каждый сегмент независимо

### Вариант 2: Полностью отдельные рыночные площадки (Выбранный)
Отдельные пользовательские интерфейсы, валидация данных, поисковые API и очереди модерации для рынков домашних животных и скота, при этом совместно используются основные компоненты (аутентификация, сущность животного, управление пользователями и т.д.).

Плюсы:
- Чёткий, фокусированный пользовательский опыт для каждого сегмента
- Отдельная логика валидации и бизнес-правил
- Оптимизированный поиск/факеты для каждого домена
- Специализированные рабочие процессы модерации
- Независимая эволюция и масштабирование
- Чёткое разделение ответственности в кодовой базе
- Лучшая производительность (меньше, фокусированные API)

Минусы:
- Немного больше начальных усилий разработки
- Необходимость тщательно поддерживать действительно общие компоненты
- Дублирование some UI patterns (смягчается общей библиотекой компонентов)

## Решение

Мы будем реализовывать **жёсткое разделение между рынком домашних животных и рынком скота** с:
- Отдельные точки входа в UI (разные разделы/табы)
- Отдельные потоки создания и управления объявлениями
- Доменно-специфичные правила валидации и бизнес-логики
- Независимые поисковые API с доменно-соответствующей фильтрацией
- Отдельные очереди модерации и рабочие процессы
- Совместно используемые основные компоненты: аутентификация, сущность животного, профили пользователей, структуры организаций, системы уведомлений

Этот подход согласуется с принципами предметно-ориентированного проектирования (DDD), рассматривая рынок домашних животных и рынок скота как отдельные ограниченные контексты со своим всеобщим языком, при этом совместно используя ядро (идентичность пользователя, основная сущность животного и т.д.).

## Последствия

### Положительные
- Пользователи получают чёткий, фокусированный интерфейс, соответствующий их потребностям
- Команды разработки могут независимо работать над каждым рынком
- Легче реализовывать доменно-специфичные функции без влияния на другой сегмент
- Чёткие пути модерации снижают когнитивную нагрузку для модераторов
- Лучшая релевантность и производительность поиска
- Простая реализация валидации и бизнес-логики

### Отрицательные
- Начальные усилия разработки увеличились примерно на 15-20%
- Необходимость тщательного управления действительно общими компонентами
- Небольшое увеличение архитектурной сложности

### Нейтральные
- Модель данных остаётся единой (сущность животного служит обоим доменам)
- Аутентификация и управление пользователями остаются общими
- Основная инфраструктура (база данных, кэш, хранилище Pai) остаются общими

## Примечания по реализации

1. **Маршрутизация**: Отдельные верхнеуровневые маршруты для разделов `/pets` и `/livestock`
2. **API**: Отдельные пространства имён API или чёткое доменное разделение в общих API
3. **Компоненты UI**: Общая библиотека компонентов для общих элементов (кнопки, формы, модальные окна) но доменно-специфичные страницы и рабочие процессы
4. **Валидация**: Отдельные наборы правил валидации, применяемые на границе API
5. **Модерация**: Отдельные интерфейсы модерации с доменно-специфичными контрольными списками и требованиями обучения
6. **Поиск**: Отдельные конечные точки поиска с доменно-адаптированной фильтрацией и факетами
7. **Аналитика**: Отдельные панели метрик с доменно-специфичными KPI

## Связанные решения

- **ADR-0004**: Животное как агрегат (подтверждает, что сущность животного является разделяемым ядром)
- **ADR-0001**: Выбор технологического стека (обеспечивает модульную реализацию)
