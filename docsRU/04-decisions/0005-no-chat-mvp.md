# ADR-0005: Нет встроенного чата в MVP

**Статус**: Принято  
**Дата**: 2026-05-31  

## Контекст и постановка проблемы

Платформа ZooLink должна обеспечивать коммуникацию между покупателями и продавцами после того, как они обнаружат объявление, представляющее интерес. Изначально мы рассматривали реализацию функции внутри-приложения чата как часть MVP для возможности реального времени переговоров, вопросов и договорённостей.

Однако анализ обнаружил несколько проблем с внедрением чата в MVP, которые увеличили бы сложность, создали бы проблемы модерации и потенциально задержали бы основную функциональность рынка. Нам нужно было определить, является ли чат необходимым для MVP или может быть отложен на более позднюю фазу, при этом всё ещё предоставляя жизнеспособный пользовательский опыт для завершения транзакций.

## Драйверы решения

1. **Основное ценностное предложение**: Основная ценность заключается в соединении покупателей и продавцов через объявления; коммуникация вторична по отношению к обнаружению
2. **Бремя модерации**: Чат в реальном времени требует активного мониторинга для предотвращения домогательств, мошенничества и неподобающего контента
3. **Сложность разработки**: Реализация чата включает WebSockets, сохранение сообщений, уведомления и компоненты UI
4. **Безопасность пользователей**: Чат открывает возможности для ухода за детьми, мошенничества и других проблем безопасности, требующих мощной модерации
5. **Технические зависимости**: Чат может потребовать дополнительной инфраструктуры (брокер сообщений, серверы присутствия)
6. **Альтернативные каналы коммуникации**: Пользователи могут обмениваться контактной информацией для внешней коммуникации
7. **Познавательная ценность**: Наблюдение того, как пользователи коммуницируют внешним образом, может информировать будущую реализацию чата
8. **Фокус области**: MVP должен проверить основные концепции рынка перед добавлением функций коммуникации

## Рассмотренные варианты

### Вариант 1: Встроенный чат в MVP
Реализовать функциональность чата в реальном времени внутри платформы как часть первоначального выпуска.

Плюсы:
- Бесшовный пользовательский опыт без выхода из приложения
- Вся коммуникация отслеживается внутри платформы (потенциал для аналитики)
- Мгновенные уведомления и ответы
- Современное ожидание для приложений-рынков

Минусы:
- Значительные затраты на разработку, задерживающие основные функции рынка
- Увеличенная сложность и стоимость модерации
- Потенциал использования платформы для домогательств или незаконной деятельности
- Требуется инфраструктура реального времени (серверы WebSocket, брокер сообщений)
- Необходимы компоненты UI для чата и системы уведомлений
- Отвлекает внимание от проверки основного обнаружения объявлений и потока транзакций

### Вариант 2: Отложенный чат с обменом контактами (Выбранный)
Отложить встроенный чат на будущую фазу; вместо этого обмениваться проверенной контактной информацией после публикации объявлений.

Плюсы:
- Сосредоточивает разработку на основной функциональности рынка (объявления, поиск, модерация)
- Устраняет бремя модерации, связанное с чатом, в MVP
- Проще техническая реализация (отсутствие инфраструктуры обмена сообщениями в реальном времени)
- Пользователи всё ещё могут коммуницировать через предпочтительные внешние каналы (телефон, Telegram, VK)
- Обмен контактами создаёт естественный стимул для пользователей завершать объявления (для получения контактной информации)
- Предоставляет возможность изучить реальные шаблоны коммуникации перед построением чата
- Более быстрый выход на рынок основной ценностной предложения

Минусы:
- Требует от пользователей переключения приложений для коммуникации
- Нет истории коммуникации, сохранённой внутри платформы
- Менее бесшовный опыт, чем встроенный чат
- Потенциал обмена ложной контактной информацией (смягчается верификацией)

### Вариант 3: Отложенный обмен контактами (Рассматривался как альтернатива)
Показывать контактную информацию только после того, как обе стороны отметили транзакцию как завершённую.

Плюсы:
- Максимальная уверенность в том, что коммуникация привела к реальным транзакциям
- Сокращает потраченную коммуникацию для неуспешных запросов

Минусы:
- Создаёт проблему яйца и курицы: пользователям нужно коммуницировать, чтобы завершить транзакцию
- Существенно ухудшает пользовательский опыт для легитимных запросов
- Отвращает от использования платформы, если пользователи не могут легко связаться с продавцами
- Трудно верифицировать завершение без внешнего подтверждения

## Решение

Мы **отложим встроенный чат до будущей фазы (Фаза 2 или позже)** и реализуем **механизм обмена контактами** в MVP следующим образом:

1. **Обмен контактной информацией**:
   - После достижения объявлением статуса `ACTIVE` заинтересованные пользователи могут запросить просмотр контактной информации
   - Клик по кнопке «Показать контакты» раскрывает номер телефона владельца и ссылки на профили Telegram/VK (если предоставлены и authorised)
   - Точный адрес никогда не раскрывается по причинам безопасности и конфиденциальности
   - Система ведёт журнал запросов контактов для аналитики и мониторинга безопасности

2. **Управление контактной информацией**:
   - Пользователи могут по желанию предоставить имена пользователей Telegram и VK во время настройки профиля
   - Номер телефона обязателен для регистрации (проверяется через SMS) и может быть показан в контактах
   - Сбор e-mail является добровольным и не отображается в контактах (используется только для уведомлений)
   - Пользователи могут выбрать, какие методы контакта показывать (телефон, Telegram, VK, любую комбинацию)

3. **Меры конфиденциальности и безопасности**:
   - Ни при каких обстоятельствах не делиться точным адресом
   - Контактная информация раскрывается только для объявлений со статусом `ACTIVE` (после модерации)
   - Журналирование того, кто запросил контактную информацию для какого объявления и когда
   - Ограничение скорости запросов контактов для предотвращения скрапинга
   - Возможность для пользователей блокировать других от просмотра их контактной информации

4. **Поток пользователя**:
   - Пользователь обнаруживает объявление интереса в результатах поиска
   - Просматривает детали объявления (описание, фотографии, цена и т.д.)
   - Нажимает кнопку «Показать контакты»
   - Система проверяет, что объявление находится в статусе `ACTIVE`, и ведёт журнал запроса
   - Показывает доступную контактную информацию (телефон, ссылки Telegram/VK, авторизованные владельцем)
   - Пользователь инициирует контакт вне платформы через свой предпочтительный метод

## Последствия

### Положительные
- Ускоряет доставку основной функциональности рынка
- Устраняет значительное бремя модерации, связанное с реальным временем чата
- Проще техническая реализация и требования к инфраструктуре
- Сосредотачивает MVP на проверке обнаружения объявлений, модерации и намерения транзакций
- Предоставляет безопасный, контролируемый механизм для внешней коммуникации
- Позволяет наблюдать за реальными шаблонами коммуникации для информирования будущего дизайна чата
- Сокращает область применения и повышает шансы на успешный запуск MVP

### Отрицательные
- Пользователи должны переключать приложения для коммуникации
- Нет истории коммуникации, сохранённой внутри платформы
- Менее интегрированный опыт по сравнению с встроенным чатом
- Требуется обучение пользователей тому, почему обмен контактами работает таким образом

### Нейтральные
- Не препятствует добавлению чата в будущих фазах (архитектура спроектирована для расширяемости)
- Основная модель данных unchanged (добавляются необязательные поля социальных сетей в профиль пользователя)
- Обмен контактами может быть развит в сторону чата постепенно (например, начиная с пересылки сообщений)

## Примечания по реализации

1. **Модель данных**:
   - Таблица `users` включает необязательные поля `telegram_username`, `vk_username`
   - `phone_number` хранится как хеш (`phone_hash`) для верификации, оригинал не сохраняется
   - Поле JSONB `contact_sharing_preferences` для управления тем, что показывается
   - Таблица `contact_requests` журнала: requester_id, listing_id, timestamp, метод контакта, показанный

2. **Проектирование API**:
   - Конечные точки профиля для установки/обновления контактной информации и предпочтений по её обмену
   - Конечная точка детали объявления включает логику условного отображения полей контакта на основе статуса и разрешений
   - Конечная точка журналирования запросов контактов (вызывается при нажатии «Показать контакты»)
   - Middleware ограничения скорости на конечных точках запросов контактов

3. **Компоненты UI**:
   - Кнопка «Показать контакты» на странице детали объявления (видна только для объявлений `ACTIVE`)
   - Модальное окно или развернутая секция, показывающая доступную контактную информацию
   - Иконки/обработчики ссылок для телефона (tel:) и Telegram/VK (глубокие ссылки или универсальные ссылки)
   - Чёткое указание того, что точный адрес никогда не делится
   - Настройки профиля для управления тем, какие методы контакта показывать

4. **Бизнес-правила**:
   - Контактная информация показывается только для объявлений со статусом `ACTIVE`
   - Пользователи должны быть authenticated для запроса контактной информации (предотвращает скрапинг)
   - Владельцы могут изменять предпочтения обмена контактами в любой момент (влияет на будущие запросы)
   - Система отслеживает запросы контактов, но не сохраняет содержание фактической коммуникации
   - Владельцы могут блокировать конкретных пользователей от просмотра их контактной информации (через систему сообщений/блокировки)

## Связанные решения

- **ADR-0003**: Премодерация рабочего процесса (обмен контактами только после модерации)
- **ADR-0004**: Животное как агрегат (объявления привязаны к проверенному животному и владельцу)
- **ADR-0002**: Жёсткое разделение рынков (обмен контактами работает одинаково в обоих доменах)
- **ADR-0001**: Выбор технологического стека (NestJS поддерживает функции реального времени через WebSockets для будущего чата)
