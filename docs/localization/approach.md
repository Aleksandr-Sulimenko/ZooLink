# Подход к локализации в проекте ZooLink

## Выбранное решение: Улучшенный JSONB-подход

После анализа различных подходов к локализации в реляционных базах данных, для проекта ZooLink выбран улучшенный JSONB-подход. Это решение представляет собой эволюцию текущей реализации проекта, сохраняя при этом простоту разработки и поддержки.

### Почему JSONB-подход?

1. **Соответствует текущей реализации** - проект уже использует JSONB-подход для локализации, что минимизирует необходимость рефакторинга существующего кода
2. **Простота разработки и поддержки** - знакомый подход для команды разработки
3. **Хорошая производительность чтения** - все данные локализации доступны в одном запросе без необходимости JOIN'ов
4. **Легкое добавление новых языков** - не требует изменений схемы БД при добавлении новых языков
5. **Гибкость** - можно хранить разные типы локализуемых данных в одном формате
6. **Индексируемость** - PostgreSQL предоставляет отличные средства для индексации JSONB полей
7. **Соответствие современным практикам** - широко используется в индустрии для аналогичных задач

### Что мы улучшаем в текущей реализации?

#### 1. Стандартизация структуры JSONB
Вводим единый формат для всех локализуемых полей:
```json
{
  "ru": "значение на русском",
  "en": "значение на английском",
  "fr": "",  // пустая строка для непереведённых языков
  "es": "",
  "zh": ""
}
```

Преимущества:
- Единый формат упрощает работу с данными
- Четкое различие между пустым переводом и отсутствующим ключом
- Возможность определять, какие языки поддерживаются приложением через конфигурацию

#### 2. Добавление функций-помощников в базе данных
Создаем SQL-функции для упрощения работы с локализованными данными:

```sql
-- Функция для получения перевода на указанном языке с fallback
CREATE OR REPLACE FUNCTION get_localized(
    data JSONB,
    lang TEXT DEFAULT current_setting('app.current_language', true),
    fallback_lang TEXT DEFAULT 'en'
) RETURNS TEXT AS $$
BEGIN
    RETURN COALESCE(
        data->>lang,
        data->>fallback_lang,
        ''  -- окончательный fallback
    );
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- Функция для проверки наличия перевода
CREATE OR REPLACE FUNCTION has_translation(
    data JSONB,
    lang TEXT
) RETURNS BOOLEAN AS $$
BEGIN
    RETURN (data->>lang) IS NOT NULL AND (data->>lang) <> '';
END;
$$ LANGUAGE plpgsql IMMUTABLE;
```

#### 3. Индексация для производительности
Добавляем GIN индексы для эффективного поиска по переводам:
```sql
-- Индекс для поиска организаций по названию на конкретном языке
CREATE INDEX idx_organizations_name_localized_en 
ON organizations USING GIN ((name_localized -> 'en'));

CREATE INDEX idx_organizations_name_localized_ru 
ON organizations USING GIN ((name_localized -> 'ru'));

-- Аналогично для других полей и таблиц
```

#### 4. Управление поддерживаемыми языками
Создаем таблицу поддерживаемых языков для централизованного управления:
```sql
CREATE TABLE supported_languages (
    code CHAR(2) PRIMARY KEY,  -- ISO 639-1 код языка
    name_localized JSONB NOT NULL,  -- локализованное название языка
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    display_order INTEGER NOT NULL DEFAULT 0,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);
```

#### 5. Механизм fallback и язык по умолчанию
Реализуем на уровне приложения:
- Устанавливаем язык сессии через `SET app.current_language = 'ru';`
- Используем предопределенный fallback язык (например, 'en')
- Логика: запрошенный язык → fallback → язык по умолчанию → пустая строка

### Как это соответствует стандартам и лучшим практикам?

#### Соответствие ГОСТ и международным стандартам:
- Поддерживает принцип доступности информации на официальных языках РФ (русский) и языках международного общения (английский)
- Обеспечивает возможность расширения для поддержки дополнительных языков в соответствии с требованиями локализации
- Позволяет сохранять культурную и лингвистическую специфику через качественные переводы

#### Современные подходы к i18n:
- Использует де-факто стандарт индустрии (JSONB в PostgreSQL) для хранения мультиязычных данных
- Соответствует принципу "один источник истины" при правильной реализации fallback механизмов
- Поддерживает подход " язык по умолчанию + переводы" который широко признан как эффективный
- Обеспечивает разделение представления и данных через уровни абстракции

## План внедрения

Внедрение улучшений локализации будет выполнено поэтапно:

### Этап 1: Стандартизация и базовые улучшения
- Приведение всех JSONB-полей локализации к единому формату
- Добавление функций-помощников в базе данных (`get_localized`, `has_translation`, `set_app_language`)
- Создание таблицы `supported_languages`
- Добавление комментариев и документации к схемам БД

### Этап 2: Оптимизация производительности
- Анализ наиболее частых запросов к локализованным данным
- Добавление целевых GIN индексов для часто используемых языков и полей
- Рассмотрение материализованных представлений для сложных композитных запросов

### Этап 3: Улучшение качества переводов
- Внедрение механизмов отслеживания незавершенных переводов
- Добавление административного интерфейса для управления переводами
- Внедрение проверок на этапе CI/CD на наличие обязательных переводов

### Этап 4: Расширение функциональности (по необходимости)
- Рассмотрение перехода на гибридный подход для критически важных полей с высокой частотой обновлений
- Внедрение механизмов версионирования переводов если потребуется аудит изменений
- Рассмотрение интеграции с системами машинного перевода для предварительного заполнения

## Ожидаемые результаты

После внедрения улучшений мы получим:
1. Стандартизированную и легко расширяемую систему локализации
2. Улучшенную производительность чтения и поиска по локализованным данным
3. Упрощенную работу с локализацией на уровне приложения благодаря функциям-помощникам
4. Централизованное управление поддерживаемыми языками
5. Готовность к легко добавлять новые языки без изменений схемы БД
6. Соответствие современным стандартам и лучшим практикам в области i18n/l10n

Это решение оптимально балансирует между требованиями расширяемости, производительностью и простотой реализации, что особенно важно для проекта среднего масштаба как ZooLink.
