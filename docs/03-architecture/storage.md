# Хранилище данных

Этот документ описывает стратегии хранения данных в системе ZooLink, включая базы данных, кэш, объектное хранилище и файлы.

## Обзор подхода к хранению

ZooLink использует многоуровневую архитектуру хранения для оптимизации производительности, стоимости и соответствия требованиям:

1. **Основное хранилище данных:** PostgreSQL (с PostGIS для геопространственных запросов)
2. **Кеш层:** Redis для быстрого доступа к часто используемым данным
3. **Объектное хранилище:** S3-совместимое хранилище для больших бинарных объектов (фото, документы)
4. **Файловая система:** Локальное хранилище только для временных файлов и кэша в процессе разработки
5. **Поисковый движок:** Планируемый переход на специализированный поисковый движок (Elasticsearch/OpenSearch) для сложных поисковых запросов

## Основное хранилище данных (PostgreSQL)

### Обоснование выбора
- ACID-транзакции для обеспечения целостности данных
- Расширяемость через JSONB для схемы без миграций
- PostGIS для эффективных геопространственных запросов
- Мощные возможности индексирования и оптимизации запросов
- Широкий набор инструментов для администрирования и мониторинга
- Зрелая экосистема и community поддержка

### Схема хранения
Схема данных документирована в [`database_schema.sql`](../database_schema.sql) и соответствует концептуальным моделям доменов.

### Особенности реализации
- **UUID в качестве первичных ключей:** Для глобальной уникальности и поддержки репликации/шардинга
- **Временные метки:** `created_at` и `updated_at` для всех сущностей через триггеры
- **Мягкое удаление:** Используется для ключевых сущностей (животные, пользователи, объявления) вместо физического удаления
- **JSONB колонки:** Для расширяемых атрибутов и сложных структур (медицинские записи, репродуктивные данные, локализация)
- **Индексы:** Стратегически подобранные для частых запросов (по владельцу, виду, породе, статусу и т.д.)
- **Ограничения:** CHECK ограничения для бизнес-правил на уровне БД (например, точно один владелец для животного)
- **Триггеры:** Для автоматического обновления полей, ведения журналов и поддержания целостности

### Геопространственные данные
- Основное хранилише: PostGIS 확장 (география/геометрия типы)
- Резервный вариант (без PostGIS): отдельные широта/долгота колонки с обычными индексами
- Индексы: GIST индексы для эффективного поиска по радиусу и пересечениям
- Единицы измерения: Метры для расстояний, SRID 4326 (WGS84) для координат

## Кеширование (Redis)

### Обоснование выбора
- In-memory производительность для горячих данных
- Поддержка структур данных (строки, хеши, списки, сеты, сортированные сеты)
- Pub/Sub для обмена сообщениями между сервисами
- TTL (время жизни) для автоматического очистки
- Кластеризация и репликация для высокой доступности
- Persistence варианты для восстановления после перезагрузки

### Что кешируется
1. **Профили пользователей и сессии:** Основные данные для авторизации и персонализации
2. **Справочные данные:** Виды, породы, города (редко меняются, часто запрашиваются)
3. **Результаты поиска:** Популярные запросы для уменьшения нагрузки на БД
4. **Домашние страницы и ленты:** Кешированные объявления для главной страницы и персональных лент
5. **Счетчики и статистика:** Просмотры объявлений, популярность, активность пользователей
6. **Очереди задач:** Фоновая обработка через Redis lists или специализированные очереди
7. **Rate limiting:** Счетчики запросов для защиты от злоупотреблений
8. **Сессии модерации:** Временные данные для модераторов при работе с очередью

### Паттерны использования
- **Чтение-through:** Приложение сначала проверяет кеш, затем БД при промахе
- **Запись-through:** При обновлении данных обновляются zarówno БД, так и кеш
- **Инвалидация по событиям:** Очистка кеша при изменении исходных данных через триггеры или обработчики событий
- **Время жизни (TTL):** Различные TTL в зависимости от типа данных (сессии: часы, справочники: дни, поиск: минуты)

## Объектное хранилище (S3-совместимое)

### Обоснование выбора
- Масштабируемость для хранения растущего объема медиа-контента
- Стоимость: платите только за используемое хранилище и передачу данных
- Высокая доступность и долговечность данных (99.999999999%)
- Интеграция с CDN для быстрой доставки контента пользователям
- Безопасность: тонко granulированные права доступа через политики и IAM
- Совместимость: возможность переключения между провайдерами (MinIO, AWS S3, Google Cloud Storage, Azure Blob)

### Что хранится
1. **Фотографии объявлений:** Основные и превью изображения в различных размерах
2. **Аватары пользователей:** Профильные изображения в разных разрешениях
3. **Медиа-файлы объявлений:** Видео, документы, аудиозаписи (по планам на будущие фазы)
4. **Экспорт/импорт данные:** Резервные копии, архивы для перемещения данных
5. **Статические ресурсы:** В редких случаях когда не используется отдельный CDN для фронтенда

### Организация данных
- **Бакетах:** Разделение по типам данных и средам (dev, staging, prod)
- **Префиксах/Папках:** Логическая структура внутри бакета для упорядочивания
- **Именование файлов:** UUID-based имена для предотвращения конфликтов и обеспечения равномерного распределения
- **Метаданные:** Хранение дополнительной информации о файлах (тип контента, размер, дата загрузки и т.д.)
- **Жизненный цикл:** Правила перемещения в более дешевые слои хранения и удаления старых данных

### Безопасность и доступ
- **Предварительно подписанные URL:** Временные ссылки с ограниченным сроком действия для загрузки/ скачивания
- **Политики доступа:** Минимальные необходимые привилегии для разных типов операций
- **Шифрование:** Серверное шифрование данных в покое (SSE-S3 или аналогичное)
- **CORS-настройки:** Ограничение прямых запросов к объектам только с авторизованных доменов
- **Журналирование доступа:** Отслеживание who accessed what и when для аудита и безопасности

### Интеграция с приложением
- **URL генерация:** Сервис уровня приложения создает корректные предварительно подписанные URL
- **Кеширование на CDN-уровне:** Прокирование часто запрашиваемых объектов через CDN
- **Обработка изображений:** Изменение размера, обрезка, форматирование при загрузке или "на лету"
- **Оптимизация:** Автоматическое оптимизирование изображений для веба (сжатие, выбор формата)
- **Резервирование:** Стратегии при сбое основного хранилища (резервные бакета в других регионах)

## Локальное хранилище (только для разработки и временных файлов)

### Использование в разработке
- **MinIO в Docker-compose:** Локальное S3-совместимое хранилище для разработки
- **SQLite/PostgreSQL локально:** Для быстрой инициализации и тестирования
- **Redis локально:** Для разработки и тестирования кэш-зависимого функционала
- **Временные файлы:** Только для операции во время запроса (загрузка, обработка и немедленное удаление)

### Ограничения в production
- **Запрет на хранение пользовательских данных:** Никакие постоянные пользовательские данные не должны храниться на локальном диске серверов приложений
- **Временный буфер:** Только для обработки загрузок перед отправкой в объектное хранилище
- **Логи и метрики:** Стандартные системные журналы, но без ПДУ или бизнес-данных
- **Кеш файлов:** Только для статических资产 при специальной конфигурации (не рекомендуется для масштабирования)

## Поисковый движок (планируемый)

### Текущее состояние (MVP)
- Основной поиск: Через PostgreSQL с PostGIS индексами
- Полнотекстовый поиск: Через PostgreSQL tsvector и GIN индексы
- Фильтрация и сортировка: Стандартные SQL операции

### Планируемое улучшение (Фаза 2)
- **Технология:** Elasticsearch или OpenSearch
- **Причина:** Более сложные поисковые запроies, аналитика, рекомендации
- **Интеграция pattern:** 
  - Первичная запись все еще в PostgreSQL (источник истины)
  - Синхронизация в поисковый движок через событие-ориентированную архитектуру أو через connective tissue (Logstash, предварительно построенные коннекторы)
  - Чтение для поиска: в основном из поискового движка с fallback на БД
- **Поля для индексации:**
  - Текстовые: заголовок, описание, кличка животного, порода, вид
  - Структурированные: цена, расположение (гео-точки), дата создания, тип объявления
  - Категориальные: статус, список животных, особенности (стерилизован, прививки и т.д.)
- **Особенности:**
  - Гео-поиск с бустингами по релевантности и свежести
  - Подсказки и автодополнение
  - Выделение найденных фрагментов
  - Факетная навигация (фильтрация по множественным измерениям)
  - Персонализация результатов на основе поведения пользователя

## Стратегия резервного копирования и восстановления

### База данных
- **Логическое резервное копирование:** Ежедневные дампы с pg_dump
- **Физическое резервное копирование:** Еженедельные base backup с WAL архивированием (если требуется PITR)
- **Географическое распределение:** Хранение резервных копий в разных регионах/зоны доступности
- **Проверка восстановления:** Регулярное тестирование восстановления из резервных копий
- **Ретенционная политика:** 30 дней для ежедневных, 12 недель для еженедельных, 12 месяцев для ежемесячных

### Объектное хранилище
- **Встроенная репликация:** Межрегиональная репликация (для критичных данных)
- **Версионирование:** Включено для защиты от случайного перезаписывания и удаления
- **Жизненный цикл:** Автоматическое перемещение в холодное хранилище и удаление старых версий
- **Резервное копирование:** Дополнительные резервные копии в отдельных аккаунтах/бакетах для критических данных

### Кеш
- **Не резервируется:** Кеш считается временным и может быть восстановлен из источника истины
- **Теплый старт:** Стратегии для быстрого заполнения кеша после перезагрузки (предугадывание популярных запросов)

## Стратегия архивирования и устаревания данных

### Данные, подлежащие архивации
- **Старые объявления:** Объявления, завершенные более года назад (продажа, окончание срока аренды и т.д.)
- **Неактивные пользователи:** Пользователи, не входившие в систему более 2 лет
- **Исторические медицинские записи:** Старше определенного периода (по законодательным требованиям)
- **Журналы активности:** Старые события старше периода хранения (по политике компании и регулятивным требованиям)

### Методы архивации
- **Архивация в БД:** Перемещение в отдельные таблицы/схемы с уменьшенным набором индексов
- **Экспорт в объектное хранилище:** Сериализация в формат для долгосрочного хранения (Parquet, Avro, JSON lines)
- **Выгрузочные наборы:** Регулярное создание аналитических выгрузов для BI и машинного обучения

### Политики доступа к архиву
- **Ограниченный доступ:** Только для аналитических и регулятивных целей
- **Процедура запроса:** Формальный процесс для доступа к архивированным данным
- **Шифрование:** Дополнительное шифрование на уровне приложения для архива чувствительных данных

## Мониторинг и обслуживание хранилищ

### База данных
- **Метрики производительности:** Время выполнения запросов, использование соединений, соотношение кэш-промахов
- **Метрики ресурсов:** Использование CPU, памяти, дискового пространства, ширины полосы диска I/O
- **Журналы медленных запросов:** Выявление и оптимизация узких мест
- **Репликация и отказоустойчивость:** Мониторинг задержек репликации и состояния узлов

### Кеш
- **Статистика попаданий/промахов:** Коэффициент эффективности кеша
- **Использование памяти:** Постоянное мониторинг, очистка иEviction политики
- **Статистика команд:** Распределение операций (get, set, incr, ttl и т.д.)
- **Подключения и клиенты:** Количество активных подключений, отказ от соединений

### Объектное хранилище
- **Статистика запросов:** Количество и объем GET, PUT, DELETE операций
- **Коды ошибок:** Отслеживание ошибок доступа, превышения квоты, проблем с авторизацией
- **Пропускная способность:** Входящий и исходящий трафик к бакетам
- **Жизненный цикл:** Выполнение правил перехода между классами хранения

### Оповещения
- **Пороговые значения:** Критические уровни для времени отклика, использования ресурсов, частоты ошибок
- **Уведомления:** Через настроенные каналы (email, SMS, Slack, PagerDuty и т.д.)
- **Дашборды:** Визуализация ключевых метрик в реальном времени и исторических трендов

## Планы эволюции

### Краткосрочная (MVP-Faza 1)
- Оптимизация индексов PostgreSQL для реальных шаблонов запросов
- Настройка политик TTL в Redis для разных типов данных
- Внедрение базового жизненного цикла в объектном хранилище
- Установка мониторинга и базовых оповещений для всех слоев хранения

### Среднесрочная (Faza 2)
- Внедрение специализированного поискового движка для сложных запросов
- Географическое распределение баз данных для мультирегионального доступа
- Расширение использования объектного хранилища для медиа-файлов и документов
- Внедрение аналитического хранилища (data warehouse) для бизнес-аналитики

### Долгосрочная (Faza 3+)
- Рассмотрение переход на cloud-native базы данных (Amazon Aurora, Google Cloud SQL)
- Исследование распределенных согласованных хранилищ для глобального_scale
- Внедрение мульти-модельной базы данных для различных типов данных (документ, граф, wide-column)
- Оптимизация затрат через многоуровневое хранение и автоматическое перемещение данных между уровнями
