## 👨‍💼 СУБ-АГЕНТ 1: System Architect (Системный архитектор)
> *Фокус: соответствие стека бизнес-целям, overengineering, кадры в РФ, горизонтальное масштабирование.*

### Что выбрано (канон)
Node.js (TypeScript) · NestJS · Prisma · REST + OpenAPI 3.0 · JWT + refresh · RBAC через CASL + NestJS Guards · модульный монолит (`identity`, `animal`, `pet-marketplace`, `livestock-marketplace`, `admin`, `moderation`).

### ✅ Что хорошо
- **NestJS — правильный выбор под DDD.** Модули = bounded contexts, DI, чёткое разделение Controller/Service/Repository. Это ровно то, что нужно проекту с явными доменами (ADR-0002 hard-split рынков ложится в модули один-в-один).
- **Модульный монолит, а не микросервисы** (в ADR и в `performance_specification.md` §Scalability — «Microservices: not for MVP; modular monolith»). Это **зрелое** решение. Для нагрузок из `performance_specification.md` (500 одновременных, 50 RPS среднее, 200 пик) монолит на 2–3 репликах за балансировщиком закрывает требования с большим запасом.
- **TypeScript end-to-end** — единый язык фронт/бэк, переиспользование типов и DTO.

### 🔴 Критический вопрос 1: Overengineering
**Да, в архитектуре (но не в ADR) есть явный overengineering для MVP:**

1. **gRPC между сервисами** (`component-diagram.md` §Interface Contracts). В монолите межмодульные вызовы — это вызов метода в DI-контейнере. gRPC здесь — это решение проблемы, которой ещё нет.
2. **Event Bus + Pub/Sub** как отдельная инфраструктура. При этом в схеме **уже есть `outbox_events`** — это правильный паттерн (Transactional Outbox), но он **подразумевает relay-процесс и брокер**, которого нет в ADR. Налицо пробел: либо убрать Event Bus из MVP (события обрабатывает фоновый воркер прямо из `outbox_events` через `pg_notify`/polling), либо явно ввести брокер. Сейчас — «ни рыба ни мясо».
3. **Полный Kubernetes в `deployment-diagram.md`** (control plane, 3 ноды, HPA/VPA/Cluster Autoscaler, etcd). Для MVP это месяцы DevOps-работы и постоянная эксплуатационная нагрузка ради нагрузки, которую держит один сервер с Docker Compose. ADR это понимает («Docker Compose for development, Kubernetes considered for Фаза 2+») — диаграмма противоречит ADR.
4. **SSR Web Gateway** (`component-diagram.md`) при том, что фронт — SPA/PWA на Vite (CSR). SSR не описан во `08-frontend-architecture.md`. Это призрачный компонент.

> **Рекомендация архитектора:** заморозить MVP-инфраструктуру на уровне ADR (Docker Compose → один managed-хост или пара VM + reverse-proxy). K8s/gRPC/service-mesh/Event-Bus вынести в явный раздел «Target Architecture (Фаза 2+)» и **не реализовывать в MVP**.

### 🔴 Критический вопрос 2: Кадры под стек в РФ
- **Node.js / TypeScript / React** — один из самых распространённых стеков в РФ. Кадры есть в избытке, рынок глубокий. ✅
- **NestJS** — популярен и растёт, но это middle+ навык: junior-у нужен онбординг (DI, декораторы, модульность). Риск управляемый. 🟡
- **Prisma** — здесь **главный кадровый и технический риск**. В РФ команды чаще владеют **TypeORM** (что и «проговаривается» в `container-diagram.md`: «SQL via TypeORM/Prisma» — рассинхрон с ADR, где однозначно Prisma). Для проекта с тяжёлой гео-логикой и активным JSONB слабость Prisma важна (см. Суб-агент 2). 🔴

> **Рекомендация:** зафиксировать ОДИН ORM. ARB склоняется к разрешению в пользу гибкости запросов (см. вердикт DBA), а не к копипасту из диаграмм.

### 🟢 Готовность к горизонтальному масштабированию
Архитектурно стек масштабируется правильно: **stateless API + JWT** (нет серверных сессий в памяти → можно множить реплики), Redis для общего состояния, read-реплики PostgreSQL для read-heavy (гео-поиск, просмотр объявлений) заложены в `component-diagram`. Целевые цифры perf-спеки скромные и достижимы. Узкое место при росте — **не приложение, а PostgreSQL и поиск** (см. Суб-агент 2), а не выбор языка/фреймворка.

### Вердикт Суб-агента 1: **🟡 УПРОСТИТЬ (для MVP), стек оставить**
Ядро (NestJS + TS + REST) — **оставить**. Инфраструктурную обвязку (K8s, gRPC, Event Bus, SSR, service mesh) — **вырезать из MVP** и пометить как Фаза 2+. Привести диаграммы в соответствие с ADR, а не наоборот.

## 🗄️ СУБ-АГЕНТ 2: Senior DBA (Администратор баз данных)
> *Фокус: выбор БД, схема, индексы, M:N, кэш, поиск, гео.*

### SQL vs NoSQL — выбор PostgreSQL
**Полностью поддерживаю.** Это маркетплейс с деньгами, владением (XOR-инвариант животного), модерацией и аудитом — нужны **ACID-транзакции и ссылочная целостность**. NoSQL здесь был бы ошибкой. PostgreSQL даёт реляционную целостность + JSONB для расширяемости + гео + полнотекст в одном движке. ✅ **9/10.**

⚠️ **Версия рассинхронизирована:** ADR говорит «PostgreSQL 14+», CLAUDE.md/аудиты валидировались на **PG14**. Рекомендация: зафиксировать минимум **PostgreSQL 16** (улучшения по логической репликации, производительности параллельных запросов, и это всё ещё LTS-актуально). Без явной фиксации команда возьмёт «что попало».

### Схема и индексы — оценка по ERD
Схема (26 сущностей) **зрелая для предпроектной стадии**: UUID для бизнес-сущностей, INT для справочников (`species_id`/`breed_id`/`city_id` — корректное решение из аудита), soft-delete, `outbox_events`, append-only `moderation_decisions`, идемпотентность платежей (`idempotency_key UNIQUE`). Индексы в `data-model.md` описаны разумно (составные `type+active`, частичные по активным, GIN по JSONB, GiST по гео).

**Замечания DBA:**

1. 🔴 **Конфликт типа денег между документами.** `data-model.md` (строка 140) объявляет `price_cents INTEGER`, а ERD и аудит — `BIGINT`. `INTEGER` (макс ~21.5 млн в рублях при копейках) **переполнится на livestock-сделках** (племенной бык/партия скота легко >215 000 ₽ → в копейках уже за пределами комфорта, а оптовые партии — тем более). **Однозначно BIGINT minor units.** Привести `data-model.md` в соответствие с ERD.
2. 🟡 **M:N связи учтены корректно.** `organization_users` (junction с UK `organization_id,user_id`) и `favorites` (UK `user_id,listing_id`) — правильно разрешённые многие-ко-многим. Родословная (`mother_id`/`father_id` self-FK) — это рекурсивная иерархия, не M:N; для глубоких запросов по дереву понадобится `WITH RECURSIVE` — заложить индексы на оба FK (в `data-model` они не перечислены явно для pedigree).
3. 🟡 **JSONB-тяжёлая модель.** Локализация, `health_records`, `reproductive_data`, `metadata` — всё в JSONB. Это гибко, но: (а) GIN-индексы обязательны и должны быть в DDL, не «по желанию»; (б) запросы/фильтры по локализованным полям и поиск по морфологии в JSONB — боль. Не злоупотреблять: то, по чему фильтруют и сортируют, должно быть колонками.
4. 🟡 **`Prisma` как ORM плохо дружит с двумя ключевыми вещами схемы:** `GEOGRAPHY(POINT,4326)` (PostGIS-тип Prisma нативно не моделирует — придётся `Unsupported()` + raw SQL) и сложными JSONB-запросами. Для этой схемы **TypeORM или Prisma + слой raw-SQL/Kysely** дадут меньше боли на гео и JSONB.

### Кэш (Redis) — достаточно ли?
**Да, для маркетплейса этого масштаба Redis 7 — правильно и достаточно.** `storage.md` грамотно перечисляет: сессии/refresh, справочники (TTL 24h), результаты поиска, ленты, счётчики, **rate-limiting**, очереди задач. Memcached не нужен — Redis покрывает и кэш, и rate-limit, и (при необходимости) лёгкую очередь. ✅ **8/10.** Одно предупреждение: **не делать Redis источником истины** для счётчиков просмотров без периодического сброса в PG (потеря при рестарте). Persistence (AOF) включить.

### Поиск — главная зона риска
Здесь документы **противоречат сами себе три раза**:
- ADR: «PostgreSQL full-text + geo-indexing, PostGIS — Фаза 2».
- `storage.md`: «PostGIS — **основное** хранилище гео» (т.е. уже в MVP).
- `container/component/deployment` диаграммы: **Elasticsearch как существующий data store**.
- `data-model.md`: `lat`/`lng` — **MVP-primary (Haversine + bounding box)**, PostGIS optional.

**Разбор DBA:**
1. **Полнотекст в PostgreSQL и русская морфология.** Дефолтный `tsvector` плохо стеммит русский без настройки словарей. Для маркетплейса с опечатками («лабродор») нужен либо **`pg_trgm`** (триграммы, fuzzy), либо нормальные русские словари (`russian` config + `hunspell`). Это **обязательно заложить**, иначе поиск будет «не находить очевидное».
2. **Гео-поиск (`docs/specs/07-geo-search-service.md`).** Решение `lat`/`lng` + Haversine + bounding-box для MVP — **прагматично и достаточно** для радиусов до 100 км и целевых 1.5 с (perf-спека). Это даже лучше для старта, чем тащить PostGIS. **Но `storage.md` надо исправить** — он ошибочно объявляет PostGIS основным.
3. **Elasticsearch** в MVP **не нужен** и в ADR его нет — это Фаза 2. Диаграммы вводят читателя в заблуждение.

> **Рекомендация DBA по поиску:** MVP = PostgreSQL FTS (`russian` + `pg_trgm`) + `lat/lng` Haversine. PostGIS — когда гео-запросы усложнятся (полигоны, KNN). Elasticsearch/OpenSearch — только когда появятся фасеты/рекомендации/аналитика. Убрать ES из MVP-диаграмм.

### 🔮 Готовность к AI (из миссии проекта)
Память проекта фиксирует долгосрочную цель: **AI-агенты как модераторы/админы** (`principal_type AGENT` уже в схеме). Для семантического поиска/матчинга/RAG понадобится **`pgvector`**. Его дешевле заложить в выбор расширений сейчас (это просто `CREATE EXTENSION`), чем городить отдельную векторную БД потом. Рекомендация: добавить pgvector в список расширений «на будущее».

### Вердикт Суб-агента 2: **🟡 ТРЕБУЮТСЯ ИЗМЕНЕНИЯ**
БД (PostgreSQL) — **достаточно и правильно**. Изменения: (1) деньги → BIGINT везде; (2) зафиксировать PG16; (3) FTS с русской морфологией + pg_trgm — в план MVP; (4) исправить `storage.md` (PostGIS не «основной» в MVP); (5) убрать ES из MVP-диаграмм; (6) пересмотреть Prisma из-за гео/JSONB; (7) заложить pgvector на будущее.

**Рассинхрон с ADR (повтор находки №1).** Deployment-диаграмма — это зрелая production-архитектура (multi-zone K8s, ALB+ILB, network policies, secrets via Vault, image scanning, non-root, read-only rootfs). **Как целевое состояние — отлично (8/10).** Как MVP — недостижимо в сроки и противоречит ADR (Docker Compose). ### Изоляция сервисов (Network Policies)
В `deployment-diagram` §Security network policies описаны грамотно (БД доступна только из API/worker pods, egress мониторится). Но это **K8s-конструкция**. В MVP (Docker Compose) изоляция = **раздельные Docker-сети + не публиковать порты PG/Redis наружу** (только внутренняя сеть), backend как единственный публичный сервис за reverse-proxy. Это нужно прописать явно — иначе типичная ошибка «PostgreSQL торчит в интернет на 5432».

### Вердикт Суб-агента 3: **🔴 ТРЕБУЮТСЯ ДОПОЛНЕНИЯ (блокирующие для РФ)**
Модель безопасности на бумаге зрелая (OWASP/NIST/ФЗ-152). Блокеры: (1) **заменить Stripe/PayPal/Twilio/SendGrid/Datadog/CloudFront на РФ-аналоги в самих доках**; (2) ввести WAF (ModSecurity+CRS) явно; (3) определить MVP-срез инфры с Docker-сетями вместо K8s NetworkPolicies; (4) зафиксировать `@nestjs/throttler`+Redis для rate-limit; (5) учесть 54-ФЗ/PCI в платёжном контуре.

## 🌐 СУБ-АГЕНТ 4: Web3 & NFT Expert (Токенизация)
> *Фокус: готовность к NFT/цифровым активам. NFT в `0001-tech-stack.md` **полностью отсутствует** — раздел носит проектный характер: что добавить, чтобы внедрить без боли.*

### Бизнес-кейсы NFT для ZooLink (где это реально полезно, а не хайп)
1. **Родословная (pedigree) как верифицируемый граф.** В схеме уже есть `mother_id`/`father_id`, `pedigree_id`, `show_titles`, `health_test_results`. NFT-сертификат родословной = неподделываемая привязка происхождения. **Самый сильный кейс** — борьба с поддельными метриками породы.
2. **Сертификаты породы/титулов** (выставки, чемпионства) как verifiable credentials.
3. **Право собственности / история передач.** Уже есть `ownership_transfers` (стейт-машина) и `animal_ownership_history`. NFT-токен владения = «цифровой паспорт животного», переходящий вместе с животным. Ложится на существующую модель **идеально**.

> ⚠️ Трезвая оценка эксперта: для **MVP** NFT **не нужен** и его не следует тащить в Фазу 1 — он не на критическом пути валидации гипотез маркетплейса. Но архитектуру надо подготовить так, чтобы внедрение в Фазе 2/3 не потребовало ломки. Ниже — что заложить сейчас «бесплатно».

### Нужен ли блокчейн и какой (контекст РФ)
**Да, если идти в верифицируемые сертификаты.** Выбор сети под РФ + mass-market:

| Сеть | Плюсы | Минусы | Вердикт |
|---|---|---|---|
| **TON** | **Telegram-нативность** (а у вас Telegram OAuth уже в стеке!), популярность в РФ, дешёвые транзакции, удобные кошельки в TG | Экосистема смарт-контрактов моложе EVM | ✅ **Рекомендация №1** — естественный фит под РФ-аудиторию и существующий Telegram-вход |
| **Polygon (PoS)** | EVM-совместимость, низкие комиссии, зрелый тулинг (огромный найм Solidity) | Меньше «нативности» в РФ | ✅ **Рекомендация №2** (если важна EVM-экосистема/мульти-рынок) |
| **Ethereum L1** | Максимальная ликвидность/доверие | Дорогой газ, оверкилл для сертификатов | ❌ Не для этого кейса |

### Где хранить метаданные NFT
**Гибрид (это индустриальный стандарт, не «или-или»):**
- **On-chain:** только токен + хэш метаданных + указатель.
- **Метаданные/медиа:** **IPFS** (через Pinata / web3.storage / собственный IPFS-узел) с **пиннингом**, либо **Arweave** для permanent-хранения сертификатов. Для РФ — собственный IPFS-узел + дублирование в существующий S3/MinIO.
- **Источник истины приложения:** **остаётся PostgreSQL.** NFT — это «зеркало доверия» поверх БД, а не замена БД. В схему добавить таблицу `digital_assets` (token_id, chain, contract_address, ipfs_cid, asset_type, animal_id FK, tx_hash, mint_status).

> ⚠️ Антипаттерн, которого избежать: хранить картинку животного прямо on-chain или класть приватные данные (ФЗ-152!) в публичные метаданные. В NFT — только публичные верифицируемые факты (происхождение, титулы), без ПДн владельца.

### Нужен ли Indexer
**Да, обязательно.** Блокчейн асинхронен и eventually-consistent; бэкенду нужен сервис, который слушает события контракта (Mint/Transfer) и синхронизирует статус в PG. Хорошая новость: **в схеме уже есть `outbox_events`** — это половина решения (исходящие события app→chain). Для входящих (chain→app) нужен:
- **The Graph (subgraph)** для EVM/Polygon — готовый индексатор, минимум кода; или
- **собственный listener-воркер** (TON: toncenter/toncenter SSE; EVM: viem/ethers + webhook), который пишет в ту же `outbox`/inbox-таблицу.

Это отдельный фоновый сервис (тот самый Worker Pod из deployment-диаграммы получает осмысленную роль).

### Кошельки и UX (критично для mass-market)
Владельцы котят **не должны** управлять seed-фразами. Рекомендация: **кастодиальный/абстрагированный кошелёк** (account abstraction, gasless meta-transactions, платформа платит газ). Для TON — TON Connect + кастодиальный режим. Это решает «боль» внедрения, о которой просит ТЗ.

### Вердикт Суб-агента 4: **➕ ОТСУТСТВУЕТ — заложить хуки, реализовать в Фазе 2+**
NFT в стеке нет (это нормально для MVP). Чтобы внедрить «без боли», заложить **сейчас**: (1) таблицу `digital_assets` + статус-поле; (2) переиспользовать `outbox_events` для chain-синхронизации; (3) выбрать сеть — **TON (через уже имеющийся Telegram) или Polygon**; (4) IPFS/Arweave для метаданных, PG остаётся истиной; (5) роль Indexer-воркера; (6) кастодиальные кошельки для mass-market. Реализация — Фаза 2+, не MVP.

### Итоговое заключение ARB

**Стек по своей сути — здоровый и грамотно подобранный.** Ядро (NestJS + TypeScript + PostgreSQL + Redis + S3 + Docker) — это правильный, кадрово-обеспеченный в РФ, масштабируемый фундамент. ADR-0001 написан зрело и без хайпа.

**Три класса проблем, требующих решения ДО первой строки кода:**

1. **🔴 РАССИНХРОН «канон vs диаграммы» (главный системный риск).** ADR описывает монолит/Compose, диаграммы — микросервисы/K8s/gRPC. Без явного «MVP-среза» команда построит не то. **Действие:** пометить диаграммы как Target State, зафиксировать MVP-архитектуру = модульный монолит на Docker Compose.

2. **🔴 РФ-НЕПРИГОДНОСТЬ ДЕФОЛТНЫХ ПРОВАЙДЕРОВ (блокер для продакшена).** Stripe, PayPal, Twilio, SendGrid, Datadog, CloudFront — не работают/недоступны в РФ. Абстракции в коде заложены — **но дефолты в доках надо заменить на РФ-набор сейчас**, плюс 54-ФЗ/PCI для платежей.

3. **🟡 ТОЧЕЧНЫЕ ТЕХ-РЕШЕНИЯ.** Деньги → BIGINT (не INTEGER); Prisma пересмотреть из-за гео/JSONB; русская морфология поиска; зафиксировать PG16; pgvector на будущее; убрать ES/PostGIS из MVP как «уже существующие».

**NFT** — отсутствует, и это **корректно для MVP**. Достаточно заложить дешёвые хуки (таблица `digital_assets`, переиспользование `outbox_events`, выбор сети TON/Polygon), чтобы Фаза 2 прошла без ломки архитектуры.

**Общий вердикт ARB: 🟡 УСЛОВНО ГОТОВО — стек одобрить с обязательной ремедиацией пунктов 1–2 до старта разработки.** Менять язык/фреймворк/БД не требуется; требуется привести документацию к одному (MVP) контуру и заменить санкционно-зависимых провайдеров.
